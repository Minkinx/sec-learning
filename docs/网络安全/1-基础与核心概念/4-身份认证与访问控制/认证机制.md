# 认证机制

> 身份认证是安全体系的第一道防线。认证机制回答"你是谁"的问题，通过验证用户声称的身份来防止未授权访问。

## 认证因素

认证因素分为三大类（以及一个补充类）：

| 因素类型 | 说明 | 示例 | 安全强度 |
|----------|------|------|---------|
| 知识因素（Something you know） | 用户知晓的秘密 | 密码、PIN码、安全问题 | 最低（易被钓鱼/猜测） |
| 持有因素（Something you have） | 用户持有的设备或令牌 | 手机、YubiKey、智能卡 | 中等（可丢失/被盗） |
| 固有因素（Something you are） | 用户的生物特征 | 指纹、面部、虹膜、声纹 | 高（但存在生物特征泄露不可撤销问题） |
| 行为因素（Something you do） | 用户的行为模式 | 击键动态、鼠标移动模式、步态 | 补充（持续认证） |

### 多因素认证（MFA）
使用两个或更多不同类别的因素组合。**同一类别的多个凭证不是多因素认证**（如密码+安全问题仍然是单因素）。

## 密码安全

### NIST SP 800-63B 密码指南

| 方面 | 传统观点 | NIST 800-63B 建议 |
|------|---------|-----------------|
| 复杂性规则 | 必须包含大小写+数字+特殊字符 | 不强制要求（鼓励长密码而非复杂密码） |
| 定期更换 | 强制每30-90天更换 | 不强制（除非有泄露迹象） |
| 长度 | 最少8位 | 最少8位，**强烈建议64位以上** |
| 提示问题 | 安全问题（母亲的 maiden name） | **不建议使用** |
| 密码提示 | 允许 | **禁止** |
| 复制粘贴 | 通常禁止 | 应允许（便于密码管理器使用） |

### 推荐实践
```python
# 使用密码管理器生成和存储随机密码（避免密码重用）
# 示例：生成一个20位随机密码
import secrets
import string

alphabet = string.ascii_letters + string.digits + "!@#$%^&*"
password = ''.join(secrets.choice(alphabet) for _ in range(20))
print(password)  # 例如：aK9#mP2$xR7@vL4%nQ6
```

- **密码重用**：最大的密码安全问题。一个平台泄露即导致多个平台沦陷。
- **Have I Been Pwned**：检查密码/邮箱是否出现在已知泄露中。NIST建议与HIBP API集成。
- **关键系统**：必须使用密码管理器 + MFA。

### 密码存储

| 存储方式 | 安全性 | 说明 |
|----------|--------|------|
| 明文存储 | ❌ 极其危险 | 任何一个漏洞即泄露所有用户密码 |
| MD5/SHA-1直接哈希 | ❌ 不安全 | 彩虹表攻击、碰撞攻击 |
| SHA-256加盐 | ⚠️ 不足 | 速度太快，GPU可离线暴力破解 |
| bcrypt/scrypt | ✅ 好 | 工作因子可调，抗GPU |
| Argon2id | ✅ **最佳** | 内存硬+CPU硬+抗侧信道 |
| 加盐（Salt） | ✅ 必须 | 每个用户随机唯一salt，防止彩虹表 |

### 密码策略实施示例（Linux PAM）
```bash
# /etc/security/pwquality.conf
minlen = 14
maxrepeat = 3
reject_username = true
enforce_for_root
# 集成HIBP（通过pam_pwquality自定义模块）
```

## 多因素认证（MFA）

### MFA技术对比

| 技术 | 安全性 | 用户体验 | 成本 | 推荐度 |
|------|--------|---------|------|--------|
| SMS验证码 | ⚠️ 低 | 好 | 低 | **不推荐**（SIM Swap攻击、SS7漏洞） |
| TOTP（时间基一次性密码） | ✅ 中 | 好 | 免费 | 常用（Google Authenticator/Authy） |
| HOTP（计数器基） | ✅ 中 | 好 | 免费 | 较少使用 |
| Push通知 | ✅ 中高 | 优 | 中等 | MFA疲劳攻击风险 |
| U2F（FIDO2硬件密钥） | ✅✅ 高 | 优 | 低（YubiKey $25-50） | **推荐**（抗钓鱼） |
| WebAuthn（平台认证器） | ✅✅ 高 | 优（指纹/面部） | 免费 | **推荐**（内建手机/笔记本） |

### TOTP工作原理
```
服务器和用户设备预共享一个密钥（Base32编码）
           │
时间每30秒递增 → TOTP = HMAC-SHA1(Key, Timestep) 截断为6-8位数字
           │
验证公式：|服务器计算的当前TOTP - 用户输入的TOTP| == 0
（通常允许±1个时间步的偏差）
```

### FIDO2/WebAuthn流程
```
1. 用户注册：
   服务器 → 客户端：挑战（Challenge）+ 依赖方ID（RP ID）
   客户端/认证器 → 服务器：公钥 + 凭据ID + 签名
   服务器：存储公钥

2. 用户登录：
   服务器 → 客户端：挑战
   客户端 → 认证器：挑战 + RP ID
   用户：触摸按钮/指纹确认
   认证器：用私钥对挑战签名
   客户端 → 服务器：凭据ID + 签名
   服务器：用存储的公钥验证签名
```

## 免密码认证（Passwordless）

2020年以来，主要平台加速推动免密码认证：

| 方案 | 原理 | 使用量 |
|------|------|--------|
| Passkeys | FIDO2 + 设备绑定私钥 + iCloud/Google同步 | Apple/Google/Microsoft全面支持 |
| Magic Link | 邮箱发送一次性登录链接 | Slack、Notion等 |
| 生物特征登录 | 设备内生物特征绑定密钥 | Windows Hello、Face ID |
| 短信验证码登录 | 仅用于登录（无密码） | 部分国内应用 |

**Passkeys 架构**：
```
用户注册设备时，设备生成一个密钥对：
- 私钥安全存储在设备安全区（Secure Enclave / TEE）
- 公钥上传至服务的认证服务器
- 通过iCloud密钥串/Google密码管理器跨设备同步（端到端加密）

用户登录时：
1. 服务端发送Challenge
2. 设备用私钥签名（用户通过面部/指纹/ PIN确认）
3. 服务端用存储的公钥验证签名

优势：完全抗钓鱼——攻击者无法伪造依赖方ID（RP ID）
```
