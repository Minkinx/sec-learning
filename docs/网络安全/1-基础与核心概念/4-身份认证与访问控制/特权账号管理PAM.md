# 特权账号管理PAM

> 特权账号管理（Privileged Access Management, PAM）是针对管理员、服务账号、应用程序所使用的**高权限账号**的安全管理体系和解决方案。PAM是等保2.0和零信任架构的关键组成部分。

## 为什么需要PAM

### 特权账号面临的风险

| 风险 | 影响 | 典型案例 |
|------|------|---------|
| 共享账号滥用 | 无法追溯操作责任人 | 多个运维人员共享root密码 |
| 凭据泄露 | 账号被外部或内部攻击者利用 | 管理员通过钓鱼邮件泄露凭据 |
| 横向移动 | 攻击者利用特权账号在内网扩展 | Pass-the-Hash、Kerberoasting |
| 权限过度授予 | 普通员工拥有不应有的管理员权限 | 开发人员拥有生产环境root |
| 僵尸账号 | 离职员工或废弃服务账号未被回收 | 已离职员工的SSH密钥仍有效 |
| 违规操作 | 特权用户执行非授权操作 | 管理员直接修改数据库记录 |

### 合规要求
- **等保2.0**：三级及以上系统要求身份鉴别、访问控制、安全审计
- **PCI DSS**：要求保护系统密码，限制特权访问
- **SOX**：要求对系统访问和变更进行审计
- **GDPR**：要求确保数据处理的安全性

## PAM核心能力

### 凭据保险库（Credential Vault）

```
┌──────────────────────────────────────┐
│          凭据保险库                    │
│  ┌──────┐ ┌──────┐ ┌──────┐         │
│  │ root  │ │admin │ │svc_  │         │
│  │@db-01 │ │@dc-01│ │backup│         │
│  └──────┘ └──────┘ └──────┘         │
│  所有凭据使用AES-256加密存储          │
│  定期自动轮换密码                     │
│  仅授权用户/系统可检索                │
└──────────────────────────────────────┘
```

- **加密存储**：凭据以加密形式存储在保险库中（AES-256-GCM或类似）
- **自动轮换**：定期自动更改密码，轮换后旧密码作废
- **即时（JIT）凭据**：只在需要时生成凭据，用完立即销毁
- **检入/检出**：凭据使用时检出（Check-out），使用完毕检入（Check-in）

### 会话管理

- **会话记录**：记录所有特权会话（视频回放 + 文本日志）
- **会话监控**：实时查看活跃会话，发现可疑操作可终止
- **命令控制**：允许/禁止特定命令（如禁止在数据库服务器上执行DROP TABLE）
- **代理架构**：用户不能直接SSH到目标服务器，而是通过PAM网关建立会话

### 权限管理

- **请求-审批流**：用户需提交申请并经过审批才能检索凭据
- **最小特权**：按需授予完成工作所需的最低权限，限定时间范围
- **动态权限**：权限在审批后自动授予，使用完自动回收

### 审计与报告

- **完整审计轨迹**：谁、何时、从哪台机器、访问了哪个资源、执行了什么操作
- **合规报告**：自动生成满足等保/PCI/SOX合规要求的报告
- **异常告警**：检测异常访问模式（如在凌晨3点以root登录）

### HashiCorp Vault 示例
```bash
# 启动开发服务器（仅用于测试）
vault server -dev

# 启用KV存储引擎
vault secrets enable -path=secret kv-v2

# 存储凭据
vault kv put secret/db/prod username=admin password=S3cur3P@ss!

# 读取凭据
vault kv get secret/db/prod

# 动态数据库凭据（MySQL）
vault write database/config/mysql \
    plugin_name=mysql-database-plugin \
    connection_url="{{username}}:{{password}}@tcp(10.0.1.100:3306)/" \
    allowed_roles="readonly"

vault write database/roles/readonly \
    db_name=mysql \
    creation_statements="CREATE USER '{{name}}'@'%' IDENTIFIED BY '{{password}}';GRANT SELECT ON *.* TO '{{name}}'@'%';" \
    default_ttl="8h" \
    max_ttl="24h"

# 获取临时数据库账号
vault read database/creds/readonly
# Key                Value
# ---                -----
# lease_id           database/creds/readonly/abc123
# lease_duration     8h
# password           ABcd12-34EF56-78GH90
# username           v-readonly-abc123
```

### Teleport 示例
```bash
# Teleport使用证书认证替代静态密码
# 流程：用户 → Teleport Auth Server → 目标机器

# 用户登录
tsh login --proxy=teleport.company.com --auth=oidc

# 查看可访问的机器

# 通过Teleport SSH到目标（自动获取短期证书）

```

## PAM部署路径

### 阶段化部署建议

| 阶段 | 时间 | 活动 | 成果 |
|------|------|------|------|
| **第一阶段：盘点** | 1-2周 | 发现所有特权账号、服务账号、SSH密钥 | 完整清单 |
| **第二阶段：关键系统优先** | 2-4周 | 锁定域管理员、root、数据库管理员账号 | 关键凭据受控 |
| **第三阶段：逐步收紧** | 1-3月 | 扩大覆盖面至所有关键系统，实施审批流程 | 全面覆盖 |
| **第四阶段：服务账号管理** | 3-6月 | 将自动化脚本、CI/CD、微服务使用的凭据纳入管理 | 全量自动化 |
| **第五阶段：持续优化** | 持续 | 定期审计、轮换策略审查、异常检测 | 持续改进 |

## 最佳实践

1. **零静态凭据**：系统默认不留静态密码，全量使用JIT凭据或动态生成
2. **默认拒绝**：所有特权访问必须经过审批，默认拒绝
3. **轮换频率**：管理员密码至少每90天轮换；服务密码至少每30天轮换
4. **紧急破碎**：设置紧急访问通道（Break Glass），在紧急情况下绕过正常审批（需审计）
5. **会话记录保留**：等保2.0要求至少保留6个月的运维操作日志
6. **定期审核**：每季度审查特权账号和权限分配
