# 哈希与数字签名

> 哈希函数（散列函数）和数字签名是数据完整性保护和身份验证的基础。哈希函数将任意长度的数据映射为固定长度的摘要值；数字签名则通过非对称密码学确保数据的来源和完整性。

## 哈希函数

### 核心性质
- **确定性**：相同输入始终产生相同输出
- **快速计算**：给定输入，哈希值计算高效
- **抗原像性（Preimage Resistance）**：给定哈希值 `h`，无法逆向找到 `m` 使得 `hash(m)=h`（单向性）
- **抗第二原像性（Second Preimage Resistance）**：给定 `m1`，无法找到 `m2≠m1` 使得 `hash(m1)=hash(m2)`
- **抗碰撞性（Collision Resistance）**：无法找到任意 `m1≠m2` 使得 `hash(m1)=hash(m2)`
- **雪崩效应**：输入微小变化导致输出约50%比特翻转

### 哈希算法对比

| 算法 | 输出长度 | 安全性 | 状态 | 说明 |
|------|---------|--------|------|------|
| MD5 | 128位 | ❌ 已破解 | **废弃** | 碰撞攻击（2004年Wang等人，可在数秒内构造碰撞） |
| SHA-1 | 160位 | ❌ 已破解 | **废弃** | 选择前缀碰撞攻击（SHAttered, 2017） |
| SHA-256 | 256位 | ✅ 安全 | **推荐** | 广泛使用，比特币和TLS标配 |
| SHA-384 | 384位 | ✅ 安全 | **推荐** | SHA-512的截断版本 |
| SHA-512 | 512位 | ✅ 安全 | **推荐** | 更高安全性，性能较慢 |
| SHA-3-256 | 256位 | ✅ 安全 | **推荐** | 不同的SPONGE结构，抗长度扩展攻击 |
| SM3 | 256位 | ✅ 安全 | 国密标准 | 与SHA-256安全性相当 |
| BLAKE2b | 可变 | ✅ 安全 | **推荐** | 比SHA-2更快，用于部分现代协议 |
| BLAKE3 | 可变 | ✅ 安全 | 新 | 极其高效，支持并行 |

### 哈希碰撞的实践影响
```
MD5碰撞攻击在2004年被发现后：
- 数字证书：CA不再签发MD5签名的证书（2011年Flame恶意软件仍利用了MD5碰撞）
- 文件完整性：Git已切换至SHA-1硬提交检测，后续计划迁移至SHA-256
- 软件签名：所有主流操作系统和包管理器已迁移至SHA-2/SHA-3

2017年SHAttered攻击：Google展示了对SHA-1的选择前缀碰撞攻击
（生成两个不同的PDF文件，具有相同的SHA-1哈希值）
```

### 生日攻击
利用生日悖论寻找碰撞：对于输出长度为 `n` 位的哈希函数，攻击者只需要约 `2^(n/2)` 次尝试就有50%概率找到碰撞。这也是为什么安全哈希函数至少需要256位输出。

## 哈希的应用场景

### 密码存储
密码绝不能明文存储，也不能使用快速哈希（如MD5/SHA-256直接哈希）：
```python
# 正确的密码哈希（Python示例）
import bcrypt
# 或使用 argon2-cffi

# bcrypt：自动加盐，可调工作因子
password = b"my_secure_pass123"
salt = bcrypt.gensalt(rounds=12)  # work factor = 2^12
hashed = bcrypt.hashpw(password, salt)

# 验证
bcrypt.checkpw(password, hashed)
```

| 算法 | 设计目的 | 工作因子 | 说明 |
|------|---------|----------|------|
| bcrypt | 密码哈希 | cost（2^cost次迭代） | **推荐**，Blowfish密码学基础 |
| scrypt | 密码哈希 | N/r/p（内存+CPU） | 内存硬函数，抗ASIC |
| Argon2id | 密码哈希 | m/t/p（内存/时间/并行） | **强烈推荐**——PHC竞赛获胜者 |
| PBKDF2 | 密钥派生 | iterations | 较弱（无内存硬性要求） |

### HMAC（基于哈希的消息认证码）
```
HMAC(K, m) = H((K' ⊕ opad) || H((K' ⊕ ipad) || m))
```
- 使用共享密钥验证消息完整性和真实性
- 密钥长度通常为哈希输出长度或更长
- 单独使用哈希不能防止中间人篡改——MAC提供完整性+认证

### 文件完整性验证
```bash
# 生成校验和
sha256sum important_file.tar.gz > checksum.sha256

# 验证
sha256sum -c checksum.sha256

# 远程分发
# 校验和文件本身应通过签名验证
gpg --detach-sign checksum.sha256
```

### 数字签名中的哈希
签名前对消息进行哈希的原因：
1. 提高效率——哈希后的数据长度固定且小
2. 防止签名伪造——直接签名长消息存在理论攻击
3. 兼容性——各种格式的消息都能签名

## 数字签名

### 签名算法对比

| 算法 | 基础 | 签名大小 | 速度 | 推荐度 |
|------|------|---------|------|--------|
| RSA-PSS | RSA | 256-512字节 | 慢（签名）/ 快（验证） | **推荐** |
| RSA-PKCS1v1.5 | RSA | 256-512字节 | 慢/快 | 兼容性使用 |
| ECDSA P-256 | ECC | 64-72字节 | 快 | **推荐** |
| EdDSA (Ed25519) | Edwards Curve | 64字节 | 非常快 | **强烈推荐** |
| SM2 | 国密ECC | 64字节 | 快 | 国密合规 |
| DSA | 离散对数 | ~40字节 | 慢 | **不推荐** |
| ML-DSA (Dilithium) | 格密码 | 2-4KB | 快 | 后量子（NIST标准） |

### EdDSA签名示例（概念代码）
```python
from cryptography.hazmat.primitives.asymmetric import ed25519

# 生成密钥对
private_key = ed25519.Ed25519PrivateKey.generate()
public_key = private_key.public_key()

# 签名
message = b"Important document content"
signature = private_key.sign(message)

# 验证
try:
    public_key.verify(signature, message)
    print("签名验证成功")
except InvalidSignature:
    print("签名验证失败")
```

## 哈希长度扩展攻击（Hash Length Extension Attack）

Merkle-Damgard结构（MD5、SHA-1、SHA-2）的哈希函数易受长度扩展攻击。

### 攻击原理
给定 `H(m)` 和 `len(m)`，攻击者可在不知道 `m` 内容的情况下计算 `H(m || padding || extra_data)`。
这导致：**使用 `H(secret || message)` 作为MAC是不安全的。**

### 受影响与不受影响的算法
| 受影响 | 不受影响 |
|--------|---------|
| MD5 ✅ | SHA-3（Sponge结构） |
| SHA-1 ✅ | BLAKE2（HAIFA结构） |
| SHA-2 ✅ | HMAC（双哈希构造） |
| — | SHA-3 KMAC |

### 正确的MAC构造
```python
# ❌ 易受长度扩展攻击
weak_mac = hashlib.sha256(secret + message).hexdigest()

# ✅ 正确做法
secure_mac = hmac.new(key=secret, msg=message, digestmod='sha256').hexdigest()
```
