# 数字取证

## 概述

数字取证（Digital Forensics）是在安全事件发生后收集、保护、分析电子证据以还原攻击过程并支持法律诉讼的学科。它是事件响应的核心环节，也是判定事件影响范围、确定根因的关键。

## 取证原则与规范

### Locard 交换原理

"每次接触都会留下痕迹"——攻击者在系统上的任何操作都会留下数字痕迹，取证分析师的任务就是找到这些痕迹并重建事件。

### 取证基本原则

| 原则 | 说明 | 实践要求 |
|------|------|---------|
| 完整性 | 确保证据不被修改 | 使用写保护、计算哈希值 |
| 可追溯 | 每个操作都有记录 | 详细记录时间、操作人、操作内容 |
| 可重现 | 其他分析师可以得出相同结论 | 使用标准的工具和方法流程 |
| 最小干扰 | 对证据的影响最小化 | 先备份再分析，不更改原始证据 |
| 及时性 | 尽快获取易失性数据 | 内存 → 连接 → 磁盘的顺序采集 |

### 证据价值层次

```
最大价值 / 最易丢失:
┌─────────────────────────────────┐
│ 内存 (RAM) — 进程、网络连接     │ ← 关机即丢失
│ 路由表、ARP 缓存               │
│ 临时文件系统 (/proc, /sys)      │
├─────────────────────────────────┤
│ 网络连接状态                    │ ← 断网即丢失
│ 当前登录用户                    │
├─────────────────────────────────┤
│ 磁盘数据                        │ ← 可保存但易篡改
│ 系统日志                        │
│ 应用程序数据                    │
├─────────────────────────────────┤
│ 备份数据                        │ ← 长期保存可用
│ 归档日志                        │
└─────────────────────────────────┘
```

## 证据采集 (Acquisition)

### 采集顺序 (Order of Volatility)

```
1. 内存 (RAM) ─────────────── 最易失
2. 网络连接和状态
3. 运行进程
4. 磁盘
5. 远程日志和备份 ─────────── 最持久
```

## 内存分析 (Memory Analysis)

### Volatility 3

Volatility 3 是目前最广泛使用的开源内存取证框架，支持 Windows、Linux 和 macOS。

```bash
# 基本信息
python3 vol.py -f mem.raw windows.info

# 列举进程
python3 vol.py -f mem.raw windows.pslist
python3 vol.py -f mem.raw windows.psscan   # 使用池扫描，可发现隐藏进程
python3 vol.py -f mem.raw windows.pstree   # 进程树视图

# 进程命令行参数
python3 vol.py -f mem.raw windows.cmdline

# 网络连接
python3 vol.py -f mem.raw windows.netscan
python3 vol.py -f mem.raw windows.netstat

# DLL 列表
python3 vol.py -f mem.raw windows.dlllist --pid XXXX

# 可疑代码检测
python3 vol.py -f mem.raw windows.malfind
python3 vol.py -f mem.raw windows.devicetree

# 注册表 hive
python3 vol.py -f mem.raw windows.registry.hivelist
python3 vol.py -f mem.raw windows.registry.printkey

# 文件扫描
python3 vol.py -f mem.raw windows.filescan

# 时间线
python3 vol.py -f mem.raw windows.timeliner

# 进程句柄
python3 vol.py -f mem.raw windows.handles --pid XXXX
```

### 内存分析常见发现

| 发现项 | Volatility 插件 | 意义 |
|--------|----------------|------|
| 隐藏进程 | windows.psscan 与 pslist 差异 | 表示存在 rootkit 或 DKOM |
| 异常进程注入 | windows.malfind | 内存区域标记可疑 |
| 无文件 C2 连接 | windows.netscan | 查找 beacon 外连 |
| LSASS 转储 | windows.cmdline | 凭据窃取检测 |
| DLL 劫持 | windows.dlllist + windows.modscan | DLL 搜索顺序劫持 |

### Volatility 分析案例

```
场景: 检测到可疑进程 notepad.exe (PID 1234)

1. 查看进程信息:
   windows.pstree → notepad.exe 父进程为 OUTLOOK.EXE (异常!)

2. 查看命令行:
   windows.cmdline --pid 1234
   → "notepad.exe" 全路径可疑 C:\Users\Public\notepad.exe

3. 检查网络连接:
   windows.netscan → PID 1234 连接 185.234.72.18:443

4. 检查内存:
   windows.malfind --pid 1234 → 发现可执行代码段标记

结论: notepad.exe 被注入恶意代码，作为 C2 beacon
```

## 磁盘分析 (Disk Forensics)

### 文件时间戳分析 (MACB)

```
MACB 时间戳分析:
  M - Modified (文件内容修改时间)
  A - Accessed (文件访问时间)  
  C - Changed (文件元数据变更时间)
  B - Born/Created (文件创建时间)

攻击检测:
  场景: 恶意 DLL 注入
  正常: DLL 的 B < M = C (创建最早，修改=元数据变更)
  异常: DLL 的 B 和 M 时间差很大 (被替换)
  
  场景: 横向移动工具使用
  PsExec 文件创建的时间戳接近执行时间
```

### Windows 关键取证 artifact

| Artifact | 路径 | 取证价值 |
|----------|------|---------|
| Prefetch | C:\Windows\Prefetch\*.pf | 应用程序执行记录 |
| Shimcache | Registry: HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\AppCompatCache | 程序执行历史 |
| Amcache | C:\Windows\AppCompat\Programs\Amcache.hve | 应用程序兼容性缓存 |
| Jumplist | C:\Users\%user%\AppData\Roaming\Microsoft\Windows\Recent\AutomaticDestinations | 最近文档列表 |
| Shellbags | Registry: USRCLASS.DAT | 文件夹访问记录 |
| UserAssist | Registry: NTUSER.DAT\Software\Microsoft\Windows\CurrentVersion\Explorer\UserAssist | GUI 执行次数 |
| SRUM | C:\Windows\System32\sru\SRUDB.dat | 系统资源使用监控 |
| $MFT | C:\$MFT | 所有文件的元数据 |
| $UsnJrnl | C:\$Extend\$UsnJrnl | NTFS 变更日志 |
| 事件日志 | C:\Windows\System32\winevt\Logs\*.evtx | 系统/安全/应用日志 |

### Autopsy/SleuthKit 分析

```bash
# 创建案例并加载镜像
autopsy --create-case IR-2026-005
autopsy --add-datasource /mnt/evidence/sda.dd

# 命令行分析 — fls 列出文件系统
fls -m C: -r /mnt/evidence/sda.dd > bodyfile.txt

# 生成时间线

# icat — 通过 inode 读取文件

# blkcat — 读取指定扇区
blkcat /mnt/evidence/sda.dd 5000 10 > sector_5000.bin
```

### 注册表分析

```bash
# 使用 regripper 分析离线注册表 hive
rip.pl -r /mnt/evidence/hives/SYSTEM -f system > system_output.txt
rip.pl -r /mnt/evidence/hives/SOFTWARE -f software > software_output.txt
rip.pl -r /mnt/evidence/hives/NTUSER.DAT -f ntuser > ntuser_output.txt

# 关键发现:
# SOFTWARE — 系统安装程序、服务、网络配置
# SYSTEM — 挂载设备、服务控制
# NTUSER.DAT — 用户行为记录 (MRU, TypedURLs, UserAssist)
# SAM — 本地用户账户哈希
# SECURITY — 安全策略、登录会话
```

## 网络取证

### pcap 分析要点

```bash
# Wireshark / tshark 命令行分析
tshark -r capture.pcap -Y "http.request" -T fields -e http.host -e http.request.uri

# 提取 HTTP 对象 (文件还原)
tshark -r capture.pcap --export-objects "http,/tmp/extracted"

# 查看 DNS 查询 — 可能发现 C2 域名
tshark -r capture.pcap -Y "dns.flags.response == 0" -T fields -e dns.qry.name | sort -u

# 检测可疑 TLS 证书
tshark -r capture.pcap -Y "ssl.handshake.certificate" \
  -T fields -e ssl.handshake.certificate_issuer

# 分析 HTTPS 会话的 JA3 指纹 (TLS 客户端指纹)
tshark -r capture.pcap -Y "tls.handshake.type == 1" \
  -T fields -e tls.handshake.ciphersuites \
  -e tls.handshake.extensions_supported_group

# TCP 流重组 — 查看完整对话
tshark -r capture.pcap -z follow,tcp,ascii,0
```
