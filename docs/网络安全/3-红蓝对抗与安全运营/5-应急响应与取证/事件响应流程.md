# 事件响应流程

## 概述

事件响应（Incident Response, IR）是组织在面对安全事件时采取的标准化应对流程。一个完善的事件响应流程能够显著降低安全事件造成的影响。NIST SP 800-61 修订版 2 是目前最广泛采用的事件响应指南。

## NIST SP 800-61 四阶段模型

```
       Preparation
           │
           ▼
Detection & Analysis ◄───────┐
           │                  │
           ▼                  │
 Containment, Eradication    │
      & Recovery              │
           │                  │
           ▼                  │
Post-Incident Activity ──────┘ (Lessons Learned → Back to Preparation)
```

| 阶段 | 核心活动 | 关键输出 |
|------|---------|---------|
| 准备 | 制定计划、组建团队、部署工具 | IR 计划、Playbooks、工具链 |
| 检测与分析 | 发现事件、确认事件、评估范围 | 事件确认、严重级别、初步 IOC |
| 遏制、根除与恢复 | 隔离影响、清除威胁、恢复业务 | 遏制措施、取证证据、恢复确认 |
| 事后活动 | 根因分析、经验教训、改进措施 | RCA 报告、改进项、演练计划 |

## 阶段 1: 准备 (Preparation)

### 事件响应团队 (CIRT)

```
CIRT 组织架构:

  Incident Commander (事件总指挥)
        │
  ┌─────┼─────┬──────┬──────┬──────┐
  │     │     │      │      │      │
 技术  取证  通信   法律    HR   管理层
 负责人 负责人 负责人  顾问   代表  代表
```

| 角色 | 职责 | 技能要求 |
|------|------|---------|
| 事件总指挥 (IC) | 协调全局、决策、资源调配 | 项目管理、压力决策 |
| 技术负责人 | 技术分析、遏制、根除 | 安全运维、逆向工程 |
| 取证分析师 | 证据收集、时间线重建 | 数字取证、内存分析 |
| 沟通负责人 | 内外沟通管理 | 危机沟通、报告撰写 |
| 法律顾问 | 法律合规、监管报告 |  cybersecurity 法律 |
| HR 代表 | 内部人员相关事务 | HR 流程、员工关系 |
| 管理层 (CISO) | 资源批准、业务决策 | 战略决策、风险管理 |

### 事件分类与严重级别

| 级别 | 名称 | 定义 | 响应时间 | 上报对象 |
|------|------|------|---------|---------|
| P1 | 紧急 (Critical) | 核心业务中断、勒索软件、数据泄露、APT 活动 | < 15 分钟 | CISO, CEO |
| P2 | 高危 (High) | 横向移动、定向攻击、严重漏洞利用 | < 1 小时 | 安全总监 |
| P3 | 中危 (Medium) | 单主机感染、钓鱼成功、违规扫描 | < 4 小时 | 安全主管 |
| P4 | 低危 (Low) | 扫描探测、配置错误、误报 | < 24 小时 | 安全分析师 |

### IR 工具包

```
事件响应工具包检查清单:

实时响应:
├── EDR 控制台 (CrowdStrike/Defender ATP)
├── SIEM 访问权限 (Splunk/ELK)
├── 远程取证 (Velociraptor)
├── 威胁情报平台 (MISP/VirusTotal)
└── 代码签名验证平台

取证分析:
├── 内存分析 (Volatility 3)
├── 磁盘取证 (FTK Imager/Autopsy)
├── 网络取证 (Wireshark/NetworkMiner)
├── 恶意软件分析 (Cuckoo/any.run)
└── 日志分析 (jq/Epoch/Log2Timeline)

沟通协作:
├── 安全通讯渠道 (Signal/Slack 安全频道)
├── 事件工单系统 (TheHive/Jira)
├── 会议桥 (隔离电话会议)
└── 管理层汇报模板
```

## 阶段 2: 检测与分析 (Detection & Analysis)

### 检测来源

```
检测源优先级:
  最高可信度:
  ├── EDR 告警 (进程创建链异常)
  ├── 勒索软件加密检测
  ├── 内部员工报告 (网络钓鱼)
  ├── SIEM 关联规则触发
  │
  中等可信度:
  ├── IDS/IPS 告警
  ├── WAF/IPS 告警
  ├── 漏洞扫描器发现
  │
  低可信度:
  ├── 外部通报 (第三方/执法部门)
  ├── 暗网信息
  └── 公开披露
```

### 事件确认流程

```
新告警/报告
    │
    ▼
Tier 1 研判 ──→ 误报 → 关闭 / 更新规则
    │
    ▼ (确认为事件)
严重级别评估
    │
    ├── P1/P2 → 立即升级到 Tier 2/3 + 启动 IR 流程
    │
    └── P3/P4 → 纳入排期处理
```

### 时间线重建

```
事件时间线示例:

[2026-05-26 09:15] — 初始访问: 员工打开钓鱼邮件附件
[2026-05-26 09:16] — 执行: Word 宏执行下载脚本
[2026-05-26 09:17] — C2: Beacon 建立 HTTPS 连接
[2026-05-26 09:20] — 信息收集: whoami, ipconfig, net view
[2026-05-26 09:25] — 凭据窃取: 尝试 LSASS 读取
[2026-05-26 09:30] — 横向移动: PsExec 到另一台服务器
[2026-05-26 09:35] — EDR 告警触发 (Tier 1 确认)
[2026-05-26 09:37] — 升级到 P1 IR 流程
[2026-05-26 09:40] — 终端隔离启动
[2026-05-26 09:50] — 确认隔离完成
```

## 阶段 3: 遏制、根除与恢复

### 遏制策略

| 遏制策略 | 适用场景 | 做法 | 优点 | 缺点 |
|---------|---------|------|------|------|
| 网络隔离 | 勒索软件、C2 活动 | 从交换机/DHCP 断开连接 | 快速有效 | 业务中断 |
| 账户禁用 | 凭据窃取、异常登录 | 禁用用户账户 | 精准隔离 | 影响用户工作 |
| 进程终止 | 恶意进程执行中 | 强制终止可疑进程 | 阻止即时破坏 | 可能未清除 rootkit |
| 防火墙规则 | C2 外连 | 拒绝特定 IP/域名 | 不需要接触终端 | 攻击者可能更换 IP |
| 密码重置 | 凭据泄露 | 全局密码重置 | 阻断后续攻击 | 用户影响大、操作耗时 |

### 根除方法

```
根除步骤:
├── 确认所有受感染主机
│   ├── 使用 IOC 扫描全量资产
│   └── 确认无漏网主机
│
├── 清除恶意软件
│   ├── 使用 EDR 清除或重新安装系统
│   └── 删除持久化机制 (计划任务、服务、启动项)
│
├── 修补漏洞入口
│   ├── 打补丁或配置修复
│   └── 软件版本升级
│
└── 确认凭据清理
    ├── 重置所有受影响账户密码
    └── 撤销泄露的 API 密钥
```

### 恢复确认

```
恢复检查清单:
□ 主机重新接入前安全扫描确认
□ 恶意文件已完全清除
□ 持久化机制已删除
□ 漏洞已修补
□ 受影响的账户密码已重置
□ 日志已额外备份供后续分析
□ 业务方确认系统恢复正常
□ 监控规则已部署用于发现二次入侵
```

## 阶段 4: 事后活动 (Post-Incident Activity)

### 根因分析 (RCA)

使用 "5 Whys" 方法追溯根本原因：

```
问题: 攻击者通过钓鱼邮件获得初始访问

Why 1: 员工打开了带有恶意宏的 Word 文档
 └── Why 2: 员工不知道这是钓鱼邮件
     └── Why 3: 钓鱼邮件通过了邮件安全网关
         └── Why 4: 邮件网关没有检测到文档中的恶意宏
             └── Why 5: 宏检测规则未及时更新到最新变种
                 
根本原因:
1. 邮件安全网关沙箱规则延迟更新
2. 员工安全意识培训不足
3. 缺少文档宏执行告警监控
```

### 事后报告模板

```markdown
## 安全事件事后报告

### 事件概况
- 事件编号: IR-2026-005
- 严重等级: P1 (紧急)
- 发现时间: 2026-05-26 09:35
- 影响范围: 2 台服务器 + 1 台员工终端
- 数据泄露: 未确认数据泄露

### 时间线
| 时间 | 事件 |
|------|------|
| 09:15 | 初始访问 (钓鱼) |
| 09:30 | 横向移动 |
| 09:35 | EDR 告警触发 |
| 09:40 | 终端隔离 |
| 10:00 | 根除开始 |
| 11:30 | 恢复完成 |

### Root Cause
1. 邮件网关未检测到新变种文档宏
2. 员工安全意识不足
3. AMSI 配置未启用有害脚本检测

### 改进行动项
| # | 行动项 | 负责人 | 截止日期 |
|---|--------|--------|---------|
| 1 | 更新邮件网关宏检测规则 | 安全运维 | 1 周 |
| 2 | 全员安全意识培训 (钓鱼 + 宏) | HR | 1 个月 |
| 3 | 部署 PS ScriptBlock Logging GPO | IT | 2 周 |
| 4 | 增加 4104/4688 SIEM 告警规则 | SOC | 1 周 |
```

### 经验教训总结

```
本次事件做得好的:
- EDR 规则有效检测到横向移动
- 响应团队 5 分钟内启动
- 沟通及时、流程顺畅

需要改进的:
- 邮件网关规则更新滞后
- AMSI 配置未覆盖所有终端
- 员工安全意识培训频率不足

行动计划:
1. 短期 (1周内): 规则更新 + 配置修复
2. 中期 (1个月): 培训 + 新增检测规则
3. 长期 (3个月): 全面渗透测试 + 紫队演练
```

## Playbook 示例

### 勒索软件应答 Playbook

```
1. 确认
   ├── 是否检测到文件加密
   ├── 是否有勒索信 (README.txt)
   ├── 是否有文件扩展名变更
   └── EDR 是否有相关告警

2. 立即行动
   ├── 隔离受影响主机 (网络断开)
   ├── 禁用受影响用户账户
   ├── 通知管理层和法务
   └── 启动 P1 IR 流程

3. 遏制
   ├── 确认所有受影响主机范围
   ├── 关闭受影响主机 (取证前)
   ├── 阻止 C2 域名/IP (防火墙)
   └── 收集勒索信息 (赎金金额、联系方式)

4. 分析
   ├── 提取恶意软件样本
   ├── 确定变种和家族
   ├── 检查是否存在解密工具
   └── 评估备份完整性

5. 恢复
   ├── 从干净的备份恢复
   ├── 确认无隐藏后门
   ├── 逐步上线恢复业务
   └── 强化安全配置
```
