# 紫队方法论

## 概述

紫队并非一个独立的安全团队，而是一种协作模式——红队和蓝队打破壁垒，共享信息和目标，共同提升组织的安全防御能力。紫队的核心目标是让红队攻击技术转化为蓝队可操作的检测和防御能力，最终实现安全能力的持续提升。

## 紫队协作模型

### 传统模式 vs 紫队模式

| 维度 | 传统红蓝对抗 | 紫队协作模式 |
|------|------------|-------------|
| 信息共享 | 零信息共享（盲测） | 完全透明（红队提前告知 TTP） |
| 目标 | 红队胜出 vs 蓝队防住 | 共同发现防御盲区并改进 |
| 衡量指标 | 红队成功次数 | 检测覆盖率、检测时间、响应时间 |
| 频率 | 季度/年度 | 持续化（每2-4周一次演练） |
| 输出成果 | 渗透测试报告 | 检测规则 + 响应流程优化 |
| 心理状态 | 对立的"猫鼠游戏" | 协作的"共同提升" |

### RTA 模型 (Red Team Activity → Blue Team Detection)

紫队训练通常基于 RTA 循环：

```
1. 选择 ATT&CK 技术
       │
2. 红队执行选定技术模拟攻击
       │
3. 蓝队进行检测和响应
       │
4. 双方复盘: 检测到吗? 响应效果如何?
       │
5. 改进: 新增/优化检测规则、完善响应流程
       │
6. 再次验证: 下一次演练中验证改进效果
       ↓
   回到步骤 1
```

## 紫队演练框架

### 标准紫队演练流程

| 阶段 | 活动 | 输出 | 时长占比 |
|------|------|------|---------|
| 计划 | 选择测试技术、定义成功标准、确定测试环境 | 演练计划、范围定义 | 20% |
| 准备 | 配置攻击工具、部署检测环境、基线确认 | 测试环境就绪 | 15% |
| 执行 | 红队执行攻击、蓝队记录检测情况 | 攻击日志、检测记录 | 35% |
| 评审 | 双方共同回顾、差距分析 | 差距分析报告 | 20% |
| 改进 | 创建新检测规则、优化响应动作、更新文档 | 规则更新、Playbook 更新 | 10% |

### 紫队演练模板

```markdown
## 紫队演练 #2026-05-27

### 测试技术
- ATT&CK ID: T1059.001 — PowerShell
- ATT&CK ID: T1003.001 — LSASS Memory (Mimikatz)

### 成功标准
1. 蓝队检测 PowerShell 编码命令执行
2. 蓝队检测 LSASS 异常进程访问
3. 响应时间 < 30 分钟

### 红队执行计划
- 阶段 1: 通过钓鱼获得初始访问
- 阶段 2: 执行 PowerShell 绕过 AMSI
- 阶段 3: 使用 Mimikatz dump 凭据

### 结果记录

| 技术 | 检测时间 | 检测源 | 是否告警 | 响应动作 |
|------|---------|--------|---------|---------|
| T1059.001 | 23s | Sysmon EID 1 + EID 4104 | 是 | 终端隔离 |
| T1003.001 | 5min | EDR 行为分析 | 是 (延迟) | 进程终止 |

### 改进项
1. 缩短 Mimikatz 检测时间 → 添加 ProcessAccess 规则
```

## 核心指标

### 紫队演练的关键度量指标

| 指标 | 定义 | 计算公式 | 目标值 |
|------|------|---------|-------|
| 检测率 | 红队执行的所有技术中被蓝队检测到的比例 | 检测数 / 执行总数 × 100% | > 90% |
| 平均检测时间 | 从攻击开始到首次检测到的时间 | Σ(检测时间) / 检测数 | < 5 分钟 |
| 平均响应时间 | 从检测到首次响应的时间 | Σ(响应时间) / 响应数 | < 15 分钟 |
| ATT&CK 覆盖率 | 组织的检测能力覆盖的 ATT&CK 技术比例 | 可检测技术 / 相关技术总数 × 100% | > 70% |
| 误报率 | 单次演练中的误告警比例 | 误报数 / 告警总数 × 100% | < 10% |
| MTTD | Mean Time to Detect | 同上平均检测时间 | < 30 分钟 |
| MTTR | Mean Time to Respond | 同上平均响应时间 | < 1 小时 |

## 自动化紫队平台

### CALDERA

MITRE 开发的自动化红队/紫队平台：

```
CALDERA 核心架构:
  ┌─────────────┐
  │  CALDERA    │ ← REST API → 自定义插件
  │  Server     │
  └──────┬──────┘
         │
  ┌──────┴──────┐
  │  Agent      │ ← Sandcat (Go), Manx (Go), Ragnarok (C)
  └──────┬──────┘
         │
  ┌──────┴──────┐
  │  Adversary  │ ← 基于 ATT&CK 构建的攻击配置文件
  │  Profiles   │
  └─────────────┘
```

```yaml
#  adversary profile 示例: APT29 模拟
id: 29c18e7a-86b4-4ac5-b3e4-c9e18d42d1fe
name: APT29 Simulation
description: Emulates APT29 (Cozy Bear) TTPs

atomic_ordering:
  # 阶段 1: 初始访问
  - a13b0e23-5fef-4b32-8b7e-7b5a6b7e8c9d  # Spearphishing Link
  
  # 阶段 2: 执行
  - 9a3f2e1d-4c5b-6a7b-8c9d-0e1f2a3b4c5d  # PowerShell Download
  
  # 阶段 3: 持久化
  - b2c3d4e5-f6a7-4b8c-9d0e-1f2a3b4c5d6e  # Scheduled Task
  
  # 阶段 4: 凭据窃取
  - c3d4e5f6-a7b8-4c9d-0e1f-2a3b4c5d6e7f  # LSASS Dump
```

### SCYTHE

商业紫队平台，提供 GUI 驱动的攻击模拟：

- 预构建的 ATT&CK 攻击模块库
- 支持自定义反制措施开发
- 自动生成紫队报告
- 与 SIEM 集成验证检测规则

### AttackIQ

专注于安全控制效果验证的自动化平台：

| 功能 | 说明 | 使用场景 |
|------|------|---------|
| 攻击模拟 | 模拟真实攻击场景验证安全控制 | 验证 EDR/NGFW 配置 |
| 持续验证 | 定期自动执行攻击模拟 | 合规、安全审计 |
| 对标评估 | 与行业基准进行比较 | CISO 报告 |
| 补救跟踪 | 跟踪改进措施落地情况 | 安全改进项目管理 |

## 紫队成熟度模型

| 级别 | 名称 | 参与度 | 频率 | 自动化 | 数据驱动 |
|------|------|--------|------|--------|---------|
| Level 1 | 临时 | 偶尔的联合演练 | 年度 | 无 | 无 |
| Level 2 | 计划 | 定期红蓝讨论 | 季度 | 有限 | 手动统计 |
| Level 3 | 集成 | 持续协作流程 | 月度 | 半自动化 | 指标驱动 |
| Level 4 | 持续 | 全量 ATT&CK 覆盖 | 每 2 周 | 自动化 | 趋势分析 |
| Level 5 | 主动 | 预测性防御验证 | 持续 | AI 辅助 | 预测建模 |
