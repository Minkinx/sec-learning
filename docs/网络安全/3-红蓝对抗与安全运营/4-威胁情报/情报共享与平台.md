# 情报共享与平台

## 概述

威胁情报的价值在很大程度上取决于共享的范围和效率。单打独斗的威胁情报无法应对现代网络威胁的规模和速度。情报共享平台和标准化协议使得跨组织、跨行业的情报协作成为可能。

## STIX / TAXII 标准

STIX（Structured Threat Information Expression）和 TAXII（Trusted Automated Exchange of Intelligence Intelligence）是 OASIS 组织维护的威胁情报国际标准，是目前最广泛采用的情报共享协议。

### STIX 2.1 核心对象

STIX 使用领域驱动设计（DDD），定义了一系列域对象（Domain Objects）和关系对象（Relationship Objects）：

| STIX 对象类型 | 描述 | 关键属性 |
|---------------|------|---------|
| Indicator | 指示器（IoC 描述） | pattern, pattern_type, valid_from |
| Threat Actor | 威胁行为体 | threat_actor_type, aliases, roles |
| Attack Pattern | 攻击模式 (ATT&CK 映射) | name, description, kill_chain_phases |
| Campaign | 攻击活动系列 | name, objective, first_seen |
| Malware | 恶意软件 | name, malware_types, is_family |
| Intrusion Set | APT 组织 | name, description, aliases |
| Course of Action | 响应措施 | name, description, action |
| Report | 情报报告 | name, published, object_refs |
| Observed Data | 观察到的技术数据 | first_observed, last_observed, objects |

### TAXII 传输协议

TAXII 定义了四种核心服务：

| 服务 | HTTP 方法 | 端点示例 | 说明 |
|------|----------|---------|------|
| Discovery | GET | /taxii2/ | 获取 API 根信息和支持的端点 |
| Collections | GET | /taxii2/collections/ | 列出可用数据集合 |
| Collection | GET | /taxii2/collections/{id}/ | 获取集合元数据 |
| Add Objects | POST | /taxii2/collections/{id}/objects/ | 向集合添加 STIX 对象 |
| Get Objects | GET | /taxii2/collections/{id}/objects/ | 从集合获取 STIX 对象 |
| Get Object | GET | /taxii2/collections/{id}/objects/{id}/ | 获取单个对象 |

### TAXII 客户端配置

```python
# 使用 taxii2-client 库拉取情报
from taxii2client import Server

# TAXII 2.0 服务器连接
server = Server("https://taxii.example.com/taxii2/")
api_root = server.api_roots[0]

# 列出集合
for collection in api_root.collections:
    print(f"Collection: {collection.title} ({collection.id})")

# 从指定集合获取最新情报
collection = api_root.collections[0]
envelope = collection.get_objects()

for obj in envelope.objects:
    if obj["type"] == "indicator":
        print(f"Indicator: {obj['name']}")
        print(f"Pattern: {obj['pattern']}")
        print(f"Valid from: {obj['valid_from']}")
```

## MISP — 恶意软件信息共享平台

MISP（Malware Information Sharing Platform）是目前最广泛使用的开源威胁情报平台。

### 核心架构

```
MISP 平台核心功能:

├── IOC 管理
│   ├── 创建/编辑/共享事件 (Events)
│   ├── 属性 (Attributes) — IOC 详细信息
│   ├── 对象 (Objects) — 关系属性组
│   └── 标签 (Tags) — 分类和标记
│
├── 自动关联
│   ├── 关联分析 (Correlation)
│   ├── 警告列表 (Warning Lists)
│   └── 分类法 (Taxonomies)
│
├── 导入/导出
│   ├── 自动 Feed 导入
│   ├── STIX/CSV/OpenIOC/JSON 导出
│   ├── SIEM 集成 (Elasticsearch/Splunk)
│   └── API 批量操作
│
├── 协作
│   ├── 同步 (Sync) — 与其他 MISP 实例共享
│   ├── 建议 (Proposals) — 建议修改事件
│   ├── 讨论 (Discussions) — 事件评论
│   └── 群组 (Sharing Groups) — 控制访问权限
│
├── 工作流
│   ├── 事件管理生命周期
│   ├── 发布审批流程
│   └── 通知机制
│
└── 扩展
    ├── 模块 (MISP Modules) — 第三方集成
    ├── 导出模块 — 自定义格式
    └── 导入模块 — 自动处理
```

### MISP Taxonomy 分类法

MISP 使用分类法（Taxonomies）标记事件的类型和敏感度：

| 分类 | 标签示例 | 用途 |
|------|---------|------|
| TLP | tlp:red/amber/green/white | 信息共享级别 |
| PAP | pap:white/green/amber/red | 操作使用限制 |
| Admiralty Scale | source-reliability:A-F, information-credibility:1-6 | 置信度评估 |
| CERT-XLM | cert-xlm:incident, cert-xlm:phishing | 事件类型分类 |
| ENISA | enisa:malware, enisa:ddos | 欧盟分类法 |
| MISP | misp:false-positive, misp:automated | 平台内部标记 |

### MISP 同步架构

```
组织 A MISP       组织 B MISP        组织 C MISP
    │                │                  │
    │ ── Sync ───→   │   ←── Sync ──   │
    │ ←── Sync ──   │   ── Sync ──→   │
    │                │                  │
        ↓                ↓                  ↓
    本地 IOC          本地 IOC           本地 IOC
        │                │                  │
        ↓                ↓                  ↓
    SIEM/SOAR         SIEM/SOAR          SIEM/SOAR
```

## OpenCTI — 开源威胁情报平台

OpenCTI 由 Filigran 开发，采用知识图谱驱动的 CTI 平台。

### 核心特性

| 特性 | 说明 | MISP 对比 |
|------|------|-----------|
| 数据模型 | 知识图谱 (STIX 2.1 原生) | 事件-属性 (扁平) |
| 可视化 | 关系图、Timeline、仪表盘 | 有限 |
| 连接器 | 50+ Connectors | MISP Modules |
| 工作流 | Case 管理、任务分配 | 有限 |
| 集成 | MISP, MITRE, Techdocs | 同 |

### OpenCTI 连接器

```
OpenCTI 连接器生态:

├── 外部收集
│   ├── MISP (双向同步)
│   ├── MITRE ATT&CK (自动导入)
│   ├── VirusTotal (文件/IP 查询)
│   ├── AlienVault OTX
│   ├── AbuseIPDB
│   ├── Shodan/Censys
│   └── Twitter/Telegram
│
├── 分析
│   ├── YARA (文件规则扫描)
│   ├── Sigma (规则导入)
│   └── STIX 导出/导入
│
└── 导出
    ├── SIEM 导出 (Elasticsearch/Splunk)
    ├── 告警推送 (Webhook/TheHive)
    └── 自定义脚本
```

## ISACs — 信息共享与分析中心

ISACs 是行业特定的信息共享组织，成员在信任框架内共享威胁情报。

### 主要 ISACs

```
全球主要 ISAC:
├── FS-ISAC (金融服务)
│   ├── 成员: 摩根大通、高盛、汇丰...
│   └── 特点: 最成熟的 ISAC，双向共享平台
│
├── IT-ISAC (信息技术)
│   ├── 成员: 微软、Google、思科...
│   └── 特点: 产品漏洞协调
│
├── Health-ISAC (医疗健康)
│   ├── 成员: 医院、药企、医疗设备商
│   └── 特点: 关注医疗器械安全
│
├── Auto-ISAC (汽车)
│   ├── 成员: 特斯拉、丰田、大众...
│   └── 特点: 车辆网络安全
│
├── Energy-ISAC (能源)
│   ├── 成员: 电力、石油、天然气企业
│   └── 特点: OT/ICS 安全
│
└── R-ISC (零售)
    └── 特点: 支付安全、POS 威胁
```

### ISAC 成熟度模型

| 级别 | 描述 | 共享类型 | 信任模型 |
|------|------|---------|---------|
| 1 | 信息接收 | 单向消费 | 开放 |
| 2 | 被动共享 | 有限制共享 | 审核制 |
| 3 | 主动共享 | 成员贡献为主 | 会员制 |
| 4 | 自动化共享 | API 自动化同步 | 联盟制 |
| 5 | 预测性共享 | AI 辅助预测 | 生态 |

## 情报共享挑战

| 挑战 | 描述 | 应对方案 |
|------|------|---------|
| 信任问题 | 担心情报质量、被追溯来源 | 匿名化处理、引入评分机制 |
| 分类等级 | 如何确定情报的敏感度 | 使用 TLP + PAP 标准化标签 |
| 自动化差距 | 手动处理无法规模扩展 | STIX/TAXII 标准化接口 |
| 法律壁垒 | 跨境数据共享的法律障碍 | 匿名化、聚合后共享 |
| 误报传播 | 一个误报被多家消费 | 共识验证、多源交叉确认 |
| 价值不对等 | 部分组织只消费不贡献 | 建立互惠机制、参与度激励 |
| 工具兼容性 | 不同平台格式不统一 | 采用标准 STIX 格式转换 |
