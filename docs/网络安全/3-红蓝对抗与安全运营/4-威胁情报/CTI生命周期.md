# CTI生命周期

## 概述

网络威胁情报（Cyber Threat Intelligence, CTI）是基于证据的知识，包括关于现有或新兴的对资产的威胁和危害的上下文、机制、指标、含义和可操作的建议。CTI 的生命周期管理是威胁情报运营的核心框架，它确保情报活动有方向、有产出、可衡量。

## CTI 生命周期六阶段

威胁情报生命周期（Intelligence Cycle）由六个阶段组成，形成一个持续改进的闭环：

```
Direction ←── Feedback
    ↓              ↑
Collection ──→ Processing ──→ Analysis ──→ Dissemination
```

| 阶段 | 核心活动 | 关键问题 | 产出 |
|------|---------|---------|------|
| 1. 方向 (Direction) | 确定情报需求、优先级 | 我们需要知道什么？ | 情报需求文档 |
| 2. 收集 (Collection) | 从多种来源获取原始数据 | 数据从哪里来？ | 原始数据/日志 |
| 3. 处理 (Processing) | 格式化、提取、标准化 | 如何使数据可用？ | 结构化的 IOC |
| 4. 分析 (Analysis) | 上下文关联、归因判断 | 这意味着什么？ | 情报分析报告 |
| 5. 分发 (Dissemination) | 推送报告、集成系统 | 谁需要知道？ | 报告/Feeds/告警 |
| 6. 反馈 (Feedback) | 评估价值、调整方向 | 我们做得对吗？ | 改进计划 |

## 阶段 1: 方向 (Direction)

确定情报收集和分析的目标，回答"我们需要知道什么"这个核心问题。

### 情报需求 (Intelligence Requirements)

```
PIRs (Priority Intelligence Requirements):
   级别 1: 组织级优先情报需求
     └── 直接影响业务的威胁 (如勒索软件、供应链攻击)

   级别 2: 战术级情报需求
     └── 特定攻击手法和工具 (如特定 RAT、C2 框架)

   级别 3: 运营级情报需求
     └── 具体 IOC 和数据 (如 IP、Hash、域名)
```

### 建立情报收集计划

```markdown
## 季度情报需求文档模板

### PIR-01: 勒索软件威胁
- 关注家族: LockBit, BlackCat (ALPHV), Clop, 8Base, Akira
- 行业相关: [金融/制造/科技]
- 收集来源: 暗网论坛、勒索谈判聊天、EDR 遥测
- 优先级: 高

### PIR-02: 供应链攻击
- 关注供应商: [核心供应商清单]
- 关注类型: 第三方软件漏洞、特权访问滥用
- 收集来源: 供应商安全公告、CVE 监控
- 优先级: 中

### PIR-03: 针对特定技术的攻击
- 关注技术: [云服务/SaaS/特定中间件]
- 收集来源: 安全社区、漏洞库、WAF 日志
- 优先级: 中
```

## 阶段 2: 收集 (Collection)

### 情报来源分类

| 来源类型 | 缩写 | 描述 | 举例 |
|---------|------|------|------|
| 开源情报 | OSINT | 公开可获取的信息 | 社交媒体、论坛、安全博客 |
| 人力情报 | HUMINT | 从人际交往获取 | 会议交流、行业关系、线人 |
| 技术情报 | TECHINT | 技术手段获取 | 蜜罐数据、恶意软件分析 |
| 信号情报 | SIGINT | 通信截获 | 通信模式分析 |
| 暗网情报 | DARKINT | 暗网数据 | 暗网论坛、黑客市场 |
| 内部遥测 | INTERNAL | 内部安全数据 | EDR/SIEM 日志、IDS 告警 |

### 自动化收集架构

```
数据收集管道:

├── OSINT 源
│   ├── Twitter/Telegram (安全研究者频道)
│   ├── 安全博客 (BleepingComputer, TheHackerNews)
│   ├── CVE 数据库 (NVD, MITRE)
│   ├── 恶意软件仓库 (MalwareBazaar, VirusTotal)
│   └── 泄露监控 (DeHashed, IntelX)
│
├── 技术源
│   ├── 蜜罐 (Cowrie, T-Pot)
│   ├── 恶意软件沙箱 (Cuckoo, any.run API)
│   ├── 被动 DNS (RiskIQ, Censys)
│   └── 扫描器 (Shodan, Censys)
│
└── 内部源
    ├── SIEM 告警
    ├── EDR 告警
    └── 邮件安全网关日志
```

## 阶段 3: 处理 (Processing)

将原始数据转换为机器可读的结构化格式。

### IOC 提取与标准化

原始数据示例（恶意软件分析报告）：

```
原始 Text: "The malware communicates with 185.234.72.18 on port 443
and hashes to 3f4b6d7e8f9a0b1c2d3e4f5a6b7c8d9e"

├── 提取 → IP: 185.234.72.18
├── 提取 → Port: 443
└── 提取 → MD5: 3f4b6d7e8f9a0b1c2d3e4f5a6b7c8d9e

标准化为 STIX 格式:
{ "type": "indicator",
  "pattern": "[ipv4-addr:value = '185.234.72.18']",
  "pattern_type": "stix",
  "valid_from": "2026-05-26T00:00:00Z",
  "indicator_types": ["malicious-activity"]
}
```

## 阶段 4: 分析 (Analysis)

分析阶段是 CTI 生命周期中价值最高的环节。将处理后的数据结合上下文信息进行分析，产出可操作的情报。

### 分析层次

| 层次 | 名称 | 问题回答 | 产出类型 |
|------|------|---------|---------|
| Tier 1 | 战术级 (Tactical) | 攻击方式和工具是什么？ | IOC 列表、Hash 告警 |
| Tier 2 | 运营级 (Operational) | 攻击者的目标和时间线？ | TTP 分析、攻击活动报告 |
| Tier 3 | 战略级 (Strategic) | 对组织的影响和趋势？ | 态势报告、趋势分析 |

### 置信度评估 — Admiralty System

来源可靠性：

| 等级 | 来源可靠性 | 说明 |
|------|-----------|------|
| A | 完全可靠 | 历史数据 100% 准确 |
| B | 通常可靠 | 历史准确率高 |
| C | 偶尔可靠 | 有一定准确率 |
| D | 不可靠 | 历史准确率低 |
| E | 未知 | 无历史参考 |
| F | 无法判断 | 数据不可用 |

信息可信度：

| 等级 | 信息可信度 | 说明 |
|------|-----------|------|
| 1 | 确认 | 多方独立证实 |
| 2 | 很可能真实 | 证据支持较充足 |
| 3 | 可能真实 | 有一定证据 |
| 4 | 可能不真实 | 证据存疑 |
| 5 | 很可能不真实 | 反面证据较多 |
| 6 | 无法判断 | 无法评估真伪 |

### 归因分析框架

```
归因分析评估维度:

├── 技术证据 (权重 50%)
│   ├── 恶意软件代码相似度
│   ├── C2 基础设施关联性
│   ├── 使用的工具和技术
│   └── 共享的代码签名证书
│
├── 运营证据 (权重 30%)
│   ├── 攻击时间窗口 (工作日/节假日)
│   ├── 目标选择模式
│   ├── 使用的语言和文化线索
│   └── 地缘政治背景
│
└── 战略证据 (权重 20%)
    ├── 攻击者的战略目标匹配
    ├── 历史攻击模式比对
    └── 政治/经济动机分析
```

## 阶段 5: 分发 (Dissemination)

将分析结果以适当格式分发给利益相关方。

### 分发渠道

```
分发给:
├── 安全运营团队 (SOC)
│   └── 格式: STIX/TAXII Feeds, SIEM 规则更新
│
├── 漏洞管理团队
│   └── 格式: CVE 影响评估, 补丁优先级建议
│
├── 事件响应团队
│   └── 格式: 告警时序, IOC 清单
│
├── 管理层 / CISO
│   └── 格式: 战略报告, KPI 仪表盘
│
└── IT 运维团队
    └── 格式: 补丁通知, 配置变更建议
```

## 阶段 6: 反馈 (Feedback)

闭环评估 CTI 产品的价值和有效性。

### 度量指标

| 指标 | 说明 | 计算方式 |
|------|------|---------|
| 情报可操作性 | 产生可操作响应的情报比例 | 可操作情报数 / 总情报数 |
| 检测贡献率 | CTI 触发的检测占比 | CTI 告警数 / 总告警数 × 100% |
| 精准度 | 正确告警比例 | True Positive / (TP + FP) × 100% |
| 时效性 | 情报到防御部署的时间 | 情报发布时间 - 检测部署时间 |

## IOC vs IOA 与 Pyramid of Pain

### IOC (Indicator of Compromise)

IOC 是攻击后留下的痕迹证据：

- 文件哈希 (MD5, SHA1, SHA256)
- IP 地址和域名
- 注册表键值
- 文件名和路径
- 互斥体名称

### IOA (Indicator of Attack)

IOA 是攻击过程中的行为模式：

- 异常进程链 (如 Word.exe → PowerShell.exe → rundll32.exe)
- 异常的登录时间/地点
- 不寻常的网络连接模式
- API 调用序列异常

### 金字塔 (Pyramid of Pain)

```
               /\
              /  \             攻击者最难改变
             / TTP\            策略、技术和程序
            /______\
           /\      /\          工具 (Mimikatz, Cobalt Strike)
          /  \    /  \
         /工具\  /恶意软件\
        /______\/__________\   网络/域名/IP
       /\      /\      /\  \
      /  \    /  \    /  \  \  哈希值
     /域名 \ / IP  \ / 哈希  \
    /______\/______\/________\  攻击者最容易改变
```

## TTPs (Tactics, Techniques, Procedures)

| 层级 | 定义 | 示例 | 对防御者的价值 |
|------|------|------|--------------|
| 策略 (Tactics) | 攻击目标的"为什么" | "凭据窃取" T1003 | 建立整体防御框架 |
| 技术 (Techniques) | 实现目标的"如何" | "LSASS 内存转储" T1003.001 | 设计针对性检测 |
| 流程 (Procedures) | 具体实现的"细节" | 特定版本的 Mimikatz 参数 | 精确签名检测 |
