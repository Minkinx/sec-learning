# 微隔离技术

> 微隔离（Micro-segmentation）是一种将数据中心和云环境中的网络分割为精细化安全区域的技术。不同于传统VLAN分割，微隔离以工作负载为中心，基于身份而非IP地址定义安全策略，实现东西向流量的精细化管控。

## 微隔离与传统网络分段对比

| 维度 | 传统VLAN分段 | 微隔离 |
|------|-------------|--------|
| 策略粒度 | 子网/网段级别 | 单个工作负载/容器级别 |
| 策略依据 | IP地址、VLAN标签 | 身份标签（应用名、环境、团队） |
| 动态适应 | 静态配置，变更需网络团队 | 自动适应，随工作负载迁移 |
| 东西向流量 | 同一子网内无限制 | 精细管控所有流量 |
| 部署复杂度 | 需网络设备支持VLAN | 软件定义，不依赖底层网络 |
| 变更管理 | 手动配置，易出错 | 自动化策略，CI/CD集成 |

## 微隔离的核心能力

### 1. 工作负载识别与标记

微隔离的第一步是为所有工作负载打标签，建立统一的身份体系：

```yaml
# 工作负载标签示例（Kubernetes + Calico）
apiVersion: v1
kind: Namespace
metadata:
  labels:
    environment: production
    compliance_level: pci
    team: platform
  name: payment-system
---
apiVersion: v1
kind: Pod
metadata:
  name: payment-api-7d8f9c
  labels:
    app: payment-api
    tier: backend
    data_classification: sensitive
    version: v2.1.0
spec:
  containers:
  - name: payment-api
    image: payment-api:2.1.0
```

### 2. 基于身份的访问控制

策略以工作负载身份为中心，不依赖IP地址：

```
允许策略示例：
"frontend pods" -> "backend pods" : TCP 8080  ✅ 允许
"frontend pods" -> "database pods" : TCP 3306  ❌ 拒绝
"cronjob pods"  -> "database pods" : TCP 3306  ✅ 允许
```

### 3. 内核级防火墙执行

#### iptables实现

Kubernetes NetworkPolicy（底层通过iptables实现）：

```yaml
# Kubernetes NetworkPolicy示例
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: payment-api-policy
spec:
  podSelector:
    matchLabels:
      app: payment-api
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          team: frontend
    - podSelector:
        matchLabels:
          app: web-gateway
    ports:
    - protocol: TCP
      port: 8080
  egress:
  - to:
    - podSelector:
        matchLabels:
          app: payment-db
    ports:
    - protocol: TCP
      port: 3306
  - to:
    - namespaceSelector:
        matchLabels:
          name: monitoring
    - podSelector:
        matchLabels:
          app: prometheus
    ports:
    - protocol: TCP
      port: 9090
```

#### eBPF/Cilium实现

Cilium基于eBPF的微隔离具有更高的性能和更细的粒度：

```yaml
# CiliumNetworkPolicy示例（L3-L7策略）
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: secure-payment-api
spec:
  endpointSelector:
    matchLabels:
      app: payment-api
  ingress:
  - fromEndpoints:
    - matchLabels:
        app: web-gateway
        env: prod
    toPorts:
    - ports:
      - port: "8080"
        protocol: TCP
      rules:
        http:
        - method: "POST"
          path: "/api/v1/payments"
        - method: "GET"
          path: "/api/v1/payments/health"
        - method: "GET"
          path: "/api/v1/payments/status"
  egress:
  - toEndpoints:
    - matchLabels:
        app: payment-db
    toPorts:
    - ports:
      - port: "3306"
        protocol: TCP
  - toEntities:
    - kube-apiserver
    toPorts:
    - ports:
      - port: "6443"
        protocol: TCP
```

**eBPF相比iptables的优势**：

| 能力 | iptables | eBPF (Cilium) |
|------|---------|---------------|
| 性能 | 线性规则匹配，规则多时延迟高 | 哈希查找，O(1)复杂度 |
| 可见性 | 仅连接追踪 | 全量可见性（网络、系统调用） |
| L7过滤 | 不支持 | 支持HTTP/gRPC/Kafka协议 |
| 可编程性 | 固定协议栈处理 | 可编程内核扩展 |
| 规则容量 | 数千条后性能下降 | 数万条规则性能稳定 |

## 主流微隔离产品

### VMware NSX

VMware NSX是虚拟化环境中微隔离的先驱：

```powershell
# NSX 分布式防火墙策略配置（PowerCLI）
$category = "Application"
$section = Get-NsxFirewallSection -Name "PCI-Environment"

# 创建安全组
$webTier = New-NsxSecurityGroup -Name "Web-Tier" -Description "Web servers"
$appTier = New-NsxSecurityGroup -Name "App-Tier" -Description "Application servers"
$dbTier = New-NsxSecurityGroup -Name "DB-Tier" -Description "Database servers"

# 创建规则：仅允许Web->App->DB的流量
New-NsxFirewallRule -Section $section -Name "Allow Web to App" `
    -Source $webTier -Destination $appTier -Service "TCP 8080" -Action Allow

New-NsxFirewallRule -Section $section -Name "Allow App to DB" `
    -Source $appTier -Destination $dbTier -Service "TCP 3306" -Action Allow

# 默认拒绝所有其他流量
New-NsxFirewallRule -Section $section -Name "Default Deny" `
    -Direction Any -Action Drop -Logged
```

### Illumio Adaptive Security Platform

Illumio专注于自适应微隔离，通过标签系统实现策略自动化：

```
Illumio的关键概念：

┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   Workload    │    │     Label    │    │     Rule     │
│ IP: 10.0.1.5  │───►│ role=web     │───►│ web allow tcp/80  │
│ Pod: app-v1   │    │ env=prod    │    │ web allow tcp/443 │
│              │    │ app=payment │    │ web deny tcp/3306 │
└─────────────┘    └─────────────┘    └─────────────┘
```

Illumio的标签系统允许自动化策略生成：

```
角色: web - 允许 TCP 80,443; 允许访问 role=app 的 TCP 8080
角色: app - 允许访问 role=db 的 TCP 3306
环境: prod - 只允许来自同环境的工作负载访问
数据中心: dc1 - 遵循地域合规策略
```

### Cisco ACI

Cisco ACI（Application Centric Infrastructure）基于意图的网络策略模型：

```
ACI策略模型：
应用网络配置文件（ANP） -> 端点组（EPG） -> 合约（Contract） -> 过滤器（Filter）

示例：
EPG "Web" -> Contract "Consume Web" -> EPG "App" (TCP 8080)
EPG "App" -> Contract "Consume DB" -> EPG "DB" (TCP 3306)
EPG "DB" -> Contract "DB Admin" -> EPG "Admin" (TCP 22, 5432)
```

## 微隔离策略设计

### 最小权限原则

微隔离策略应遵循反向白名单原则：默认全部拒绝，仅允许明确授权的流量。

```yaml
# 策略设计示例
# 第一阶段：可视化 - 仅监控，不阻断
monitoring_mode:
  action: LOG
  description: "记录所有流量，构建通信基线"

# 第二阶段：告警异常 - 对未授权流量告警
alert_mode:
  action: LOG_WITH_ALERT
  description: "对不允许的流量发出告警"

# 第三阶段：阻断 - 执行最小权限策略
enforce_mode:
  action: DENY
  description: "阻断未授权流量，仅允许已知通信模式"
```

### 分阶段部署策略

```
阶段1（观察期）：部署微隔离代理，只监控不阻断
  ├─ 流量可视化：发现所有工作负载之间的通信关系
  ├─ 构建流量基线：记录所有正常通信模式
  └─ 识别异常流量：发现意料之外的通信

阶段2（告警期）：开始应用策略，仅告警不阻断
  ├─ 定制白名单策略：基于阶段1的流量基线
  ├─ 告警调优：处理误报，完善策略
  └─ 异常响应预案：准备紧急回滚方案

阶段3（执行期）：正式阻断未授权流量
  ├─ 逐步执行：从非关键环境开始
  ├─ 监控阻断事件：及时发现阻断影响
  └─ 持续优化：定期更新策略以匹配业务变化
```

## 微隔离的挑战与应对

| 挑战 | 描述 | 应对策略 |
|------|------|---------|
| 策略复杂性 | 分布式环境中策略数量可能指数级增长 | 使用标签自动化策略生成，避免手动编写 |
| 性能影响 | 内核级包处理可能引入延迟 | 采用eBPF等高性能技术，避免iptables的线性匹配 |
| 调试困难 | 策略错误导致业务中断难以排查 | 完善的监控日志、hubble可视化工具 |
| 与现有网络集成 | 传统网络设备不支持微隔离 | 混合模式，Overlay网络封装 |
| 多集群管理 | Kubernetes多集群环境策略一致性问题 | 使用Cluster API和Global NetworkPolicy |

## Hubble与流量可视化

Cilium的配套组件Hubble提供实时的微隔离流量可视化：

```bash
# Hubble CLI查看微隔离策略和流量
hubble observe --namespace payment --verdict DROPPED
hubble observe --from-pod payment-api-7d8f9c --to-pod payment-db-*
hubble status --namespace payment

# 输出示例
May 26 10:30:15.123: payment-api-7d8f9c:53412 -> payment-db-abcde:3306 DROPPED (TCP)
May 26 10:30:15.456: payment-api-7d8f9c:53413 -> payment-db-abcde:3306 DROPPED (TCP)
May 26 10:30:15.789: payment-api-web-gateway:44321 -> payment-api-7d8f9c:8080 FORWARDED (TCP)
```

## 总结

微隔离是实现"东西向流量防护"的关键技术，它将网络安全从网络边界下沉到工作负载级别。在零信任架构中，微隔离承担了"策略执行点"的角色，确保即使攻击者突破了一个工作负载，也无法横向移动到其他工作负载。结合Kubernetes原生策略（NetworkPolicy）或Cilium的增强策略（eBPF），企业可以在容器化环境中高效实现微隔离。
