# 零信任核心原则

> 零信任（Zero Trust）安全模型的核心思想是"永不信任，始终验证"——不再基于网络位置给予隐式信任，而是对每一次访问请求进行独立的身份验证和授权评估。

## 零信任的核心理念

### Never Trust, Always Verify

传统的边界安全模型假设"内网=信任"，一旦攻击者突破边界，就可以在内网中横向移动。零信任完全抛弃了这一假设：

```
传统模型：
[互联网] -> [防火墙] -> [内网（信任）] -> [内网应用]
                                  攻击者一旦进入内网，即可横向移动

零信任模型：
[互联网] -> [策略执行点] --验证--> [应用]
           每个访问请求都需独立评估身份、设备、上下文
```

### Assume Breach

零信任架构的设计前提是假设攻击者已经存在于网络中。因此安全设计需要考虑：

1. **横向移动防护**：即使一个工作负载被攻陷，也不能自动信任其他工作负载
2. **最小权限访问**：每个实体只拥有完成工作所需的最小权限
3. **持续验证**：不是仅在登录时验证，而是持续评估会话安全性
4. **端到端加密**：所有通信（包括东西向流量）都需加密
5. **遥测与分析**：全面收集日志和行为数据，检测异常

## NIST SP 800-207 零信任架构七大核心原则

NIST SP 800-207 定义了零信任架构（ZTA）的七个核心原则：

### 原则1：所有数据源和计算服务都视为资源

无论是企业数据中心内的应用、云中的SaaS服务、还是IoT设备，都应视为受保护的资源，同等地接受安全策略的管控。

### 原则2：所有通信都需安全保障，无论网络位置

通信的安全级别不应取决于发起方的网络位置。这意味着：

- 内网到内网的通信（东西向流量）必须加密
- 跨数据中心的通信必须加密
- 云服务之间的API调用必须使用mTLS

```yaml
# Istio mTLS配置示例：强制Pod之间的双向TLS
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: production
spec:
  mtls:
    mode: STRICT  # 强制mTLS，禁止明文流量
```

### 原则3：资源访问按会话粒度授予

```
访问决策生命周期：
用户请求访问 -> 策略引擎评估 -> 授予会话级别访问令牌 -> 
持续评估会话风险 -> （如果风险升高）撤销访问 -> 结束会话
```

每次访问请求都是一个独立的授权事件，不得缓存或重用授权决策。

### 原则4：访问决策由动态策略决定

访问决策基于以下维度的综合评估：

```python
# 简化策略评估伪代码
def evaluate_access(user, device, resource, context):
    score = 0
    
    # 身份维度
    if user.mfa_enabled and user.risk_score < 0.3:
        score += 25
    
    # 设备维度
    if device.managed and device.compliant and device.patch_level >= CURRENT:
        score += 25
    
    # 上下文维度
    if context.location == "office" or context.vpn:
        score += 10
    elif context.location == "public_wifi":
        score -= 20  # 公共WiFi风险更高
    
    # 数据敏感度
    if resource.sensitivity == "high":
        score -= 10  # 访问高敏感数据门槛更高
    
    # 行为异常
    if context.unusual_behavior:
        score -= 30
    
    return score >= 40  # 需要达到最低分数
```

### 原则5：企业监控所有资产的安全性和完整性

所有资产都必须持续监控其安全状态和完整性：

```bash
# 设备合规性检查示例（伪代码）
设备合规检查清单：
[✓] 磁盘加密启用（BitLocker/FileVault）
[✓] 操作系统版本符合最低要求
[✓] 安全补丁已更新到最新
[✓] 防病毒软件正在运行
[✓] 防火墙已启用
[✓] 无越狱/root（移动设备）
[✓] 屏幕锁定策略符合要求
[✓] 无已知的恶意软件
```

### 原则6：所有身份验证和授权都是动态的，且在访问前严格执行

| 传统模型 | 零信任模型 |
|---------|-----------|
| 登录时验证一次 | 每次访问都验证 |
| 静态角色权限 | 动态风险评估 |
| 基于IP的白名单 | 基于身份+设备+上下文的策略 |
| 隐式信任 | 显式验证 |
| 定期权限审查 | 实时权限评估 |

### 原则7：收集信息以持续改进安全态势

通过持续收集访问日志、安全事件、合规数据，利用安全分析不断优化策略：

```sql
-- 收集策略违规事件，分析异常模式
SELECT 
    user_id, 
    resource_id, 
    COUNT(*) as access_attempts,
    AVG(risk_score) as avg_risk,
    MAX(risk_score) as max_risk
FROM access_logs
WHERE denied = true
  AND timestamp > NOW() - INTERVAL 7 DAY
GROUP BY user_id, resource_id
HAVING COUNT(*) > 10
ORDER BY access_attempts DESC;
```

## 隐式信任区的消除

传统安全架构中的隐式信任区：

| 信任区域 | 传统观念 | 零信任观点 |
|---------|---------|-----------|
| 内网 | 内网是安全的 | 内网是最危险的（攻击者突破后的乐园） |
| VPN连接 | VPN用户等于内部用户 | VPN只是加密通道，仍需验证 |
| 分公司网络 | 分公司网络连接总部，自动信任 | 分公司同样需要安全检查 |
| 合作伙伴VPN | 合作伙伴等同于内部 | 合作伙伴严格限制访问范围 |

## 策略驱动的访问控制

零信任的策略模型可以用以下公式概括：

```
Access_Decision = f(Identity, Device_Health, Context, Data_Sensitivity, Behavioral_Risk)
```

```
if Identity.validated and Device_Health.compliant and not Behavioral_Risk.anomalous:
    if Data_Sensitivity == "public":
        GRANT_ACCESS
    elif Data_Sensitivity == "internal" and Context.location == "trusted":
        GRANT_ACCESS
    elif Data_Sensitivity == "confidential":
        GRANT_ACCESS with session recording and DLP
    elif Data_Sensitivity == "restricted":
        REQUIRE additional approval and JIT privilege escalation
```

## 零信任与SASE的关系

SASE（Secure Access Service Edge）融合了网络和安全功能：

```
SASE = SD-WAN + SWG + CASB + ZTNA + FWaaS
       \_____网络层_____/  \________安全层________/
```

零信任网络访问（ZTNA）是SASE的核心安全组件，负责提供基于身份的应用层访问控制。

## 总结

零信任不是单一产品或技术，而是一种安全架构理念。其核心是将信任从网络位置转移到身份和设备上，并对每次访问进行独立的动态授权。NIST SP 800-207提供了零信任架构的权威框架，指导企业从传统的边界安全向零信任模型迁移。
