# 数据访问审计

## 概述

数据访问审计是数据安全运营的基础能力，通过对数据访问行为的全量记录与分析，实现可追溯、可问责的安全管控目标。审计体系覆盖数据从产生、存储、使用、共享到销毁的全生命周期，是企业满足 GDPR/PIPL/SOX/PCI-DSS 等合规要求的核心技术手段。

## 审计架构

企业数据访问审计体系通常采用分层架构设计：

| 层次 | 组件 | 职责 | 技术实现 |
|------|------|------|----------|
| 数据源层 | 数据库、文件服务器、云服务、应用系统 | 产生原始审计日志 | 数据库原生审计、操作系统日志、云 API 日志 |
| 采集层 | 日志采集器/Agent/Syslog 转发 | 统一采集和转发 | Fluentd/Logstash/云服务商日志导出 |
| 存储层 | 集中式日志存储 | 海量日志持久化，支持高效检索 | Elasticsearch/Splunk/ClickHouse/S3 |
| 分析层 | 关联分析引擎 | 跨数据源行为关联和异常检测 | SIEM 规则引擎 / ML 模型 |
| 展示层 | 仪表盘和报表 | 可视化展示审计结果 | Kibana/Splunk Dashboard/Grafana |

## 审计数据采集架构

审计日志的采集方式根据数据源类型和性能要求选择：

| 采集方式 | 技术实现 | 性能影响 | 适用场景 |
|----------|----------|----------|----------|
| 本地 Agent 采集 | 在数据库/服务器上部署 Agent | 低-中（CPU < 5%）| 自建数据中心 |
| 日志文件采集 | 读取数据库审计日志文件 | 低 | 支持日志文件输出的系统 |
| Syslog 转发 | 通过 Syslog 协议发送日志 | 极低 | 网络设备、Linux 服务器 |
| JDBC/ODBC 抓取 | 通过 JDBC 接口轮询 | 高（不推荐线上使用）| 应急场景 |
| API 拉取 | 通过云服务商 API 拉取 | 低-中 | 云服务（CloudTrail/Azure Monitor）|
| 网络镜像采集 | TAP/SPAN 端口镜像数据库流量 | 无（旁路模式）| 数据库审计专用设备 |

## 审计类型

| 审计类型 | 覆盖范围 | 主要数据源 | 审计频率 | 保留期限 |
|----------|----------|-----------|----------|----------|
| 数据库审计 | 数据库的 DML/DDL/DCL 操作 | 数据库原生日志、专用审计设备 | 实时/准实时 | 6个月-2年 |
| 文件服务器审计 | 文件访问、修改、权限变更 | SMB/NFS 审计日志、云存储访问日志 | 实时 | 90天-1年 |
| 应用层审计 | 应用 API 调用、业务数据操作 | 应用日志、WAF 日志、API 网关日志 | 实时 | 6个月-1年 |
| 操作系统审计 | 服务器本地文件访问、进程行为 | Syslog、Windows Event Log、auditd | 实时 | 90-180天 |
| 云平台审计 | IaaS/PaaS/SaaS 管理操作 | CloudTrail、Azure Monitor、活动日志 | 实时 | 1-7年（按法规）|

## 数据库审计

主流数据库的审计能力各有侧重：

- **SQL Server**：通过 SQL Server Audit 功能审计服务器级和数据库级事件，支持细粒度级别（SCHEMA_OBJECT_ACCESS_GROUP），可记录成功/失败的 SELECT、INSERT、UPDATE、DELETE 操作。建议启用对敏感表的 SCHEMA_OBJECT_ACCESS_GROUP 审计组，监控所有 DML 操作。
- **Oracle**：采用 Unified Auditing 架构，将传统审计和细粒度审计统一为策略化配置，支持对特权用户（SYS/SYSTEM）的强制审计。Oracle 审计可配置为基于条件的审计（Conditional Auditing），仅在高风险操作时记录以减少开销。
- **MySQL**：基于通用日志（General Query Log）或审计插件（Audit Plugin/MariaDB Audit），性能开销约 5-10%，需根据业务流量选择性开启。推荐使用 McAfee MySQL Audit Plugin，支持按用户/对象/操作级别精细配置。
- **PostgreSQL**：通过 pgaudit 扩展实现详细审计，支持会话级和对象级审计，可记录完整的 SQL 语句绑定参数。建议结合 log_statement 参数配置审计粒度。

## 文件服务器审计

Windows 环境下通过 SACL（System Access Control List）启用 SMB 审计，在安全事件日志中生成 Event ID 5145（网络共享对象访问检查）和 4663（对象访问尝试）。关键审计维度包括用户身份、源 IP、访问时间、文件路径和操作类型（读/写/删除/权限变更）。Linux 环境使用 auditd + inotify 实现文件系统审计，结合 ausearch 工具进行日志查询和分析。

## 云审计日志

- **AWS CloudTrail**：记录所有 API 调用（管理事件和数据事件），支持组织级跟踪和多区域聚合，事件持久化至 S3 并用 Athena 查询。建议开启 CloudTrail Insights 自动检测异常 API 活动。Data Events 针对 S3/Lambda/DynamoDB 的数据操作，按需启用并控制成本。
- **Azure Monitor**：整合平台日志、活动日志和资源日志，通过 Log Analytics 工作区实现集中分析，支持 KQL 查询语言。Azure 利用诊断设置（Diagnostic Settings）将资源日志路由到 Log Analytics、Storage 或 Event Hubs。
- **GCP Cloud Audit Logs**：分为管理活动审计日志（Admin Activity）、数据访问审计日志（Data Access）和系统事件审计日志（System Event），其中数据访问日志默认禁用，需要显式启用。

## SIEM 集成

将分散的数据源统一接入 SIEM 平台（Splunk/Elastic/Sentinel/QRadar），建立数据源标准化（Common Information Model），实现跨数据源的关联分析：

| 场景 | 检测逻辑 | 数据源组合 | 告警行动 |
|------|----------|-----------|----------|
| 特权用户监控 | 非工作时间高频率查询敏感表 | 数据库审计 + AD 用户属性 | 发送告警给 DBA 负责人 |
| 异常地理位置 | 从未知地理位置访问数据库 | VPN 日志 + 数据库审计 + GeoIP | 锁定账号 + MFA 挑战 |
| 权限提升探测 | 普通用户尝试访问高敏感数据 | 数据库审计 + AD 安全组 + 数据分类 | 创建事件工单 + 通知安全团队 |
| 批量数据导出 | 短时内大量数据导出操作 | 数据库审计 + DLP 事件 + 网络流 | 阻断 + 录屏 + 事件升级 |
| 合规报告 | 定期生成数据访问合规报表 | 全量审计日志 | SOX/PCI-DSS/GDPR 合规报告 |

## 合规映射

| 合规要求 | 审计需求 | 审计证据 | 保留期限 |
|----------|----------|----------|----------|
| GDPR 第30条 | 处理活动记录 | 数据访问日志、处理目的说明 | 至少3年 |
| PIPL 第51条 | 个人信息处理记录 | 个人信息的收集、使用、提供记录 | 不少于3年 |
| SOX 第404条 | 财务数据访问控制审计 | 财务系统访问日志、权限变更记录 | 7年 |
| PCI-DSS 要求10 |  cardholder 数据访问审计跟踪 | 所有访问 cardholder 数据的日志 | 至少1年（在线3个月）|
| 等级保护三级 | 安全审计要求 | 网络/主机/应用/数据库审计日志 | 不少于6个月 |

## 审计分析技术

### 关联分析

将不同类型的日志关联以发现复合攻击模式：
- **用户关联**：将 VPN 登录、数据库访问、文件操作归因到同一用户
- **时间窗口关联**：在特定时间窗口内关联多个事件（如非工作时间登录后触发数据批量导出）
- **路径关联**：追踪数据从源到目标的完整访问路径

### 行为分析

通过机器学习建立行为基线，检测偏离基线的异常访问：
- 用户历史访问模式对比（正常查询量 vs 异常高峰）
- 同群组行为对比（同一部门同事的访问模式偏差）
- 时序模式分析（工作日 vs 周末、工作时间 vs 非工作时间）

### 趋势分析

基于时间序列的聚合分析，发现长期风险趋势：
- 权限滥用趋势：特权用户数据访问量月环比变化
- 数据暴露趋势：敏感表访问频次的长期变化
- 合规趋势：不合规操作的数量变化趋势

## 最佳实践

1. **最小审计策略**：实施基于风险的最小审计策略，避免全量审计影响数据库性能。建议对核心业务表开启详细审计，对低频访问的归档表仅记录元数据变更。性能基准测试：详细审计约引入 5-15% 的性能开销。
2. **审计日志独立存储**：审计日志独立于被审计系统存储，配置不可篡改（Write-Once-Read-Many, WORM），建议写入云对象存储的不可变桶或专用的日志归档系统。审计管理员账号应与系统管理员分离。
3. **保留策略分层**：建立审计日志的分层保留策略：热存储（最近 90 天，在线可查）、温存储（90天-1年，低成本对象存储）、冷存储（1-7年，归档存储，可使用 Glacier/Deep Archive）。
4. **日志完整性校验**：对审计日志实施完整性保护，采用哈希链（Hash Chain）或数字签名确保日志未被篡改，满足法庭证据的采信要求。推荐方案：每 1000 条日志生成一个哈希校验块，链式链接。
5. **定期审计有效性测试**：每季度执行审计覆盖测试，确认所有目标系统均已纳入审计范围，审计日志完整无误。测试方法：触发已知操作后验证日志是否被正确记录和索引。
