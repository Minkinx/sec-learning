# 数据访问审计

## 概述

数据访问审计是数据安全运营的基础能力，通过对数据访问行为的全量记录与分析，实现可追溯、可问责的安全管控目标。

## 数据库审计

主流数据库的审计能力各有侧重：

- **SQL Server**：通过 SQL Server Audit 功能审计服务器级和数据库级事件，支持细粒度级别（SCHEMA_OBJECT_ACCESS_GROUP），可记录成功/失败的 SELECT、INSERT、UPDATE、DELETE 操作。
- **Oracle**：采用 Unified Auditing 架构，将传统审计和细粒度审计统一为策略化配置，支持对特权用户（SYS/SYSTEM）的强制审计。
- **MySQL**：基于通用日志（General Query Log）或审计插件（Audit Plugin/MariaDB Audit），性能开销约 5-10%，需根据业务流量选择性开启。

## 文件服务器审计

Windows 环境下通过 SACL（System Access Control List）启用 SMB 审计，在安全事件日志中生成 Event ID 5145（网络共享对象访问检查）和 4663（对象访问尝试）。关键审计维度包括用户身份、源 IP、访问时间、文件路径和操作类型（读/写/删除/权限变更）。

## 云审计日志

- **AWS CloudTrail**：记录所有 API 调用（管理事件和数据事件），支持组织级跟踪和多区域聚合，事件持久化至 S3 并用 Athena 查询。
- **Azure Monitor**：整合平台日志、活动日志和资源日志，通过 Log Analytics 工作区实现集中分析，支持 KQL 查询语言。

## SIEM 集成

将分散的数据源统一接入 SIEM 平台（Splunk/Elastic/Sentinel），建立数据源标准化（Common Information Model），实现跨数据源的关联分析。典型用例如下：

| 场景 | 检测逻辑 |
|------|----------|
| 特权用户监控 | 非工作时间高频率查询敏感表 |
| 异常检测 | 从未知地理位置访问数据库 |
| 合规报告 | 定期生成数据访问合规报表（SOX/PCI-DSS/GDPR） |

## 最佳实践

1. 实施最小审计策略，避免全量审计影响数据库性能
2. 审计日志独立存储，配置不可篡改（Write-Once-Read-Many）
3. 建立审计日志保留策略（至少 90 天在线 + 1 年归档）
