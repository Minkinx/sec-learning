# UEBA 行为分析

## 定义与原理

用户与实体行为分析（User and Entity Behavior Analytics, UEBA）通过建立主体（用户、设备、应用、数据）的正常行为基线，利用统计和机器学习算法识别偏离基线的异常行为，从而检测内部威胁、账号接管和数据泄露。UEBA 区别于传统基于规则的检测方法的核心优势在于能够发现未知威胁和低慢型攻击。

## 架构组件

UEBA 平台的典型架构包含以下组件：

| 组件 | 功能 | 实现示例 |
|------|------|----------|
| 数据采集管道 | 从多源采集行为日志，归一化为 UEBA 标准事件格式 | Kafka + Logstash + 自定义 Connector |
| 特征提取引擎 | 提取时间、频率、量级、序列等行为特征 | Python 特征工程 pipeline |
| 基线构建模块 | 基于历史数据建立多维度行为基线 | 周期性离线训练（每日/每周） |
| 异常评分引擎 | 对实时行为进行评分，输出异常分数 | 集成 Isolation Forest/Autoencoder 等模型 |
| 告警降噪模块 | 通过上下文关联和规则过滤减少误报 | 白名单 + 业务上下文 CMDB 集成 |
| 调查分析界面 | 可视化调查工具，支持时间线和关联分析 | Web 控制台 + 图分析 |
| 案件管理 | 将告警转化为可追踪的安全事件 | 集成 SOAR 工单系统 |

## 基线建立

UEBA 的基线分为三个层次：

- **历史基线**：基于主体自身的历史行为数据（过去 30/90 天的访问模式），周期性更新以反映正常变化。采用滑动窗口模型，赋予近期数据更高权重（指数加权移动平均 EWMA）。
- **同群组基线**：将相似角色（如同一部门、同一职级）的用户聚合建模，检测偏离同群组的行为（如普通员工访问高管权限）。群组划分维度包括部门、职级、地理位置、工作时间模式等。
- **时间序列基线**：考虑时间周期性（工作日 vs 周末、上班时段 vs 非工作时间），消除周期性因素的干扰。采用 STL（Seasonal-Trend Decomposition）分解趋势和季节性成分。

## 异常检测算法

| 方法 | 技术栈 | 适用场景 | 优点 | 缺点 |
|------|--------|----------|------|------|
| 统计方法 | Z-Score、MAD、分位数 | 单一维度的简单异常（如下载量暴增） | 计算快、可解释性强 | 无法处理高维特征 |
| 传统 ML | Isolation Forest、One-Class SVM | 多维特征的异常（时间+地点+频率+数据量） | 支持高维、无需标注 | 对参数敏感 |
| 深度学习 | Autoencoder、LSTM | 序列行为异常（时序模式偏离） | 捕捉复杂模式 | 训练成本高、可解释性差 |
| 图分析 | 图神经网络（GNN） | 访问关系异常（实体之间权限传播分析） | 发现关系型异常 | 图构建复杂 |
| 集成方法 | Stacking、模型投票 | 综合多种算法提高检测率 | 精度高 | 运维复杂度高 |

## 数据源与实体类型

UEBA 引擎整合的信号包括：身份认证日志（SSO/VPN）、数据访问日志、DLP 事件、邮件元数据、终端行为数据、HR 系统数据（入离职/转岗）、物理访问控制日志。

核心实体类型及分析维度：

| 实体类型 | 分析维度 | 异常行为示例 |
|----------|----------|-------------|
| 用户 | 登录时间/地点、访问数据量、访问频次、失败登录次数、打印活动、与 HR 记录偏差 | 离职前批量下载文件、非正常工作时间登录 |
| 设备 | 操作系统/补丁版本、已装软件、网络连接目标、进程启动链 | 设备突然连接未知 C2 域名 |
| 应用/服务 | API 调用频率、数据返回量、错误率、调用来源 IP | 服务账号异常提升权限调用 |
| 数据 | 分类标签、访问者身份、最近修改时间、复制移动次数 | 高敏感数据被非授权角色访问 |

## 异常评分与阈值设定

UEBA 系统针对每个行为事件计算异常评分（Anomaly Score），通常归一化到 0-100 范围。评分高于预设阈值的产生告警。阈值设定策略：

| 策略 | 方法 | 适用场景 |
|------|------|----------|
| 固定阈值 | 对所有实体使用统一阈值（如评分>80 告警） | 初期部署，简化运维 |
| 动态阈值 | 基于实体的历史评分分布自动调整阈值 | 成熟期，适应不同行为特征 |
| 百分位阈值 | 设定评分处于前 x% 的实体产生告警 | 资源约束明确，控制告警量 |

## 告警研判

UEBA 告警需经过三级研判流程：

1. **自动降噪**：过滤已知误报（白名单 + 业务上下文判定），通过 CMDB 集成判断告警涉及的资产是否属于正常运维活动。
2. **分析师调查**：关联上下文数据（登录历史、访问记录、工单系统），使用时间线视图还原事件全貌，在 15 分钟内完成初判。
3. **升级处置**：确认恶意行为后移交 IR 团队，按事件严重等级启动响应流程。

## 使用场景案例

| 场景 | UEBA 检测方法 | 价值 | 误报控制 |
|------|-------------|------|----------|
| 离职前数据窃取 | 检测用户下载量在离职前 30 天的异常增长、打印量激增、访问非工作范围的数据 | 防止知识产权流失 | 联合 HR 系统确认离职状态，标记为高风险 |
| 账号劫持 | 检测登录 IP 地理位置的异常变化、登录设备指纹突变、异常时间的登录 | 减少账户被盗损失 | 结合 MFA 挑战确认，避免孤立告警 |
| 内部特权滥用 | 检测 DBA 非工作日操作生产库、超出职责范围的数据访问、大量导出操作 | 保护核心数据资产 | 对比运维排班计划和变更工单调优 |
| 恶意应用授权 | 检测 OAuth 应用授权激增、API 调用模式的突变、异常的数据共享范围 | 防止数据通过第三方应用泄露 | 与 SSPM 联动确认应用风险评分 |

## 工具对比

| 平台 | 核心能力 | 数据源要求 | 部署方式 | 定价模型 |
|------|----------|-----------|----------|----------|
| Splunk UBA | 丰富 ML 算法库、与 Splunk ES 深度集成 | 需 Splunk 平台支撑 | 私有化或云 | 按数据量 |
| Microsoft UEBA | 与 Azure AD/Defender/Office 365 深度集成 | Microsoft 365 生态 | SaaS | E5 许可 |
| Exabeam | 基于时间线的行为分析引擎、用户画像 | 独立部署或 SIEM 集成 | 私有化或云 | 按用户数 |
| Securonix | 大数据架构、内置威胁内容库 | 多数据源（100+ 种） | 私有化或云 | 按 EPS/用户数 |
| 奇安信 NGSOC UEBA | 国产化、AI 辅助研判、等保适配 | 需配套 NGSOC 平台 | 私有化 | 项目制 |
