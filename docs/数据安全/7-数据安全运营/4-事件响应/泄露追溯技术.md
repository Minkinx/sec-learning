# 泄露追溯技术

## 概述

数据泄露追溯是在事件发生后，通过技术手段定位数据泄露的源头、路径和责任人。追溯能力直接影响事件定责、合规报告和法律诉讼的证据效力。追溯技术分为主动防御型（水印、蜜标）和被动分析型（日志链重建、行为分析），两者结合形成完整的追溯能力体系。

## 数字水印技术

水印技术通过在数据中嵌入不可见标识，便于泄露后追溯到源头。

### 三类水印对比

| 水印类型 | 嵌入方式 | 鲁棒性 | 容量 | 适用数据 | 典型抗攻击 |
|----------|----------|--------|------|----------|-----------|
| 数字水印（Digital Watermarking） | 在文档/图片/视频中嵌入不可见数字标识 | 高 | 低-中 | 文档、图片、音视频 | 截图、重编码、压缩 |
| 结构水印（Structural Watermarking） | 数据库字段中加微量噪声或伪记录作为标识 | 中 | 低 | 数据库表（数值型字段） | 字段筛选、排序变化 |
| 行为水印（Behavioral Watermarking） | 生成/分发过程中植入访问时间偏移、请求排顺序 | 低 | 高 | API 数据分发 | 数据重排、定时传输 |

### 数字水印技术要求

鲁棒数字水印需要抵抗以下常见变换操作：

| 变换操作 | 影响 | 对抗策略 |
|----------|------|----------|
| JPEG 压缩/重编码 | 水印信号衰减 | 使用基于频域的水印嵌入（DWT/DCT） |
| 截图/裁剪 | 水印区域丢失 | 冗余水印嵌入，多区域分布 |
| 格式转换（PDF→图片→PDF） | 像素级变换 | 校准算法 + 同步头设计 |
| 缩放/旋转 | 几何变换 | 使用几何不变域水印 |
| 颜色空间变换 | 色值偏移 | 亮度通道嵌入（对色值变换鲁棒） |

### 结构水印实现示例

在数据库客户表中插入伪记录：
```sql
-- 原始记录：10001, '张三', '13800138000', '北京市'
-- 蜜标记录：99999, '李蜜', '13999999999', '上海市'
-- 泄露后通过该伪记录的出现位置识别泄露数据来源
```
每条伪记录具有唯一的哈希特征码，记录分发版本号，阅后即能从数据片段中识别来源。

## 蜜标技术

蜜标（Honeytoken）和蜜罐（Honeypot）是主动防御与追溯结合的技术手段：

| 类型 | 部署方式 | 检测机制 | 定责方式 |
|------|----------|----------|----------|
| 数据库蜜标 | 在数据库表中插入虚假客户记录，内容含唯一特征码 | 蜜标出现在非授权渠道时触发告警 | 蜜标特征码关联到分发记录 |
| 文档蜜标 | 在文档中嵌入独特短语/段落，不同版本微差异 | 文档被公开或扫描时匹配蜜标文本 | 文本差异追溯文档版本 |
| 文件蜜标（Canary Tokens） | 在文件/URL/目录中嵌入可追溯的 Token | Token 被访问（Web 请求/DNS 查询）时触发 | Token 唯一 ID 关联到分发对象 |
| API 蜜标 | 在 API 返回数据中包含可标识的伪数据点 | 外部出现匹配的伪数据 | 伪数据关联 API 密钥和调用者 |

### 蜜标部署策略

1. **嵌入真实数据**：蜜标数据在格式和统计特性上与真实数据不可区分
2. **唯一标识**：每份分发的数据中含独特的蜜标组合，形成唯一指纹
3. **触发告警**：蜜标数据被访问或出现在未授权渠道时，触发高优先告警
4. **关联定责**：蜜标 ID 关联到分发记录（谁+何时+通过什么渠道获得）

## 日志链重建

### 时间线重建

将分散在多个系统的日志按时间对齐，重建事件全貌：

| 数据源 | 日志类型 | 关键字段 | 对齐方式 |
|--------|----------|----------|----------|
| AD 认证日志 | Event ID 4624/4625 | LogonID、Account Name、Source IP | 时间戳 + 用户 SID |
| 数据库审计日志 | DML/DDL 操作 | User、SQL Text、Rows Affected | 时间戳 + 数据库 Session ID |
| VPN 日志 | 连接/断开 | User、Public IP、Connection Time | 时间戳 + 用户 |
| DLP 事件 | 策略命中 | User、Action、File、Destination | 时间戳 + 用户 + 事件 ID |
| 文件服务器审计 | 4663/5145 | User、File Path、Access Mask | 时间戳 + 用户 + 文件 |
| 邮件日志 | 发送/接收 | Sender、Recipient、Subject、Attachment | 时间戳 + 发件人 |

### 访问路径追溯

沿着用户访问链追溯：谁 -> 通过什么系统 -> 访问了什么数据 -> 做了什么操作 -> 数据流向哪里。

```
用户 A (10.0.1.100)
  │
  ├─→ [09:00] VPN 登录 (VPN 日志)
  │
  ├─→ [09:05] RDP 连接到 DB 跳板机 (Windows Event 4624)
  │
  ├─→ [09:10] SQL 查询: SELECT * FROM customers (数据库审计)
  │        返回 50,000 行 (超正常量 5 倍)
  │
  ├─→ [09:15] 导出到 CSV 文件 (应用日志)
  │
  ├─→ [09:20] 压缩加密文件 (终端进程日志)
  │
  └─→ [09:25] 通过 HTTPS POST 将文件上传到个人云盘 (DLP 事件 -> 网络日志)
```

### 关联分析

将账号活动、网络流、文件操作多源关联，补全碎片化的证据链。常用关联引擎包括：

- **Splunk ES**：基于 CIM 的数据模型关联，支持事件关联搜索
- **Elastic Security**：基于 ECS 的数据关联，使用 KQL 查询语言
- **自定义关联规则**：SQL/Spark 作业处理海量日志的时间窗口关联

## 溯源定责

溯源的最终目标是归因（Attribution），需要区分：

| 责任人类型 | 定义 | 证据要求 | 处置 |
|-----------|------|----------|------|
| 直接责任人 | 执行泄露操作的用户 | 账号 + 操作日志 + 时间戳匹配 | 纪律处分/法律追责 |
| 间接责任人 | 权限授予者（审批了不应授予的权限） | 权限变更记录 + 合规要求 | 流程改进 |
| 间接责任人 | 系统管理员（未及时修复已知漏洞导致入侵） | 漏洞扫描报告 + 修复记录 | 流程改进/绩效考核 |
| 辅助因素 | 缺失的安全控制（未部署 DLP/审计） | 控制评估报告 | 安全规划投入 |

归因结果需要形成法律可采信的证据链，满足法庭证据的三性要求：
- **真实性**：证据未被篡改（哈希校验 + 数字签名）
- **合法性**：证据采集符合法律程序（符合电子数据取证规定）
- **关联性**：证据与待证事实之间存在逻辑联系

建议在准备阶段就建立与法务部门的协作机制，确保证据采集流程符合司法要求。
