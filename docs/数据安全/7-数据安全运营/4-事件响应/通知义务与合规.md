# 通知义务与合规

## 法律法规通知时限对比

| 法规 | 通知时限 | 通知对象 | 触发条件 | 主要要求 |
|------|----------|----------|----------|----------|
| 中国《个人信息保护法》第57条 | **立即** 采取补救措施 + **72小时内**通知 | 监管机构（网信办/行业主管） | 个人信息泄露、篡改、丢失 | 立即补救 + 72h 通知 + 记录全过程 |
| 欧盟 GDPR 第33/34条 | **72小时内** | 监管机构（DPA）+ 数据主体 | 个人数据泄露可能导致风险 | 超时需提供延迟理由；高风险须通知个人 |
| 中国《数据安全法》 | 及时 | 行业主管/网信部门 | 重要数据和核心数据泄露 | 按行业具体要求执行 |
| 美国 CCPA/CPRA | **30天** 整改期 | 用户 | 未加密/未脱敏的个人信息泄露 | 30天内可补救则避免处罚 |
| 美国 SEC 规则（上市公司） | **4个工作日** | SEC | 重大网络安全事件认定 | 2023年12月生效；是否重大由公司判断 |
| 美国 HIPAA | **60天**（超500条记录需**立即**通知） | HHS + 受影响个人 | 受保护健康信息（PHI）泄露 | 超500条须立即联系媒体公告 |
| 韩国 PIPA | **立即**（知悉后） | 韩国个人信息保护委员会 | 超过1万名用户数据泄露 | 超过1万人须立即通知 |
| 新加坡 PDPA | **尽快** | 个人数据保护委员会 | 造成重大损失或大量数据泄露 | 评估后可延迟通知 |
| 巴西 LGPD | **合理期限** | ANPD + 数据主体 | 可能对主体造成风险或损害 | 描述事件性质、影响和处置措施 |

## 通知内容标准

根据 PIPL 和 GDPR 的通行要求，数据泄露通知应包含以下要素：

1. **事件描述**：泄露发生时间和范围、发现时间、事件性质、泄露原因（已知时）
2. **数据类型**：涉及的个人信息类别（身份信息/联系方式/财务信息/健康信息等）、数据字段名称、受影响记录数量
3. **影响评估**：已造成或可能造成的危害说明、风险评估结论（对数据主体的具体风险）
4. **处置措施**：已采取的应急措施（阻断/隔离/补丁）、后续响应计划、已通知哪些机构
5. **联系人信息**：数据保护官（DPO）或安全负责人的联系方式（电话/邮箱/办公地址）
6. **建议措施**：对受影响个人的建议（修改密码、监控账户、挂失证件等）

## 通知对象分类

| 通知对象 | 通知方式 | 时间要求 | 内容要求 | 审批流程 |
|----------|----------|----------|----------|----------|
| 监管机构 | 正式书面报告（公文格式） | 按法规时限（通常 72h） | 法定内容 + 格式要求 | 法务审核 + CISO 签发 |
| 受影响个人 | 邮件/短信/电话/应用内通知 | 确认影响后尽快 | 简明语言 + 防护建议 | 法务审核 + 公关审核 |
| 合作伙伴 | 加密邮件通知 | 确认涉及后 24h 内 | 仅涉及合作方的部分 | 业务负责人 + 法务 |
| 媒体/公众 | 新闻稿/官方声明 | 大规模泄露事件 | 经公关团队审定版本 | CEO + 法务 + 公关 |
| 员工（内部） | 内部公告邮件 | 确认后 24h 内 | 适当的透明度 | HR + 法务 |

### 监管机构通知模板（参考）

```
致：[监管机构名称]

根据 [适用法律名称] 第 X 条，现就数据安全事件报告如下：

1. 报告方信息
   - 名称：[企业全称]
   - 统一社会信用代码：[代码]
   - DPO 联系人：[姓名] [电话] [邮箱]

2. 事件概述
   - 发现时间：[YYYY-MM-DD HH:MM]
   - 事件类型：[数据泄露/数据丢失/数据篡改]
   - 事件描述：[简述经过]

3. 涉及数据
   - 数据类型：[PII/财务/健康/其他]
   - 受影响记录数：[约 X 条]
   - 受影响个人数：[约 X 人]

4. 已采取措施
   - [已阻断/隔离/修复措施]
   - [已通知的相关方]

5. 影响评估
   - [对数据主体的潜在影响]
   - [风险评估结论]

6. 下一步计划
   - [后续处置措施和时间表]

报告人：[姓名]
职务：[职务]
日期：[YYYY-MM-DD]
```

## 跨境通知

当数据泄露涉及多国/多司法管辖区时，需要协调不同法规的通知义务：

| 挑战 | 建议做法 | 注意事项 |
|------|----------|----------|
| 多法规时限冲突 | 以最早的通告时限优先（72 小时内） | 确认所有管辖区的通知期限 |
| 多语言通知 | 统一通知内容模板，按各辖区要求微调语言 | 翻译须经法律确认，避免歧义 |
| 法律特权保护 | 指定跨境协调律师（满足律师-客户特权保护） | 聘请当地法律顾问 |
| 监管沟通 | 与各监管机构保持透明沟通 | 主动沟通优于被动响应 |
| 数据主体差异 | 不同地区对通知内容的要求不同 | 按当地法律调整通知要点 |

### 跨境泄露响应流程

1. **确定涉及司法管辖区**：根据受影响用户的位置、数据存储位置、服务提供位置判定
2. **建立多国协调团队**：包含各国法律顾问
3. **确定最早通知时限**：以最严格的法律要求为基准
4. **准备多语言通知模板**：经各辖区法务审查
5. **分批次发出通知**：先监管机构，再个人
6. **跟踪通知状态**：确认各辖区通知义务已履行

## 通知触发决策树

```
数据安全事件发生
    │
    ├─ 涉及个人信息/PII？
    │   ├─ 是 → 是否触发监管通知义务？
    │   │       ├─ 是 → 按对应法规时限通知
    │   │       └─ 否 → 内部记录，无需通知
    │   └─ 否 → 仅涉及商业秘密/企业数据？
    │           ├─ 是 → 内部处置 + 业务通知
    │           └─ 否 → 常规事件处置
    │
    ├─ 涉及重要数据/核心数据？
    │   └─ 是 → 按《数据安全法》向行业主管报告
    │
    ├─ 涉及跨境数据？
    │   └─ 是 → 按跨境数据传输合规要求处理
    │
    └─ 涉及多司法管辖区？
        └─ 是 → 启动跨境通知流程
```

企业应预先准备好通知模板和审批流程，经过法务审核，按场景（监管机构/个人/媒体）分版本预制备用。通知发出前须经事件响应指挥组（含法务和公关）审批。
