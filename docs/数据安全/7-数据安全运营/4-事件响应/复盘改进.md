# 复盘改进

## 事后复盘流程

事件处置结束后，应召开正式复盘会议（Post-Incident Review, PIR），覆盖以下六个维度：

1. **时间线审查**：回顾从事件发生到发现、响应、遏制的完整时间线，识别各环节的耗时瓶颈。重点检查 MTTD（平均检测时间）和 MTTR（平均响应时间）是否达到预期 SLA。
2. **事实确认**：基于日志和取证结果确认事件的事实链，避免依赖个人回忆或主观判断。所有结论应有证据支撑。
3. **根因分析**：识别导致事件的直接原因和根本原因，区分技术缺陷、流程漏洞和人为因素。
4. **纠正措施**：针对根因制定立即修复措施（如补丁更新、配置修正）。
5. **预防措施**：建立系统性改进方案，防止同类事件再次发生（如增加监控规则、改进权限模型）。
6. **经验教训**：总结事件的正面和负面经验，纳入知识库和改进计划。

## 改进措施跟踪

改进措施的管理应遵循 SMART 原则：

- **指定责任方**：每项措施明确到人（RACI 矩阵）
- **设定截止日期**：根据措施优先级设定合理完成期限（紧急措施 7 天内、系统性改进 30 天内、长期项目按季度追踪）
- **量化效果指标**：修复完成、规则覆盖提升、告警减少百分比等
- **定期复查**：每两周复查进展，到期未完成的措施自动上报

## 改进指标

衡量复盘改进效果的关键指标：

| 指标 | 定义 | 目标值 |
|------|------|--------|
| MTTD 提升 | 平均检测时间环比改善 | 季度提升 20% 以上 |
| MTTR 提升 | 平均响应时间环比改善 | 季度提升 15% 以上 |
| 重复事件率 | 同类事件再次发生占比 | 低于 10% |
| 措施完成率 | 按时完成改进措施比例 | 高于 90% |
| 检测覆盖率 | 具备检测能力的事件类型占比 | 持续提升 |

## 根因分析方法

- **5 Whys 分析法**：连续追问"为什么"直到找到根本原因，适合简单到中等复杂的事件。
- **鱼骨图（因果图）**：将问题分类（人/流程/技术/外部），从多个维度系统性分析根因，适合复杂事件。
- **故障树分析**：自上而下的演绎分析，适合对关键系统失效进行深度剖析。

## 无指责文化

有效的复盘需要建立在无指责文化（Blameless Culture）基础上。核心理念：流程和系统的缺陷而非个人的疏忽是大多数事件的根源。惩罚个人只会导致问题被掩盖，鼓励"做对的事"而非"规避责任"的行为。组织应将复盘视为改进机会而非追责程序。
