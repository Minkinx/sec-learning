# 网络 DLP

## 概述

网络数据防泄露（Network DLP）在数据流经网络时进行检测与阻断，监控邮件、Web 流量、即时通讯、文件传输等通道。作为企业 DLP 架构的流量层防线，网络 DLP 覆盖所有经过网络边界的数据流。其核心价值在于集中管控：在一个控制台上定义策略，在网络出口统一执行，无需在每个终端或应用上单独配置。

## 架构设计

网络 DLP 在企业网络架构中位于核心交换层或出口边界，通常串联在以下位置：

```
┌─────────┐     ┌──────────────┐     ┌──────────────┐     ┌─────────┐
│ 内网用户  │────▶│   DLP 检测引擎  │────▶│ 策略执行点     │────▶│ 互联网   │
│          │     │ (内容分析/指纹) │     │ (阻断/告警/限速)│     │         │
└─────────┘     └──────────────┘     └──────────────┘     └─────────┘
```

## 部署模式

| 模式 | 部署方式 | 阻断能力 | 延迟影响 | HA 要求 |
|------|----------|----------|----------|---------|
| Inline（在线模式） | 串联部署在网络路径上 | 实时阻断 | <1ms 延迟，需 bypass 机制 | Active-Standby 双机 |
| Out-of-Band（旁路模式） | TAP/SPAN 端口镜像流量 | 无法实时阻断 | 零延迟影响 | 无（纯监控） |
| ICAP 模式 | 通过 ICAP 协议调用 DLP 引擎 | 近实时阻断 | 受 ICAP 服务器处理能力影响 | 负载均衡集群 |
| Proxy 模式 | DLP 作为正向代理 | 实时阻断 | 新增代理跳数 | Proxy 集群 |

Inline 模式下，需要重点设计高可用性：
- **Bypass 机制**：硬件 Bypass（DLP 设备故障时自动短接）和软件 Bypass（超出处理能力时直通）
- **故障切换**：Active-Standby 双机热备，切换时间 <1 秒
- **性能容量**：通常按峰值带宽的 1.5-2 倍规划，预留扩容空间

## 协议覆盖

| 协议/通道 | 监控内容 | 检测能力 | 典型配置 |
|-----------|----------|----------|----------|
| SMTP/SMTPS | 邮件正文和附件（含压缩包） | 文本/附件/元数据/嵌套压缩 | 集成邮件安全网关，TLS 解密需 MTA 级配置 |
| HTTP/HTTPS | Web 上传、网页邮件（Webmail） | URL/表单上传/文件下载 | 与 SWG/代理集成，透明代理 SSL 解密 |
| SMB/CIFS | 文件共享协议 | 文件复制到非授权共享、批量迁移 | 需部署在文件服务器出口 |
| FTP/SFTP/FTPS | 文件传输协议 | 深度内容检测 + 协议合规检查 | 支持被动/主动模式，需深度包检测 |
| IM（Teams/Slack/微信） | 即时通讯文件传输和消息 | 文本消息/文件/链接 | API 集成或 SSL 解密后检测 |
| DNS | DNS 查询 | 检测 DNS 隧道（数据外传隐蔽通道） | 结合 NDR 分析异常 DNS 请求 |
| API/REST | API 调用的数据返回 | JSON/XML 内容中的敏感数据 | 与 API 网关集成 |

## 内容检测技术

网络 DLP 的内容检测引擎包括：

| 检测技术 | 原理 | 适用内容 | 性能开销 |
|----------|------|----------|----------|
| 正则表达式 | 模式匹配（卡号/身份证号等） | 结构化敏感数据 | 低 |
| 关键字/字典 | 关键词列表匹配 | 特定术语/项目代码 | 低-中 |
| 精确哈希指纹 | SHA256 文档哈希比对 | 受控文档精确识别 | 低 |
| 模糊指纹 | 基于向量化的相似度匹配（编辑距离/SimHash） | 文档部分内容匹配 | 中-高 |
| ML 分类 | 训练内容分类模型判断文档类型 | 非结构文档识别（合同/报告/源码） | 高 |
| OCR 识别 | 图片文字识别 | 截图/扫描件中的文字 | 高（需 GPU 加速） |

## TLS 解密挑战

随着 HTTPS 成为默认协议，网络 DLP 面临 TLS 解密的两难：

**技术方案对比：**

| 方案 | 原理 | 优点 | 缺点 | 合规风险 |
|------|------|------|------|----------|
| 透明代理解密 | 部署堡垒 CA 证书，中间人方式 | 全覆盖，检测彻底 | 需管理根证书，增加 SSL 握手延迟 | GDPR 员工隐私风险 |
| 选择性解密 | 仅解密已知 SaaS 和企业应用的流量 | 降低隐私风险 | 不能覆盖未知应用 | 需用户知情同意 |
| API 集成 | 通过云服务商 API 直接检查内容 | 无需解密，无延迟 | 仅覆盖支持 API 的应用 | 低 |
| 客户端代理 | 终端 Agent 解密后检测 | 无网络改造 | 需安装 Agent | 低 |

理想的方案是选择性解密 + API 集成组合，在覆盖率和隐私合规之间取得平衡。同时应在员工手册和公司隐私政策中明确告知网络监控范围。

## 集成架构

网络 DLP 需与企业网络基础设施深度集成：

| 集成组件 | 集成方式 | 提供能力 |
|----------|----------|----------|
| Secure Web Gateway (SWG) | ICAP/WCCP 协议 | HTTP/HTTPS 流量通道 + URL 过滤 |
| Email Security Gateway | SMTP 转发/API | 邮件流量通道 + 防钓鱼检测 |
| CASB | REST API/反向代理 | SaaS 流量通道 + 影子 IT 发现 |
| NDR | NetFlow/PCAP 集成 | 异常流量行为关联 + C2 检测 |
| SIEM/SOAR | Syslog/REST API | 事件归一化 + 自动编排响应 |

## 性能规划

网络 DLP 的性能瓶颈通常在于深度内容检测，性能规划参考值：

| 企业规模 | 带宽需求 | 实时并发连接 | 设备推荐 | 延时预算 |
|----------|----------|-------------|----------|----------|
| 小型（<500 人） | <1Gbps | 5,000 | 软件部署（x86 服务器） | <5ms |
| 中型（500-2000 人） | 1-10Gbps | 20,000-50,000 | 专用硬件 DLP 设备 | <2ms |
| 大型（>2000 人） | 10-40Gbps | 100,000+ | 集群部署 + 流量负载均衡 | <1ms |
