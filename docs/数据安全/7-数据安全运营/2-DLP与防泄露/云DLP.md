# 云 DLP

## 概述

随着企业业务上云，数据分布在 SaaS、PaaS、IaaS 等多种云环境中，传统网络 DLP 无法覆盖云端数据流动。云 DLP 通过 API 集成和代理部署，弥合云环境与数据安全之间的鸿沟。

## 部署模式

- **API 模式**：通过云服务商 API 进行扫描和检测，无需修改网络拓扑，延迟较低，但无法实现实时阻断（依赖于云服务商的事件驱动触发）。
- **Inline Proxy 模式**：通过正向代理或反向代理串联数据流，实现实时检测与阻断，但引入额外的网络跳数和数据流路径。
- **Agent 模式**：在云工作负载上部署代理，适用于 IaaS 场景，可覆盖云主机本地存储和网络流量。

## CASB 集成

云 DLP 通常与云访问安全代理（CASB）深度集成，形成完整的云数据保护方案：

| 集成能力 | 说明 |
|----------|------|
| 影子 IT 发现 | 通过 CASB 发现未授权的云应用使用 |
| 策略同步 | DLP 策略通过 CASB 统一推送到各云应用 |
| 事件关联 | 云 DLP 事件与 CASB 日志关联分析 |
| 用户上下文 | 从 CASB 获取用户身份和设备信息 |

## 多云策略一致性

在多云环境下保持 DLP 策略一致性是核心挑战。策略定义应遵循"一次定义、多端执行"的原则，使用统一的策略管理平面（如 CSPM/CNAPP 平台）将策略转换为各云服务商的原生格式后分发。

## 数据驻留控制

云 DLP 需要结合数据驻留要求实施区域级管控。通过云服务商的策略引擎（如 AWS SCP、Azure Policy）限制数据跨区域存储和传输。检测到数据被移动到不允许的区域时触发告警或阻断。

## 云原生 DLP 服务对比

| 服务 | 核心能力 | 适用场景 |
|------|----------|----------|
| AWS Macie | S3 桶自动发现、ML 敏感数据分类 | AWS 环境敏感数据发现 |
| Microsoft Purview | 统一数据治理与 DLP、Teams/Exchange/SharePoint 集成 | Microsoft 365 生态 |
| Google DLP | 内容检测 API、模板化的分类方法、去标识化 | GCP 和异构数据源 |
