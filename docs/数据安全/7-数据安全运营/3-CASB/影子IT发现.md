# 影子 IT 发现

## 概述

影子 IT（Shadow IT）指未经企业 IT 部门批准而使用的技术资源、设备、软件或服务。在云服务场景下，员工使用未经审核的 SaaS 应用存储和处理企业数据，是数据泄露的主要风险来源之一。

## 发现方法

| 方法 | 原理 | 覆盖范围 | 优缺点 |
|------|------|----------|--------|
| 网络流量分析 | 分析防火墙/代理日志中的目标域名和应用特征 | 公司网络内所有流量 | 覆盖广；无法检测加密流量以外的应用 |
| API 集成 | 通过 OAuth/API 与云应用目录集成获取应用列表 | 已授权应用 | 精度高；仅覆盖授权应用 |
| 终端代理 | 终端 Agent 采集 DNS/进程信息 | 受管端点 | 覆盖端侧全流量；部署成本高 |
| 用户调查 | 匿名问卷调查员工使用的应用 | 主观意见 | 只能作为补充手段 |

## 风险评估

发现影子 IT 后需要进行风险评估，核心评估维度：

- **数据敏感性**：员工放入影子 IT 的数据是否包含个人身份信息（PII）、财务数据、商业秘密等
- **用户规模**：使用该应用的员工数量，影响风险暴露面
- **访问方式**：使用企业 SSO 登录 vs 个人账号注册，是否支持多因子认证
- **合规认证**：应用是否通过 SOC 2/ISO 27001/HIPAA 等认证
- **数据驻留**：数据中心是否位于允许的司法管辖区

## 风险评分模型

综合上述维度建立风险评分模型，示例权重分配：应用安全功能（30%）、数据保护能力（25%）、合规状态（20%）、用户使用规模（15%）、历史安全事件（10%）。得分范围 0-100，按阈值分为低风险（0-30）、中风险（31-60）、高风险（61-100）。

## 处置工作流

针对影子 IT 的发现结果，建立标准处置流程：

1. **评估**：安全团队评估应用风险评分
2. **决策**：判定应用的处理方式
3. **处置**：批准（纳入 IT 管理）-> 整改（要求增加安全控制）-> 阻断（在网关层阻止访问）-> 迁移（将数据迁移至企业认可的应用）
4. **验证**：定期复查确保阻断生效或整改已完成
