# 部署模式

## 三种核心模式

CASB 的部署模式决定了其能力边界和性能特征，主要分为三种：

### 正向代理（Forward Proxy）

终端用户通过配置代理访问云服务，数据流经 CASB 平台进行检测。

- **优点**：对所有云服务流量可见，可实施精细的访问控制和 DLP 策略
- **缺点**：需要终端配置代理（PAC 文件或浏览器配置），部分非浏览器流量难以捕获
- **典型场景**：企业办公环境下对员工访问 SaaS 进行全面管控

### 反向代理（Reverse Proxy）

用户访问云应用时，DNS 解析指向 CASB 平台，CASB 作为中间人代理用户请求。

- **优点**：不需要终端配置，对用户透明；可实时检测阻断
- **缺点**：仅覆盖配置了反向代理的特定应用，存在 SSL 证书冲突问题
- **典型场景**：访问受控的 SaaS 应用（如 Salesforce/Servicenow）

### API 模式

通过云服务商的 API 直接集成，获取应用配置和数据共享状态。

- **优点**：不限用户网络位置或设备类型，可覆盖已授权的云服务配置审计
- **缺点**：无法实时检测或阻断，受限于云服务商 API 的能力和速率限制
- **典型场景**：云服务安全配置检查和数据发现

## 模式对比

| 维度 | 正向代理 | 反向代理 | API 模式 |
|------|----------|----------|----------|
| 延迟影响 | 中（增加一跳） | 低（优化路由） | 无（异步调用） |
| 可视性深度 | 高 | 中 | 中 |
| 实时阻断 | 支持 | 支持 | 不支持 |
| 用户体验 | 依赖代理配置 | 透明 | 无影响 |
| 应用兼容性 | 广（所有 HTTP 流量） | 窄（特定应用） | 看 API 支持范围 |

## 混合架构

业界最佳实践是采用混合部署：API 模式提供持续合规监控和审计（覆盖所有联网的设备），代理模式提供实时访问控制和 DLP（覆盖企业网络内的流量）。两种模式数据统一汇入 CASB 管理平台进行关联分析。

## PoC 方法论

实施 CASB 的 PoC 流程：1）确定 PoC 范围（选择 2-3 个关键云应用）；2）部署 API 集成（快速获取配置状态和影子 IT 数据）；3）选择试点用户组部署代理；4）验证策略执行效果和用户体验影响；5）输出评估报告（包含可视性覆盖率、误报率、性能影响）。
