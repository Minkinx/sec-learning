# 一般数据

> 除核心数据和重要数据之外的数据，是组织数据资产的主体部分。

## 法律定义

《数据安全法》采用"等外"模式定义一般数据：即不构成核心数据和重要数据的数据。一般数据虽保护等级相对较低，但仍需满足基本安全要求，不能放任不管。

## 一般数据的细分

### 内部数据
- **定义**：仅限组织内部使用，不对外公开的数据
- **示例**：内部邮件、会议纪要、组织架构、内部制度文件、员工通讯录、项目过程文档
- **保护要求**：身份认证后访问，操作日志记录，离职权限回收

### 公开数据
- **定义**：经评估可以对外公开或已对外公开的数据
- **示例**：产品手册、对外公告、招聘信息、开源代码、已披露的财务报告
- **管理要求**：发布前须经审核，防止敏感信息误嵌入公开数据

## 最小保护基线

即使是一般数据，也应满足以下最低安全基线：

| 控制领域 | 基线要求 |
|---------|---------|
| 访问控制 | 基于身份的访问控制（IBAC），离职即回收 |
| 身份认证 | 用户名+密码认证，有条件的启用 MFA |
| 传输安全 | HTTPS/TLS 加密传输 |
| 操作审计 | 关键操作日志（增删改），保留不少于6个月 |
| 备份恢复 | 定期备份，可容忍恢复时间不超过72小时 |
| 数据销毁 | 数据生命周期结束后安全删除，介质报废 |
| 安全培训 | 员工年度数据安全意识培训 |

## 处理工作流

1. **数据采集**：明确采集目的和最小必要范围，告知用户
2. **数据使用**：在授权范围内使用，超出目的须重新授权
3. **数据共享**：内部共享需审批，外部共享需签订保密协议
4. **数据归档**：按保留期限归档，归档数据访问须审批
5. **数据销毁**：到期后物理销毁或密码学安全擦除

## 与重要数据的边界

实践中容易出现"一般数据误标为重要数据"或"重要数据漏标为一般数据"的问题。建议：
- 建立"边界案例"评判机制，由数据安全委员会裁决
- 有疑问时偏向高级别（安全保守原则）
- 定期审查边界案例，积累判定经验

## 合规要求

一般数据虽不触发核心数据和重要数据的严格监管要求，但仍需满足：
- 不得非法收集、使用、加工、传输他人数据
- 数据处理活动须符合数据安全管理制度
- 发生数据安全事件须及时报告并采取补救措施
