# 动态升降级机制

> 数据的分级并非一成不变，需建立动态调整机制以适应业务变化和数据生命周期。

## 为什么需要动态升级机制

数据的分级会随时间、场景和处理方式的变化而改变：
- 原始数据脱敏后级别降低
- 多源数据汇聚后产生新的敏感信息
- 业务背景变化导致数据重要性提升
- 法律法规更新要求提高保护等级

## 触发升降级的场景

### 升级触发条件
| 场景 | 示例 | 响应要求 |
|------|------|---------|
| 数据汇聚 | 多个一般数据集合后形成重要数据集 | 检测到汇聚后30天内完成重新定级 |
| 时间推移 | 项目资料达到保密期后解密 | 按预设定时器自动调整 |
| 法规变更 | 新法规将某类数据列入重要数据范畴 | 法规生效前完成升级 |
| 安全事件 | 数据泄露后需提高保护等级 | 事件处置完成后7天内评估 |
| 业务变更 | 收购新业务后数据资产纳入 | 整合完成后30天内 |

### 降级触发条件
| 场景 | 示例 | 前置条件 |
|------|------|---------|
| 数据脱敏/匿名化 | 原始数据经脱敏处理后不再包含个人标识 | 验证匿名化不可逆 |
| 保密期满 | 保密协议到期后的商务资料 | 法律审查确认可解密 |
| 时效过期 | 超过保留期限的日志数据 | 确认无法律/合规保留义务 |
| 业务下线 | 已停用的遗留系统数据 | 数据迁移或销毁方案已确认 |

## 基于时间的自动衰减

建议建立"数据时效性"辅助维度：
- **活跃期**：数据产生后1-6个月，保护等级最高
- **半活跃期**：6个月-2年，可适当降低访问门槛
- **归档期**：2年以上，按归档策略管理，访问需审批
- **销毁评估期**：超过保留期限，评估销毁或降级

## 聚合效应管理

数据聚合是升降级管理的难点。n条一般数据汇聚后可能形成重要数据：
- **地址聚合风险**：单条地址为一般数据，批量地址+时间戳可推断用户行为模式
- **交易聚合风险**：单笔交易信息一般，全量交易数据为重要数据（影响经济运行分析）
- **代码聚合风险**：单个模块代码一般，完整系统源代码为核心资产

建议措施：
- 在数据仓库、数据湖环境部署聚合风险检测规则
- 对跨数据源的 JOIN 查询进行敏感度评估
- 数据中台层建立"聚合敏感度评分"模型

## 升降级审批流程

1. **发起**：数据所有者或数据安全团队提出升降级申请
2. **评估**：安全团队进行安全影响评估（SIA）
3. **复核**：所在部门负责人及合规部门复核
4. **审批**：升级由数据安全委员会审批，降级由数据安全负责人审批
5. **执行**：在数据资产目录中更新分级标签，联动调整访问控制策略
6. **通知**：通知数据使用方分级变更

## 自动化支持

- **定时任务**：基于预设规则的定时扫描更新（如到期降级）
- **事件驱动**：监测到数据汇聚、脱敏操作时自动触发重新定级评估
- **人工确认**：自动化初判后，关键数据域的升降级须人工确认
- 推荐将动态升降级逻辑集成至数据资产管理平台，与 DLP、CASB 策略联动
