# 数据资产地图

> 建立统一的数据资产视图，实现全环境覆盖，为数据安全运营提供"一张图"。

## 构建统一数据资产视图

数据资产地图的目标是整合多来源的资产信息，形成统一的数据资产目录。核心能力包括：

1. **资产统一标识**：为每一份数据资产分配全局唯一的资产ID
2. **属性统一建模**：定义标准资产属性模型（位置、类型、所有者、分类、分级、生命周期状态）
3. **关联关系表达**：资产与业务系统、组织架构、物理存储的映射关系
4. **搜索发现**：支持按数据内容、字段名、标签等多维度搜索数据资产

## 与 CMDB 集成

数据资产地图不应独立建设，而应与现有 CMDB 体系集成：
- CMDB 管理 IT 基础设施（服务器、数据库实例、网络设备）
- 数据资产地图管理数据层面资产（数据库表、文件、数据流）
- 二者通过"数据存储在哪个 IT 资产上"的关系关联

集成模式建议：
- 数据资产地图从 CMDB 获取基础设施清单，作为数据发现的输入
- CMDB 从数据资产地图获取数据分级信息，关联到 CI（配置项）
- 变更管理：当 IT 资产变更时，联动评估数据安全影响

## 多环境映射

### 开发环境
- 数据通常为脱敏或不完整数据，安全分级可低于生产环境
- 但需注意：开发人员可能拥有高权限访问开发库，存在泄露风险
- 建议：开发环境使用生产数据的脱敏副本

### 测试环境
- 性能测试可能需要与生产环境等量级的数据
- 功能测试应使用构造数据而非真实数据
- 所有测试环境数据应在测试结束后及时清理

### 生产环境
- 核心保护对象，数据资产地图的基准环境
- 重点维护字段级分类分级信息
- 访问权限记录须精确到表/字段级别

### 云环境
- 多云环境需统一纳管（AWS + Azure + 阿里云 + 私有云）
- 利用各云平台的原生数据安全能力（Macie、SDDP 等）
- 重点关注跨云数据传输和影子数据资产

## 治理工作流

数据资产地图的持续治理流程：

```
数据发现 -> 资产登记 -> 分类分级 -> 安全评估 -> 标签打标 -> 持续监控 -> 定期复审
```

其中关键环节：
- **资产登记**：新数据资产48小时内完成登记
- **分类分级**：登记后7天内完成初分类
- **安全评估**：对重要级别以上资产30天内完成安全评估
- **持续监控**：通过自动化工具监控资产变化（新库、新表、新字段）
- **定期复审**：每季度对全量资产进行一次分类分级复审

## 组织保障

- 设立数据资产管理员角色，负责资产地图的日常维护
- 各业务部门指定数据资产联络人，配合资产登记
- 数据治理委员会定期（季度）审核资产地图的完整性和准确性
