# 数据资产盘点

> 全面掌握数据资产分布是分类分级工作的前提。没有完整的资产清单，分类分级就是无源之水。

## 发现方法论

数据资产盘点的核心方法论为"三阶段递进"：

1. **摸底阶段**：通过网络扫描、CMDB 导出、云平台 API 查询等方式，获取全量 IT 资产清单
2. **识别阶段**：对每项资产进行数据指纹分析，识别其中包含的数据类型和数据量级
3. **合规阶段**：对识别的数据资产进行合规标签打标（PII/重要数据/核心数据等）

## 范围定义

盘点的边界需要明确界定：
- **组织边界**：是否包含子公司、合资公司、海外分支机构？
- **环境边界**：生产、测试、开发、灾备环境是否全覆盖？
- **存储边界**：数据库、文件服务器、对象存储、SaaS平台、终端设备、移动存储？
- **生命周期边界**：在线数据、近线归档、离线备份是否都在盘点范围内？

建议"先宽后窄"：首次盘点范围尽量宽，后续通过优先级管理聚焦高风险资产。

## 工具类别

| 工具类别 | 代表产品 | 适用场景 | 优势 | 局限 |
|---------|---------|---------|------|------|
| 数据库扫描器 | Imperata、Guardium | 结构化数据发现 | 深度字段级识别 | 仅覆盖数据库 |
| 文件爬虫 | Varonis、Digital Guardian | 文件服务器、NAS | 权限分析 + 数据发现 | 扫描耗时长 |
| 云API | AWS Macie、阿里云SDDP | 云上对象存储、RDS | 原生集成，自动发现 | 多厂商难统一 |
| 终端扫描 | Spirion、Endpoint Protector | 终端/笔记本 | 覆盖影子数据 | Agent部署运维成本 |
| 数据目录平台 | BigID、Collibra | 统一管理 | 关联分析、合规报告 | 依赖其他工具采集 |

## 资产清单模板字段

每条数据资产记录至少包含：

| 字段 | 说明 | 示例 |
|------|------|------|
| 资产ID | 唯一标识 | AST-001 |
| 资产名称 | 可读名称 | 用户订单数据库 |
| 所属系统 | 业务系统名称 | 电商平台-订单中心 |
| 存储类型 | 数据库/文件/对象存储 | MySQL 8.0 |
| 数据库/表/字段 | 层级路径 | ecommerce.orders.order_detail |
| 数据量级 | 行数/文件数 | 1.2亿行 |
| 数据所属部门 | 业务方 | 电商事业部 |
| 是否包含 PII | 是/否/部分 | 是 |
| 初步分类 | 业务分类 | 客户-交易数据 |
| 初步分级 | 安全分级 | 重要数据 |
| 盘点人 | 执行人 | 张三 |
| 盘点日期 | 时间戳 | 2026-05-01 |

## 常见挑战

### 影子 IT
业务部门未经 IT 审批自行采购的 SaaS 服务、自建数据库、本地脚本等形成影子 IT，不在正式资产清单中。应对方法：
- 结合网络流量分析（NTA）发现未登记的数据流
- 使用 CASB 发现未授权的 SaaS 应用
- 建立"资产发现-台账登记-安全评估"的闭环流程

### 临时/短周期资源
大数据平台的临时表、数据管道的中间数据、测试环境快照等短生命周期资源易被遗漏。建议：
- 对动态资源使用标签自动注册机制
- 数据平台层面嵌入资产发现能力（如 Spark 作业自动登记输出数据）
- 定期（月度）对比 CMDB 与实际环境，识别遗漏资产
