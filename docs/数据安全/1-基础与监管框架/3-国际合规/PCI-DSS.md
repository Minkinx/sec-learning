# PCI DSS

> Payment Card Industry Data Security Standard（支付卡行业数据安全标准），由PCI安全标准委员会制定，适用于所有处理、存储或传输持卡人数据的实体。最新版本为PCI DSS v4.0，于2024年3月31日全面生效，替代v3.2.1。

## PCI DSS概览

PCI DSS由五大支付卡品牌（Visa、Mastercard、American Express、Discover、JCB）联合制定，旨在保护持卡人数据安全。合规是强制性的——尽管不是法律，但违反PCI DSS可能导致高额罚款、提高交易费率甚至被禁止受理支付卡交易。

## 6大目标与12项要求（v4.0）

| 目标 | 编号 | 要求 | 说明 |
|------|------|------|------|
| **A. 构建和维护安全网络** | 1 | 安装和维护网络防火墙配置 | 保护CDE边界，限制进出流量 |
| | 2 | 不使用厂商默认密码和默认参数 | 更改所有系统默认凭据 |
| **B. 保护持卡人数据** | 3 | 保护存储的持卡人数据 | 数据最小化：仅存储业务必需的数据 |
| | 4 | 加密传输中的持卡人数据 | 使用TLS 1.2+等强加密协议 |
| **C. 维护漏洞管理计划** | 5 | 使用并定期更新防恶意软件软件 | 覆盖所有受影响的系统 |
| | 6 | 开发并维护安全系统和应用 | 定期更新、漏洞修复、安全编码 |
| **D. 实施强访问控制** | 7 | 按业务需要限制数据访问 | 基于"业务需要知晓（Need-to-Know）"原则 |
| | 8 | 为每个用户分配唯一ID | 禁用共享账号；v4.0强化MFA要求 |
| | 9 | 限制物理访问持卡人数据 | 物理安全：锁柜、监控、访问记录 |
| **E. 定期监控和测试** | 10 | 跟踪和监控网络和数据访问 | 日志记录、日志保护、日志留存至少12个月 |
| | 11 | 定期测试安全系统和流程 | 渗透测试（每年+重大变更后）、ASV季度扫描 |
| **F. 维护信息安全策略** | 12 | 维护针对所有人员的信息安全策略 | 安全策略、风险流程、供应商管理 |

## v4.0关键变化

### 1. 自定义控制 vs. 规定控制
v4.0引入了两种合规验证方式：

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| **规定控制（Defined Approach）** | 严格遵守标准的具体控制要求 | 大多数实体，尤其是合规能力有限的实体 |
| **自定义控制（Customized Approach）** | 通过目标导向的方式自主设计控制措施 | 安全成熟度高、有定制化环境的实体 |

使用自定义方法需要：
- 对目标要求和测试程序的深入理解
- 完善的风险评估文档
- 第三方评估员的审核确认

### 2. MFA要求扩展
| 原要求范围 | v4.0扩展范围 |
|-----------|-------------|
| 仅远程访问CDE | 所有访问CDE的人员（非控制台访问也需MFA） |

### 3. 持续安全验证
从"合规检查"转向"持续安全验证"，强调：
- 持续监控安全控制的有效性
- 自动化合规验证工具的普及
- 安全态势的实时可见性

## CDE（Cardholder Data Environment）

持卡人数据环境包括持有、处理、传输持卡人数据或敏感认证数据的系统组件、人员和流程。

**CDE边界定义四步法：**
1. 识别所有持卡人数据存储位置（数据库、文件、日志、备份）
2. 识别处理、传输持卡人数据的所有系统和网络路径
3. 连接CDE的支撑系统（如认证、日志、管理）也需纳入范围
4. 使用网络分段技术最小化CDE范围

## 商户等级与验证要求

| 等级 | 年交易量（Visa标准） | 验证要求 |
|------|-------------------|---------|
| **1级** | >600万笔交易 | 年度ROC（合规报告）+ QSA现场评估 + ASV季度扫描 |
| **2级** | 100万-600万笔交易 | 年度SAQ（自评问卷）+ ASV季度扫描 |
| **3级** | 2万-100万笔在线交易 | 年度SAQ + ASV季度扫描 |
| **4级** | <2万笔在线交易 | 年度SAQ + ASV季度扫描（由收单行确定具体要求） |

## SAQ类型与适用场景

| SAQ类型 | 适用场景 | 要求数 |
|---------|---------|-------|
| **SAQ A** | 仅外包处理、不存储/处理/传输电子持卡人数据 | 22项 |
| **SAQ A-EP** | 电子商务，外包支付处理但有电子传输影响 | 44项 |
| **SAQ B** | 仅刷卡终端（非电子传输） | 25项 |
| **SAQ B-IP** | 仅独立PA-DSS验证的终端（IP连接） | 28项 |
| **SAQ C** | 连接互联网的支付应用（非电子商务） | 35项 |
| **SAQ C-VT** | 通过虚拟终端人工输入持卡人数据 | 33项 |
| **SAQ D（商户）** | 所有其他商户 | 229项 |
| **SAQ D（服务商）** | 所有服务提供商 | 259项 |

## 合规验证流程

```
第一步：范围界定
├── 识别CDE边界
├── 绘制数据流图（Data Flow Diagram）
└── 确认纳入范围的系统和人员

第二步：差距分析
├── 对照12项要求逐一评估现状
├── 识别不符合项
└── 制定整改计划

第三步：整改实施
├── 技术控制实施（加密、MFA、日志等）
├── 策略和流程编写
└── 人员培训和意识提升

第四步：合规验证
├── 1-2级：聘请QSA进行现场评估（ROC）
├── 3-4级：填写SAQ自评问卷
└── ASV季度漏洞扫描（所有等级）

第五步：提交确认
├── 向收单行或PCI委员会提交ROC/SAQ
├── 提交ASV扫描报告和AOC（合规证明）
└── 保持持续合规
```

## 常见合规难点

| 难点 | 原因 | 解决方案 |
|------|------|---------|
| CDE范围界定不当 | 未充分识别持卡人数据存储位置 | 数据发现扫描+网络流量分析 |
| 日志留存不足 | 日志量大，存储成本高 | SIEM集中管理，分层存储策略 |
| MFA覆盖不全 | 用户抵触，运维成本高 | 分阶段推广，选择用户体验好的MFA方案 |
| SAQ选择错误 | 业务场景复杂 | 逐项对照标准，必要时咨询QSA |
| 供应链合规 | 第三方服务商不合规 | 供应商尽职调查+合同约束+定期审查 |
| 持续合规 | "一次性合规"后管理松懈 | 建立自动化合规监控和定期内部审计 |

## 合规时间线（v4.0过渡）

| 里程碑 | 日期 | 说明 |
|-------|------|------|
| v3.2.1退役 | 2024年3月31日 | v4.0全面生效 |
| 未来生效要求 | 2025年3月31日 | v4.0中标记为"未来生效"的要求正式生效 |
| 持续验证要求 | 持续 | 需要实体持续监控控制有效性 |

**总结**：PCI DSS合规不仅是满足标准的问题，更是建立支付数据安全文化的契机。从v4.0的变革可以看出，行业正从"合规检查"向"持续安全"转变。
