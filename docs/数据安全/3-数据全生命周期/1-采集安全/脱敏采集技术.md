# 脱敏采集技术

## 概述

脱敏采集是指在数据进入采集管道时即对敏感信息进行遮蔽或转换，确保数据在后续存储和处理环节降低敏感度。脱敏时机和方式的选择直接影响安全效果与业务可用性。"数据保护设计"（Data Protection by Design / Privacy by Design）理念要求在系统设计阶段就将脱敏机制嵌入数据采集流程。

## 采集前脱敏 vs 采集后脱敏

| 维度 | 采集前脱敏 (Pre-collection) | 采集后脱敏 (Post-collection) |
|------|-----------------------------|------------------------------|
| 处理位置 | 客户端 / 采集端点 | 服务端采集管道 |
| 原始数据暴露 | 原始数据不进入网络 | 原始数据在采集管道中短暂暴露 |
| 优势 | 隐私保护最强，原始数据不出设备 | 可保留原始数据用于审计和重处理 |
| 风险 | 不可逆，数据质量受损 | 原始数据存在暴露窗口 |
| 适用场景 | 第三方数据采集、高敏感数据 | 内部数据采集系统 |
| GDPR 合规 | 强（data protection by design） | 需额外控制措施 |
| 维护成本 | 需客户端 SDK 升级 | 服务端集中管理 |
| 灵活性 | 低（脱敏策略变更需客户端更新） | 高（策略集中调整） |

**选型建议**：对于高敏感数据（生物识别、医疗数据），优先采用采集前脱敏；对于需保留审计溯源能力的场景，采用采集后脱敏并配合严格的访问控制。

## 采集点动态脱敏 (Dynamic Masking at Collection Point)

在 API Gateway 或采集代理层实施动态脱敏，确保敏感数据在进入内部服务前即被处理：

```
Client → API Gateway (动态脱敏) → Internal Service (脱敏后数据)
                                → Audit Log (原始数据, 受限访问)
```

### 实现方案

| 方案 | 实现方式 | 延迟影响 | 维护成本 | 适用规模 |
|------|----------|----------|----------|----------|
| 基于字段注解 | API 定义中标注敏感字段，gateway 自动脱敏 | <1ms | 低 | 中小规模 |
| 配置驱动 | 脱敏规则引擎（如 Open Policy Agent）动态应用策略 | 2-5ms | 中 | 企业级 |
| 上下文感知 | 根据调用方角色动态调整脱敏强度 | 3-8ms | 高 | 多租户场景 |
| 正则匹配 | 预定义正则识别敏感模式（信用卡号、身份证等） | <1ms | 低 | 快速部署 |

## 可逆脱敏 vs 不可逆脱敏

### 不可逆脱敏 (Irreversible)

| 技术 | 说明 | 适用字段类型 | 安全性 | 数据可用性 |
|------|------|-------------|--------|-----------|
| 哈希 (SHA-256 + Salt) | 单向散列，加盐防彩虹表 | ID、用户名 | 高 | 中（不支持反查） |
| 截断 (Truncation) | 保留部分字符，丢弃其余 | 信用卡号前 6 后 4 | 低-中 | 高 |
| 令牌化 (Tokenization) | 替换为无意义 token，映射表分离存储 | 支付卡号 (PCI DSS) | 高 | 高（一致性） |
| 泛化 (Generalization) | 降低精度，扩大值域 | 年龄→年龄段，地址→城市 | 中-高 | 中 |
| 置零 (Nulling) | 将敏感值替换为 NULL | 非必需字段 | 高 | 低 |

### 可逆脱敏 (Reversible)

| 技术 | 说明 | 安全要求 | 查询支持 | 典型应用 |
|------|------|----------|----------|----------|
| 格式保持加密 (FPE) | 密文保持原格式和校验规则 | 密钥管理必须完善 | 不支持模糊查询 | 卡号、SSN |
| 确定性加密 | 相同明文→相同密文，支持等值查询 | 密钥管理、频率分析风险 | 支持等值查询 | 外键字段 |
| 可调加密 (Tweakable Encryption) | 加入 tweak 参数防频率分析 | 密钥 + tweak 管理 | 需调整查询方式 | 防止关联 |
| 可搜索加密 (SE) | 支持在密文上直接搜索 | 高性能开销 | 支持关键词搜索 | 加密数据库 |

## 差分隐私在采集边缘的应用

### Local Differential Privacy (LDP)

在数据离开用户设备之前，由客户端在本地应用差分隐私机制，确保任何第三方（包括数据采集方）都无法从单条数据中推断个体信息。这是 Privacy by Design 的最强实践形式。

```
用户设备                    数据采集服务
 ┌─────────────┐          ┌──────────────┐
 │ 原始数据: 25 │   LDP   │ 随机响应结果:  │
 │  (真实年龄)  │ ──────→ │  27 (25 概率 p)│
 │             │          │  或随机值(1-p) │
 └─────────────┘          └──────────────┘
 原始数据不离设备           仅收集搅动后的数据
```

### 常见 LDP 机制

| 机制 | 核心思想 | ε 范围 | 适用数据类型 | 实现复杂度 |
|------|----------|--------|-------------|-----------|
| 随机响应 (Randomized Response) | 以概率 p 回答真值，1-p 回答随机值 | 1-10 | 类别型/布尔型 | 低 |
| RAPPOR | Google 提出的编码+随机响应方案 | 3-8 | 类别型（含低频值） | 高 |
| 分段机制 (PM) | 将数值分段后加拉普拉斯/均匀噪声 | 1-5 | 数值型 | 中 |
| 哈达玛变换 (HR) | 哈达玛变换+随机响应组合 | 2-8 | 高维类别数据 | 高 |

### 随机响应示例

以采集用户"是否患有某种疾病"为例，ε=3 的 LDP 实现：

```python
import random
import math

def local_dp_boolean(true_value: bool, epsilon: float) -> bool:
    p = math.exp(epsilon) / (1 + math.exp(epsilon))
    if random.random() < p:
        return true_value  # 真实回应
    else:
        return not true_value  # 随机翻转

# 单个用户采集时调用
reported = local_dp_boolean(user_has_disease, epsilon=3.0)
# 服务端收集大量扰乱结果后，通过校正公式估计真实分布
```

### LDP 采集架构

```
[用户设备 1] → LDP 扰动 → 采集数据₁
[用户设备 2] → LDP 扰动 → 采集数据₂      数据采集服务
[用户设备 3] → LDP 扰动 → 采集数据₃  →  [聚合 & 校正]
[用户设备 4] → LDP 扰动 → 采集数据₄        ↓
[用户设备 5] → LDP 扰动 → 采集数据₅    统计报告 (无个体信息)
    ...                                    ↓
                                    丢弃原始扰动值
```

**核心特性**：
- 单条扰动数据对攻击者无意义（无法区分真值和噪声）
- 数据采集方本身也无法从单条数据中推断个体信息
- 仅聚合后的统计结果（均值、直方图）具有统计学意义
- 适用于 Apple、Google、Microsoft 等大规模遥测场景

### 各隐私保护技术在采集阶段的对比

| 技术 | 保护粒度 | 采集阶段适用 | 统计可用性 | 实现成本 |
|------|----------|-------------|-----------|----------|
| 哈希 | 字段级 | 采集前/后均可 | 不支持统计 | 低 |
| 令牌化 | 字段级 | 采集后 | 不支持统计 | 中 |
| 格式保持加密 | 字段级 | 采集后 | 不支持统计 | 中 |
| 泛化 | 字段级 | 采集前/后均可 | 有限统计 | 低 |
| 本地差分隐私 | 记录级 | 采集前（客户端） | 聚合统计 | 高 |
| 集中式差分隐私 | 查询级 | 采集后（服务端） | 聚合统计 | 中 |

### LDP 实战注意事项

1. **预算分配 (Budget Budgeting)**：多个统计任务的 ε 值需遵循顺序组合定理：
   - 总 ε = ε₁ + ε₂ + ... + εₙ
   - 建议单次采集任务 ε ≤ 3，全年总 ε ≤ 10

2. **校正偏差 (Debiasing)**：聚合时必须对 LDP 引入的噪声进行统计校正：
   ```python
   def estimate_frequency(observed_count: float, n: int, epsilon: float) -> float:
       p = math.exp(epsilon) / (1 + math.exp(epsilon))
       return (observed_count / n - (1 - p)) / (2 * p - 1)
   ```

3. **重放攻击防范**：对相同用户重复采集需加入时间戳防止攻击者通过多次采样平均去除噪声

4. **组合效应**：多个属性分别用 LDP 采集后，组合后的记录仍可能增加重识别风险

## Format-Preserving Encryption (FPE) 在采集管道中的应用

FPE 确保加密后的数据格式与原始数据一致（如信用卡号 16 位数字），在不破坏下游系统格式要求的前提下实现脱敏：

```
原始: 6222-1234-5678-9012
FPE:  4398-7654-3210-8765 (仍为 16 位数字, 符合 Luhn 算法)
```

### FPE 算法选型

| 算法 | 标准 | 特点 | 推荐场景 |
|------|------|------|----------|
| FF1 | NIST SP 800-38G | 支持任意 alphabet, 推荐使用 | 通用场景 |
| FF3-1 | NIST SP 800-38Gr1 | 更高效，但最小明文长度限制 (≥6 字符) | 高性能场景 |
| VAES3 | 非标准 | 仅用于 legacy 系统迁移 | 不推荐新项目 |

### 采集管道中的 FPE 部署

```
采集端点 → FPE 加密 → 传输 → 存储 (密文)
                        ↓
                  密钥服务器 (HSM/KMS)
```

- 加密在采集代理层执行，密钥从 KMS 获取
- 使用 per-field 独立密钥防止关联分析
- 加密上下文中记录算法版本和 key ID 以便轮换
- 配合 FPE 轮换策略：每 90 天更换一次字段密钥
