# 合法合规采集

## 概述

数据采集的合法合规是数据安全治理的基石。在全球监管趋严的背景下，企业必须在业务需求与合规要求之间取得平衡，确保数据采集活动具备合法的法律基础。数据采集合规涉及告知同意、最小必要、目的限制、存储期限、跨境传输等多个法律和技术维度。

## 合法处理基础

根据《个人信息保护法》第 13 条和 GDPR 第 6 条，企业开展数据采集和处理活动必须至少具备以下法律基础之一：

| 处理基础 | 说明 | 典型场景 | 中国 PIPL | 欧盟 GDPR |
|----------|------|----------|-----------|-----------|
| 用户同意 | 用户主动、知情、明确授权 | 营销推送、行为分析 | 第 13 条 | Art. 6(1)(a) |
| 合同必要 | 为履行合同所必需 | 电商订单处理、物流配送 | 第 13 条(二) | Art. 6(1)(b) |
| 法定义务 | 法律法规要求的处理 | 税务申报、反洗钱 | 第 13 条(三) | Art. 6(1)(c) |
| 公共利益 | 为公共安全、公共卫生 | 疫情防控、人口统计 | 第 13 条(四) | Art. 6(1)(e) |
| 合法利益 | 企业正当利益（需平衡测试） | 网络安全、反欺诈 | 第 13 条无直接对应 | Art. 6(1)(f) |

## 告知同意要求

### 同意的有效要件

根据《个人信息保护法》第 14 条和 GDPR 第 7 条，有效的同意需满足以下要素：

| 要素 | 说明 | 检查要点 | 实施方法 |
|------|------|----------|----------|
| 自主性 | 用户自愿作出，不受胁迫 | 拒绝不影响基本服务 | 不捆绑授权 |
| 具体性 | 针对特定处理目的 | 禁止一揽子授权 | 分场景逐项授权 |
| 知情性 | 充分了解处理方式与范围 | 隐私政策清晰易懂 | 分层隐私通知 |
| 明确性 | 通过 affirmative action 作出 | 禁止默认勾选 | 主动勾选/滑动确认 |

### 分层同意机制 (Layered Consent)

```
采集层级         同意粒度                   用户操作
─────────────────────────────────────────────────────────
[L1] 基础功能    账号注册、订单处理          首次使用时同意
[L2] 个性化功能  推荐算法、历史记录          功能开启时授权
[L3] 营销功能    推送通知、邮件营销          单独弹窗征求
[L4] 第三方共享  数据共享、广告追踪          逐项选择加入
```

**最佳实践**：采用分层同意的隐私设置界面，用户可随时在偏好中心调整同意范围。

### 同意生命周期管理

```
用户注册 → 首次同意采集
            ↓
   [偏好中心] ← 随时修改/撤回同意
            ↓
  同意变更 → 记录变更日志
            ↓
  定期重征 → 每年重新确认同意有效性
            ↓
  账户注销 → 删除数据 / 撤销所有授权
```

- 每次同意变更需记录时间戳、用户标识、变更后的同意范围
- 同意的撤回不应影响撤回前基于同意处理的合法性
- 撤回同意的操作流程不应比授权操作更复杂

## 最小必要原则

### 技术实施手段

| 技术手段 | 说明 | 实施方式 |
|----------|------|----------|
| 字段级配置 | 仅采集业务必需的字段 | API 接口定义中标注 required/optional |
| 渐进式采集 | 按功能模块分阶段采集 | 用户首次使用仅采集基础字段 |
| 数据消减 | 采集时即丢弃非必需数据 | 采集函数直接筛选必要字段 |
| 聚合采集 | 采集聚合统计值而非原始值 | 客户端预聚合后上报 |
| 默认关闭 | 非核心功能采集默认关闭 | 用户行为追踪默认 opt-out |

### 最小必要审查清单

```yaml
采集场景: 电商用户注册
必需字段: 手机号/邮箱（账号标识）, 密码（认证）
可选字段: 姓名（可后续补充）, 地址（下单时采集）
禁止字段: 身份证号（非实名场景）, 位置（非配送场景）
审查频率: 每半年审查一次采集字段必要性
```

- **目的关联性**：采集的数据必须与处理目的直接相关，禁止为了"将来可能需要"而过度采集
- **数据最小化**：仅采集实现特定目的所必需的最少数据类型和数量
- **存储期限最小化**：设置数据保留期限，到期自动删除或匿名化
- **定期审查机制**：建立周期性字段必要性审查流程，移除不再必需的字段

## 数据分类分级采集策略

不同类型的数据应适用不同的采集合规要求：

| 数据分类 | 定义 | 采集要求 | 典型字段 |
|----------|------|----------|----------|
| 个人基本信息 | 可识别个体身份的信息 | 明确告知目的，获取同意 | 姓名、身份证号、手机号 |
| 个人敏感信息 | 一旦泄露可能危害人身财产安全 | 单独同意、影响评估 | 金融账户、生物识别、行踪轨迹 |
| 生物识别信息 | 人脸、指纹、虹膜等 | 替代方案的充分告知 | Face ID、指纹模板 |
| 未成年人信息 | 14 周岁以下个人信息 | 监护人同意 | 实名认证 + 监护人确认 |
| 非个人数据 | 不可识别个体的数据 | 告知即可 | 设备型号、匿名化统计数据 |

## 跨境数据传输合规

| 合规机制 | 适用场景 | 实施要求 |
|----------|----------|----------|
| 安全评估 | CII 运营者、处理 100 万人以上个人信息 | 通过国家网信办安全评估 |
| 标准合同 | 非 CII 企业、非大量个人信息 | 签署国家网信办标准合同 |
| 认证 | 跨国公司内部传输 | 获得个人信息保护认证 |
| 充分性认定 | 接收国已获充分保护认定 | 所涉国家在认定目录中 |

## Legitimate Interest 平衡测试

当企业以 legitimate interest 作为处理基础时，需完成三要素测试：

1. **目的测试 (Purpose Test)**：处理目的是否 legitimate？
2. **必要性测试 (Necessity Test)**：该处理是否为实现目的所必要？
3. **平衡测试 (Balancing Test)**：处理对用户权利的影响是否与利益相称？

**测试示例**：
```
场景: 金融机构使用交易数据检测欺诈行为
目的测试: √ (反欺诈是合法目的)
必要性测试: √ (交易数据是检测欺诈的必要输入)
平衡测试: √ (对用户隐私影响小，且保护用户资金安全)
结论: 可基于 Legitimate Interest 处理
```

## Consent Management Platform (CMP)

CMP 是管理用户同意状态的核心基础设施：

| 功能模块 | 说明 | 推荐方案 |
|----------|------|----------|
| 偏好采集 | Cookie Banner / Preference Center | OneTrust CMP |
| 同意存储 | 集中化同意记录 | Consent Database |
| 信号传递 | 向 DSP/SSP 传递同意信号 | IAB TCF 2.0 |
| 审计日志 | 同意变更全记录 | append-only 日志表 |
| 多语言 | 适配运营地区语言 | i18n 框架 |

### 同意状态记录表 Schema

```sql
CREATE TABLE consent_audit_log (
    id            BIGSERIAL PRIMARY KEY,
    user_id       VARCHAR(64) NOT NULL,
    consent_scope JSONB       NOT NULL,  -- { "analytics": true, "marketing": false }
    consent_version VARCHAR(16),         -- "v2.1_202405"
    ip_address    INET,
    user_agent    TEXT,
    created_at    TIMESTAMPTZ DEFAULT NOW(),
    updated_at    TIMESTAMPTZ
);
CREATE INDEX idx_consent_user ON consent_audit_log(user_id, created_at DESC);
```

## 采集范围文档化 — ROPA

通过 ROPA（Record of Processing Activities）管理采集活动：

```
采集活动 ID: DPIA-2024-001
处理目的: 用户行为分析用于产品优化
采集字段: user_id, session_id, page_path, event_type, duration
法律基础: 同意 (Art. 6(1)(a))
数据分类: 个人信息（行为数据）
保留期限: 26 个月，到期自动匿名化
数据流向: Web Server → Kafka → Data Lake → Analytics DB
数据接收方: 内部分析团队（中国）, Amplitude（美国, DPA 已签署）
跨境机制: 标准合同条款 (SCC)
```

## Audit Trail 要求

- 记录每次采集请求的元数据（时间、来源、scope）
- 同意状态的变更历史需保留至少 3 年
- 定期进行合规审计，保留审计报告
- 向监管机构证明合规的能力（accountability principle）
- 依据 SOX/PCI-DSS 要求，审计日志不可篡改
