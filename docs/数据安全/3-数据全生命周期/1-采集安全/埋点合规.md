# 埋点合规

## 概述

用户行为数据采集（埋点）是产品运营的基础设施，但也是数据合规风险的高发区。个保法、GDPR、ePrivacy Directive 等多重法规对追踪行为提出了严格约束。

## 个保法对追踪的要求

《个人信息保护法》对埋点采集的核心要求：

| 要求 | 具体内容 | 合规措施 |
|------|----------|----------|
| 告知义务 | 明确告知收集方式、范围、目的 | 隐私政策中描述所有埋点 |
| 单独同意 | 敏感信息需单独同意 | 弹窗获取 explicit consent |
| 拒绝权 | 用户有权拒绝 tracking | 提供 opt-out 机制 |
| 撤回权 | 用户可随时撤回同意 | Preference Center 支持撤回 |
| 删除权 | 用户可要求删除已采集数据 | 数据删除 API 支持 |

## User Opt-Out 机制

实现跨平台的 opt-out 能力：

```
Web:    Cookie Opt-Out → 设置 opt-out cookie + localStorage flag
Mobile: IDFA/AAID reset → 重置广告标识符
App:    系统 Tracking Authorization (iOS ATT / Android Privacy Sandbox)
全局:   Global Privacy Control (GPC) 信号识别
```

### Opt-Out 技术实现要点

- **持久性**：opt-out 信号需跨会话保留
- **同步性**：用户同意状态在所有终端实时同步
- **优先级**：opt-out 优先级高于默认同意设置
- **审计**：opt-out 操作记录需纳入审计日志

## 基于同意的追踪框架

### Consent Categories

```
必要追踪 (Strictly Necessary): 无需同意
  └── 会话维持、CSRF token、负载均衡
功能追踪 (Functionality): 需告知
  └── 用户偏好、语言设置
分析追踪 (Analytics): 需同意 (GDPR)
  └── 页面浏览量、点击事件、漏斗分析
营销追踪 (Marketing): 需明确同意
  └── 广告追踪、跨站追踪、画像建立
```

### 同意采集接口规范

```javascript
// IAB TCF 2.2 标准化同意字符串
```

## GDPR ePrivacy Directive

ePrivacy Directive (2002/58/EC) 对追踪的核心要求：

- **存储/访问设备信息需经同意**（Cookie Law）
- **严格必要性例外**：仅通信必需的技术 cookie 可豁免
- **Cookie Wall 争议**：以服务访问为前提强制同意可能违规
- **追踪列表更新**：第三方追踪伙伴变更需重新获取同意

## Cookie Consent 实现

| 要求 | 实现方式 |
|------|----------|
| 明确同意 | 主动勾选 + 保存记录，非默认 pre-ticked |
| 细粒度控制 | 按 purpose 分类展示 toggle |
| 撤回机制 | Preference Center 提供一键撤回 |
| 生命周期 | Cookie 最长期限不超过 13 个月 |
| 透明度 | Cookie 清单展示每个 cookie 的名称、用途、到期时间 |

## First-Party vs Third-Party Tracking

| 维度 | First-Party | Third-Party |
|------|-------------|-------------|
| 数据归属 | 自身业务域 | 第三方服务商 |
| 控制权 | 完全控制 | 受限 |
| 合规风险 | 较低 | 高（第三方共享需告知） |
| 阻断影响 | 无 | 浏览器 ITP/ETP 限制 |
| 典型场景 | 自建分析平台 | Google Analytics、广告追踪 |

## 分析匿名化技术

- **差分隐私**：在统计数据中添加 calibrated noise
- **k-anonymity**：确保每条记录至少与 k-1 条记录不可区分
- **数据聚合**：仅上报聚合指标而非事件级数据
- **本地化处理**：在客户端完成统计后仅上报摘要
- **剥离标识符**：从事件数据中移除 device_id, user_id 等直接标识
