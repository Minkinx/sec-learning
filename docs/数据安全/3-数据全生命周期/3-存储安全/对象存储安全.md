# 对象存储安全

## 概述

对象存储（如 AWS S3、MinIO、Azure Blob Storage）已成为企业数据存储的主流选择。其扁平命名空间、RESTful API 和丰富的访问控制策略带来了独特的安全挑战。对象存储安全需要关注访问控制、加密、日志审计和数据治理等多个方面。

## S3-Compatible 存储安全架构

### 安全模型层次

```
Account/Project Level
  └── IAM Policies (用户/角色级别授权)
       └── Bucket Policies (存储桶级访问控制)
            └── ACLs (对象级访问控制, 已不推荐)
                 └── Presigned URLs (临时访问授权)
                      └── Object Ownership (对象所有权)
```

### 配置评估优先级

```yaml
# 推荐的安全配置清单
1. Block Public Access (阻止公共访问)
2. Enable Bucket Versioning (启用版本控制)
3. Enable Server-Side Encryption (启用服务端加密)
4. Enable Access Logging (启用访问日志)
5. Implement Object Lock (实施对象锁定)
6. Enable MFA Delete (多因子删除)
7. Configure Lifecycle Policies (配置生命周期策略)
8. Enable Data Events in CloudTrail (云审计跟踪)
```

## Bucket Policies vs IAM

| 维度 | IAM Policies | Bucket Policies |
|------|--------------|-----------------|
| 绑定对象 | User/Role/Group | Bucket |
| 控制范围 | 跨服务授权 | 仅对象存储 |
| 权限类型 | 受信实体操作权限 | 谁可以访问 Bucket |
| 跨账号访问 | 必须配合 Bucket Policy | 原生支持 |
| 条件键 | IAM 全局条件 | Bucket-specific 条件 |
| 策略复杂度 | 支持多种服务 | 仅对象存储操作 |

### 最佳实践：最小权限原则

```json
// Bucket Policy 示例：仅允许特定 VPC 访问
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::data-bucket",
        "arn:aws:s3:::data-bucket/*"
      ],
      "Condition": {
        "StringNotEquals": {
          "aws:sourceVpc": "vpc-12345678"
        }
      }
    }
  ]
}
```

## Access Logging

对象存储访问日志是安全审计和异常检测的基础数据源。

### 日志字段清单

| 字段 | 描述 | 安全用途 |
|------|------|----------|
| Bucket Owner | 存储桶拥有者账号 | 归属审计 |
| Bucket | 存储桶名称 | — |
| Time | 请求时间 | 时序分析 |
| Remote IP | 客户端 IP | 来源追踪 |
| Requester | 请求者 ARN | 用户审计 |
| Request ID | 请求唯一标识 | 链路追踪 |
| Operation | RESTful API 操作 | 行为分析 |
| Key | 对象 Key | 数据识别 |
| Request-URI | 完整请求路径 | — |
| HTTP Status | 响应状态码 | 异常检测 |
| Error Code | 错误码 | 错误分析 |
| Bytes Sent | 传输字节数 | 数据量监控 |
| Object Size | 对象大小 | — |
| Total Time | 处理时间 | 性能监控 |
| Turn-Around Time | 延迟 | 性能监控 |
| Referrer | HTTP Referer | 来源分析 |
| User-Agent | 客户端 Agent | 设备识别 |
| Version ID | 版本 ID | 版本审计 |

## 对象加密选项

### SSE-S3 / SSE-KMS / SSE-C 对比

| 特性 | SSE-S3 | SSE-KMS | SSE-C |
|------|--------|---------|-------|
| 密钥管理 | AWS 托管 | AWS KMS 管理 | 客户管理 |
| 加密算法 | AES-256 | AES-256 | AES-256 |
| 密钥轮换 | 自动 | 可配置 | 手动 |
| 审计 | 有限 | 完整 (KMS CloudTrail) | 无 |
| 额外费用 | 无 | KMS API 费用 | 无 |
| 合规支持 | 基础 | 强 (FIPS) | 最强 |
| 使用复杂度 | 低 | 中 | 高 |

### 客户端加密 vs 服务端加密

```yaml
服务端加密 (SSE):
  数据在 S3 服务端加密
  密钥管理可选 (S3/KMS/客户)
  对应用透明
  传输路径可能暴露明文 (HTTPS→S3)

客户端加密 (CSE):
  数据在客户端加密后上传
  服务端看不到明文
  完全控制密钥
  需管理客户端 SDK + 密钥
  查询/搜索受限
```

## 版本控制与 MFA Delete

### Versioning 安全价值

- **数据恢复**：防止误删除和覆盖
- **审计追溯**：查看对象所有版本变更
- **合规留存**：配合 Object Lock 实现 WORM

### MFA Delete 配置

MFA Delete 要求删除对象版本时必须通过多因子认证：

```bash
# 启用 MFA Delete (需 root 用户)
aws s3api put-bucket-versioning \
  --bucket my-secure-bucket \
  --versioning-configuration Status=Enabled,MFADelete=Enabled \
  --mfa "arn:aws:iam::123456789:mfa/root-account-mfa-device 123456"
```

**注意**：一旦启用 MFA Delete，所有版本删除操作（包括控制台/SDK）都需要 MFA。

## Retention Policies

### Object Lock 模式

| 模式 | 说明 | 使用场景 |
|------|------|----------|
| Governance Mode | 特殊权限用户可覆盖 | 常规合规保留 |
| Compliance Mode | 任何用户（包括 root）不可覆盖 | 严格合规要求 |
| Legal Hold | 基于"诉讼保留"标记 | 法律调查期间 |

### Lifecycle 策略

```yaml
生命周期规则:
  当前版本:
    - 30 天: 迁移到 Standard-IA
    - 90 天: 迁移到 Glacier Instant Retrieval
    - 180 天: 迁移到 Deep Archive
    - 365 天: 过期删除

  非当前版本:
    - 立即删除 (如果不需要版本恢复)
    - 或保留 30 天后删除
```
