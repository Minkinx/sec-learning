# 加密存储技术

## 概述

加密存储是数据安全的核心防线。从数据库层的透明加密到应用层的字段级加密，不同层次的加密策略提供差异化的安全保护粒度。选择合适的加密层次和密码学原语是安全架构师的关键决策。

## Encryption at Rest 层次模型

```
Application Layer: 应用层加密 (字段级/列级)
        ↓
Database Layer: TDE (表空间级/页级)
        ↓
Filesystem Layer: 文件系统加密 (eCryptfs/LUKS)
        ↓
Storage Layer: 存储加密 (SAN/NAS/SSD Self-Encrypting)
        ↓
Physical Layer: 物理介质加密 (FIPS 140-2 硬件)
```

## TDE (Transparent Data Encryption)

TDE 在数据库层自动加密写入磁盘的数据，对应用透明。

### 各数据库 TDE 实现对比

| 数据库 | 加密粒度 | 密钥管理 | 性能影响 |
|--------|----------|----------|----------|
| SQL Server TDE | 数据库级 (Pages) | 证书 (DMK → SMK → DEK) | 3-5% overhead |
| Oracle TDE | 列级/表空间级 | Wallet (Master Key + TDE Key) | 5-8% overhead |
| MySQL InnoDB TDE | 表空间级 | Keyring Plugin | 5-10% overhead |
| PostgreSQL (pg_tde) | 表级 | External KMS | 3-8% overhead |
| MongoDB Encryption | WiredTiger 存储引擎 | Local Key Manager/KMS | 10-15% overhead |

### TDE 密钥架构 (以 SQL Server 为例)

```
Windows Data Protection API (DPAPI)
  └── Service Master Key (SMK)
       └── Database Master Key (DMK)
            └── Certificate/Asymmetric Key
                 └── Database Encryption Key (DEK)
                      └── 数据加密 (AES-256)
```

## Column-Level Encryption

针对特定敏感列进行加密，粒度更细但需权衡查询性能。

| 特点 | 说明 |
|------|------|
| 保护范围 | 仅标记列，非全表 |
| 查询支持 | DETERMINISTIC (等值查询) / RANDOMIZED (仅检索) |
| 索引支持 | 确定性加密可建索引 |
| 应用透明性 | 需改造 SQL (解密函数) |
| 密钥粒度 | 可 per-column per-user |

### 实现方式对比

```sql
-- 方法 1: 数据库内置函数 (如 SQL Server Always Encrypted)
INSERT INTO users (name, ssn) 
VALUES ('张三', ENCRYPTBYKEY(Key_GUID('SSN_Key'), '123-45-6789'));

-- 方法 2: 应用层加密后存入
INSERT INTO users (name, encrypted_ssn) 
VALUES ('张三', 'AES256_GCM_BASE64_ENCODED_CIPHERTEXT');
```

## Application-Level Encryption

在应用代码中实现加密，提供最强控制力：

```
Client App → [Encrypt] → API Server → [Store Encrypted] → Database
                 ↑
           Key Management Service
```

### 优势与挑战

| 维度 | 优势 | 挑战 |
|------|------|------|
| 安全控制 | 精细到字段级，密钥与应用隔离 | 密钥分布管理 |
| 数据库无关 | 任何数据库/存储均可 | 查询、索引受限 |
| 端到端加密 | 数据库管理员不可见 | 对搜索、排序不友好 |
| 合规审计 | 满足严格合规要求 | 需开发额外 filtering 层 |

## 密钥层次结构 (Key Hierarchy)

采用密钥层次结构可以将数据加密与主密钥管理解耦：

### 信封加密 (Envelope Encryption)

```
Customer Master Key (CMK) — KMS 管理, HSM 保护
  └── Key Encryption Key (KEK) — 加密 DEK
       └── Data Encryption Key (DEK) — 实际加密数据

加密流程:
1. KMS 生成 DEK
2. KMS 使用 CMK 加密 DEK → 存储 Encrypted DEK
3. 使用 DEK 加密数据
4. 丢弃明文 DEK, 仅保留 Encrypted DEK

解密流程:
1. 读取 Encrypted DEK
2. KMS 使用 CMK 解密 DEK
3. 使用 DEK 解密数据
4. 解密完成后丢弃 DEK
```

## AEAD 模式

AEAD (Authenticated Encryption with Associated Data) 模式同时提供机密性和完整性校验：

| 模式 | 加密 + 认证 | 典型场景 |
|------|-------------|----------|
| AES-256-GCM | AEAD，12 字节 nonce，16 字节 tag | TLS 1.2/1.3，通用加密 |
| AES-256-CCM | AEAD，CTR + CBC-MAC | IoT，资源受限设备 |
| ChaCha20-Poly1305 | AEAD，流密码 + 多项式 MAC | 移动设备，无 AES-NI 场景 |
| AES-GCM-SIV | AEAD，非ce 误用容忍 | 多线程高并发场景 |

### GCM 模式注意事项

```
安全风险: Nonce 重用将完全破坏安全性
解决方案: 
- 使用 96-bit random nonce (12 字节)
- 使用 deterministic 计数器 (确保全局唯一)
- 每次加密使用唯一 nonce
```

## 加密算法对比分析

### 主流对称加密算法

| 算法 | 密钥长度 | 分组大小 | 安全强度 | 性能 (AES-NI) | 国密替代 | 推荐场景 |
|------|---------|---------|----------|---------------|----------|----------|
| AES-128-GCM | 128 bit | 128 bit | 128 bit | 最快 | SM4-GCM | 通用推荐，性价比最优 |
| AES-256-GCM | 256 bit | 128 bit | 256 bit | 快（比 AES-128 慢 10-20%） | SM4-GCM | 高安全场景、机密数据 |
| ChaCha20-Poly1305 | 256 bit | 流密码 | 256 bit | 无 AES-NI 时快 | — | 移动设备、无 AES-NI 硬件 |
| SM4-CBC | 128 bit | 128 bit | 128 bit | 无硬件加速 | — | 中国合规场景 |
| SM4-GCM | 128 bit | 128 bit | 128 bit | 无硬件加速 | — | 中国合规 + 认证加密 |
| 3DES | 168 bit | 64 bit | 112 bit | 极慢 | — | 已弃用 |
| RC4 | 40-2048 bit | 流密码 | < 64 bit | — | — | 已禁止 |

### AES-256-GCM vs SM4-GCM 对比

| 对比维度 | AES-256-GCM | SM4-GCM |
|----------|-------------|---------|
| 算法设计者 | NIST (美国) | 中国国家密码管理局 |
| 标准化 | FIPS 197, NIST SP 800-38D | GB/T 32907-2016 |
| 密钥长度 | 256 bit | 128 bit |
| 硬件加速 | 广泛支持 (AES-NI, ARMv8 Crypto Extensions) | 国产 CPU 支持（兆芯、海光） |
| 认证加密 | 原生 GCM 模式支持 | SM4-GCM 模式支持 |
| 国际认可 | 全球通用 | 主要中国认可 |
| 性能表现 | 约 1-2 GB/s (AES-NI) | 约 200-400 MB/s (软件实现) |
| 适用场景 | 国际化业务、高性能场景 | 中国合规、政务系统、等保三级 |

**选型建议**：
- 国际化业务优先选择 AES-256-GCM
- 中国合规场景（等保三级、密评）使用 SM4-GCM
- 可采用"双算法"策略：对同一数据同时使用 AES-256 和 SM4 加密（不同字段），满足跨境合规
- 关注性能影响：软件实现下 SM4 约为 AES-NI 硬件 AES-256 速度的 20-30%

### 非对称加密算法对比

| 算法 | 密钥长度 | 安全强度 | 速度 | 推荐替代 |
|------|---------|----------|------|----------|
| RSA-2048 | 2048 bit | 112 bit | 慢 | ECDSA P-256 |
| RSA-4096 | 4096 bit | 128 bit | 更慢 | ECDSA P-384 |
| ECDSA P-256 | 256 bit | 128 bit | 快 | — |
| ECDSA P-384 | 384 bit | 192 bit | 中 | — |
| Ed25519 | 256 bit | 128 bit | 极快 | 新项目推荐 |
| SM2 | 256 bit | 128 bit | 中 | 中国合规场景 |

## 加密层次对比：全盘加密 vs 文件系统加密 vs 数据库列加密

选择合适的加密层次取决于安全需求、性能要求和运维复杂度：

| 对比维度 | 全盘加密 (FDE) | 文件系统加密 (FSE) | 数据库列加密 |
|----------|---------------|-------------------|-------------|
| 保护范围 | 整个磁盘/分区 | 指定文件和目录 | 数据库指定列 |
| 粒度 | 粗（全盘） | 中（文件级） | 细（列级） |
| 性能影响 | 1-3% overhead | 3-8% overhead | 10-30% overhead |
| 查询支持 | 完全支持（操作系统层） | 完全支持（文件系统层） | 受限（需解密函数） |
| 密钥管理 | LUKS/BitLocker/FileVault | eCryptfs/EncFS | KMS/per-column 密钥 |
| 管理复杂度 | 低 | 中 | 高 |
| 物理窃取防护 | 强 | 强 | 弱（取决于上层实现） |
| DBA 不可见数据 | 否（OS 挂载后 DBA 可见） | 否（挂载后 DBA 可见） | 是（仅应用可见明文） |
| 适用数据 | 全部数据 | 部分敏感文件 | 高敏感字段（PII/PCI） |
| 典型工具 | LUKS, BitLocker, dm-crypt | eCryptfs, fscrypt, APFS | Always Encrypted, TDE |
| 密钥轮换难度 | 低 | 中 | 高 |
| 备份兼容性 | 备份自动加密 | 需验证备份加密 | 密文备份 |

### 分层加固推荐

对于高安全需求系统，建议组合使用多种加密层次形成纵深防御：

```
[1] 物理层:    自加密 SSD (SED) / FIPS 140-2 硬件
     ↓
[2] 全盘加密:  LUKS2 + AES-256-XTS (dm-crypt)
     ↓
[3] 文件系统:  fscrypt (特定目录)
     ↓
[4] 数据库:    TDE (表空间级) + 列加密 (敏感字段)
```

**纵深防御原则**：即使上层加密被绕过，下层加密仍可提供保护；即使物理介质被盗，FDE 保护数据；即使 DBA 访问数据库，列加密保护敏感字段。

### 各层加密性能测试参考数据

| 加密层次 | 延迟影响 | 吞吐量影响 | CPU 额外开销 |
|----------|----------|-----------|-------------|
| 全盘加密 (LUKS + AES-256-XTS) | +0.1-0.5 ms | -1-3% | 2-5% |
| 文件系统 (fscrypt) | +0.5-2 ms | -3-8% | 5-10% |
| TDE (SQL Server AES-256) | +1-3 ms | -3-5% | 3-8% |
| 列加密 (Always Encrypted) | +5-20 ms | -10-30% | 10-25% |
