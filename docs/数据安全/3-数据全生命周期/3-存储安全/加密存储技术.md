# 加密存储技术

## 概述

加密存储是数据安全的核心防线。从数据库层的透明加密到应用层的字段级加密，不同层次的加密策略提供差异化的安全保护粒度。选择合适的加密层次和密码学原语是安全架构师的关键决策。

## Encryption at Rest 层次模型

```
Application Layer: 应用层加密 (字段级/列级)
        ↓
Database Layer: TDE (表空间级/页级)
        ↓
Filesystem Layer: 文件系统加密 (eCryptfs/LUKS)
        ↓
Storage Layer: 存储加密 (SAN/NAS/SSD Self-Encrypting)
        ↓
Physical Layer: 物理介质加密 (FIPS 140-2 硬件)
```

## TDE (Transparent Data Encryption)

TDE 在数据库层自动加密写入磁盘的数据，对应用透明。

### 各数据库 TDE 实现对比

| 数据库 | 加密粒度 | 密钥管理 | 性能影响 |
|--------|----------|----------|----------|
| SQL Server TDE | 数据库级 (Pages) | 证书 (DMK → SMK → DEK) | 3-5% overhead |
| Oracle TDE | 列级/表空间级 | Wallet (Master Key + TDE Key) | 5-8% overhead |
| MySQL InnoDB TDE | 表空间级 | Keyring Plugin | 5-10% overhead |
| PostgreSQL (pg_tde) | 表级 | External KMS | 3-8% overhead |
| MongoDB Encryption | WiredTiger 存储引擎 | Local Key Manager/KMS | 10-15% overhead |

### TDE 密钥架构 (以 SQL Server 为例)

```
Windows Data Protection API (DPAPI)
  └── Service Master Key (SMK)
       └── Database Master Key (DMK)
            └── Certificate/Asymmetric Key
                 └── Database Encryption Key (DEK)
                      └── 数据加密 (AES-256)
```

## Column-Level Encryption

针对特定敏感列进行加密，粒度更细但需权衡查询性能。

| 特点 | 说明 |
|------|------|
| 保护范围 | 仅标记列，非全表 |
| 查询支持 | DETERMINISTIC (等值查询) / RANDOMIZED (仅检索) |
| 索引支持 | 确定性加密可建索引 |
| 应用透明性 | 需改造 SQL (解密函数) |
| 密钥粒度 | 可 per-column per-user |

### 实现方式对比

```sql
-- 方法 1: 数据库内置函数 (如 SQL Server Always Encrypted)
INSERT INTO users (name, ssn) 
VALUES ('张三', ENCRYPTBYKEY(Key_GUID('SSN_Key'), '123-45-6789'));

-- 方法 2: 应用层加密后存入
INSERT INTO users (name, encrypted_ssn) 
VALUES ('张三', 'AES256_GCM_BASE64_ENCODED_CIPHERTEXT');
```

## Application-Level Encryption

在应用代码中实现加密，提供最强控制力：

```
Client App → [Encrypt] → API Server → [Store Encrypted] → Database
                 ↑
           Key Management Service
```

### 优势与挑战

| 维度 | 优势 | 挑战 |
|------|------|------|
| 安全控制 | 精细到字段级，密钥与应用隔离 | 密钥分布管理 |
| 数据库无关 | 任何数据库/存储均可 | 查询、索引受限 |
| 端到端加密 | 数据库管理员不可见 | 对搜索、排序不友好 |
| 合规审计 | 满足严格合规要求 | 需开发额外 filtering 层 |

## 密钥层次结构 (Key Hierarchy)

采用密钥层次结构可以将数据加密与主密钥管理解耦：

### 信封加密 (Envelope Encryption)

```
Customer Master Key (CMK) — KMS 管理, HSM 保护
  └── Key Encryption Key (KEK) — 加密 DEK
       └── Data Encryption Key (DEK) — 实际加密数据

加密流程:
1. KMS 生成 DEK
2. KMS 使用 CMK 加密 DEK → 存储 Encrypted DEK
3. 使用 DEK 加密数据
4. 丢弃明文 DEK, 仅保留 Encrypted DEK

解密流程:
1. 读取 Encrypted DEK
2. KMS 使用 CMK 解密 DEK
3. 使用 DEK 解密数据
4. 解密完成后丢弃 DEK
```

## AEAD 模式

AEAD (Authenticated Encryption with Associated Data) 模式同时提供机密性和完整性校验：

| 模式 | 加密 + 认证 | 典型场景 |
|------|-------------|----------|
| AES-256-GCM | AEAD，12 字节 nonce，16 字节 tag | TLS 1.2/1.3，通用加密 |
| AES-256-CCM | AEAD，CTR + CBC-MAC | IoT，资源受限设备 |
| ChaCha20-Poly1305 | AEAD，流密码 + 多项式 MAC | 移动设备，无 AES-NI 场景 |
| AES-GCM-SIV | AEAD，非ce 误用容忍 | 多线程高并发场景 |

### GCM 模式注意事项

```
安全风险: Nonce 重用将完全破坏安全性
解决方案: 
- 使用 96-bit random nonce (12 字节)
- 使用 deterministic 计数器 (确保全局唯一)
- 每次加密使用唯一 nonce
```
