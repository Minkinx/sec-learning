# 数据库安全审计

## 概述

数据库安全审计是数据安全治理的核心环节。通过持续监控和记录数据库活动，企业可以及时发现异常行为、满足合规要求、追踪数据泄露源头。数据库审计需要覆盖查询、变更、权限和访问模式等多个维度。

## 数据库审计日志方案对比

| 数据库 | 审计技术 | 性能影响 | 日志存储 | 粒度 |
|--------|----------|----------|----------|------|
| Oracle | Oracle Audit Vault + Fine-Grained Auditing (FGA) | 5-15% | OS files / SYSAUX | DML, DDL, SELECT |
| SQL Server | SQL Server Audit (Server/DB Level) | 3-10% | .sqlaudit files | Action Groups / Objects |
| MySQL | MySQL Enterprise Audit (audit_log plugin) | 5-10% | JSON/XML/CSV | Class/Subclass |
| PostgreSQL | pgAudit extension | 5-15% | CSV/JSON | Session/Object |
| MongoDB | MongoDB Audit | 5-10% | BSON/JSON/Console | Schema/System/Auth |
| MariaDB | MariaDB Audit Plugin | 5-10% | Syslog/File | Server/Database |

### Oracle Fine-Grained Auditing (FGA)

```sql
-- 创建 FGA 策略：监控对敏感列的查询
BEGIN
  DBMS_FGA.ADD_POLICY(
    object_schema   => 'HR',
    object_name     => 'EMPLOYEES',
    policy_name     => 'audit_ssn_access',
    audit_condition => 'salary > 100000',
    audit_column    => 'ssn, salary',
    handler_schema  => 'SEC_ADMIN',
    handler_module  => 'alert_ssn_access',
    enable          => TRUE
  );
END;
```

### PostgreSQL pgAudit

```sql
-- 数据定义语言审计
shared_preload_libraries = 'pgaudit'
pgaudit.log = 'write,ddl,role'
pgaudit.log_level = 'notice'
pgaudit.log_catalog = off
pgaudit.log_relation = on
pgaudit.log_statement_once = off

-- 对象级审计
CREATE TABLE audit_config (id int, data text);
SECURITY LABEL FOR pgaudit ON TABLE audit_config 
  IS 'READ,WRITE';
```

## DAM (Database Activity Monitoring)

DAM 通过网络嗅探或数据库代理实时分析数据库流量，实现无侵入的审计监控。

### DAM 架构模式

```
模式 1: 网络嗅探 (Inline/TAP)
[Application] ──→ [Network TAP] ──→ [Database]
                        ↓
                  [DAM Analyzer]

模式 2: 数据库代理
[Application] ──→ [DAM Proxy] ──→ [Database]
                        ↓
                  [DAM Analyzer]

模式 3: 日志解析
[Database] ──→ [Audit Logs] ──→ [Log Collector] ──→ [DAM Analyzer]
```

### DAM 核心能力

| 能力 | 说明 | 技术实现 |
|------|------|----------|
| SQL 分析 | 解析所有 SQL 语句 | SQL 语法解析 |
| 行为基线 | 建立正常访问模式 | ML 模型分析 |
| 异常检测 | 识别异常查询行为 | 规则引擎 + 机器学习 |
| 实时告警 | SQL 注入/异常访问 | 实时规则匹配 |
| 会话关联 | 追踪用户完整操作链路 | 应用层 + 数据库层关联 |
| 敏感数据发现 | 识别返回结果中的敏感数据 | 正则 + 模式匹配 |

### 主流 DAM 方案

- **商业**：Imperva SecureSphere (现 Thales)、Guardium、McAfee Database Security
- **云原生**：AWS Database Activity Streams、Azure SQL Auditing 内置 DAM
- **开源**：pgBadger (PostgreSQL 日志分析)、MySQL Enterprise Audit 日志 + ELK

## 审计策略设计

### 分层审计策略

```yaml
审计层级:
  第 1 层 - 关键操作审计 (DDL, DCL, 权限变更):
    - 审计所有 DDL 操作
    - 审计用户创建/删除/角色变更
    - 审计权限授予/回收
    - 保留期限: 5 年

  第 2 层 - 敏感数据访问审计:
    - 审计查询 PII/PCI/PHI 列的操作
    - 审计批量导出 (SELECT INTO/OUTFILE)
    - 审计异常时间段访问
    - 保留期限: 3 年

  第 3 层 - 常规操作审计:
    - 审计 DML 操作摘要
    - 审计连接/断连
    - 审计失败登录
    - 保留期限: 1 年
```

### 审计字段标准

| 字段 | 说明 | 示例 |
|------|------|------|
| event_id | 事件唯一 ID | 5f8a3b2c-... |
| timestamp | 精确到毫秒 | 2024-05-15T10:30:00.123Z |
| session_id | 数据库会话 ID | 4928 |
| user_name | 数据库用户名 | app_user |
| os_user | 操作系统用户 | root |
| host | 客户端 IP/主机名 | 10.0.1.50 |
| program_name | 连接程序名 | jdbc:mysql:// |
| sql_text | SQL 语句 | SELECT * FROM employees |
| object | 操作对象 | HR.EMPLOYEES |
| rows_affected | 影响行数 | 1500 |

## Privileged User Monitoring

特权用户（DBA、系统管理员）需要额外的监控措施：

| 监控措施 | 说明 | 实现方式 |
|----------|------|----------|
| 会话录制 | 录制 DBA 操作的完整会话 | 基于 Jump Server / Bastion Host |
| 命令审计 | 记录所有执行命令 | OS audit + DB audit 联合 |
| 权限提升审计 | 监控临时权限提升操作 | 即时告警 |
| 分离职责 | DBA 不能修改审计日志 | 审计日志写入 append-only 存储 |
| 定期审查 | 定期审查特权用户活动 | 手工 + 自动化报告 |

## 审计日志分析

### 异常模式检测

```
时间序列模式:
  20:00-06:00 的高频查询         → 疑似数据窃取
  批量导出大量数据               → 异常数据迁移
  频繁失败登录后成功            → 暴力破解
  同一用户多地同时连接          → 凭证泄露
  查询从未访问过的敏感表       → 权限探索
  异常大的结果集               → 批量数据外泄
```

### 分析技术栈

```
日志采集: Filebeat / Fluentd / Audit Log Forwarder
  └── 中央存储: Kafka / S3 / Data Lake
       └── 处理引擎: Spark / Flink
            └── 搜索/告警: Elasticsearch + Kibana / Splunk
                 └── 分析: 规则引擎 + ML (Isolation Forest / LSTM)
```
