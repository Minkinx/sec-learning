# 云数据销毁

## 概述

云环境中的数据销毁与本地环境有本质差异。由于多租户共享、虚拟化抽象和 API 驱动的管理方式，云数据销毁面临验证困难、残留数据和共享责任等独特挑战。云用户需要依赖云服务商提供的删除机制，同时承担验证和审计的责任。

## 云存储删除机制

### S3 Object Deletion

```
对象级删除:
  DELETE /bucket/object/key → HTTP 204
  - 立即标记为删除
  - 后台异步清理存储空间
  - 新 PUT 请求返回 404

批量删除:
  POST /bucket/delete → HTTP 200
  - 一次可删除最多 1000 个对象
  - 部分失败时返回错误列表

版本控制下的删除:
  未启用版本控制: 永久删除
  启用版本控制: 添加 DeleteMarker (软删除)
  指定 VersionId: 永久删除该版本

多版本删除陷阱:
  - 不指定 VersionId 仅添加 DeleteMarker
  - 需要显式删除所有版本（含 DeleteMarker）才能彻底清除
```

### EBS Volume Deletion

```
EBS Volume 删除流程:
  1. 确认 Volume 状态为 available (已从实例分离)
  2. DeleteVolume API → HTTP 200
  3. 存储空间标记为可用
  4. 后台零化 (Zeroization) 过程
  
残余数据风险:
  - 删除后数据不会立即从物理介质清除
  - 新 Volume 不会看到旧数据 (存储分配前已清零)
  - AWS 声明 "Storage is scrubbed before reuse"
  
Snapshot 关联:
  - 删除 Volume 不影响已创建的 Snapshot
  - Snapshot 包含创建时间点的全量数据
  - 必须同时删除 Snapshot 才能彻底清除数据
```

### 其他存储类型删除

| 存储类型 | 删除方式 | 残余风险 | 验证方法 |
|----------|----------|----------|----------|
| S3 Standard | DELETE API | 后台异步清理 | GET 返回 404 |
| S3 Glacier | DELETE API + 等待恢复窗口 | 有恢复窗口期 | lifecycle 策略 |
| EBS Volume | DeleteVolume API | 需同时删 Snapshot | API 状态检查 |
| RDS Instance | DeleteDBInstance (SkipFinalSnapshot) | 检查快照残留 | describe API |
| DynamoDB Table | DeleteTable API | 立即生效 | describe API |
| CloudWatch Logs | DeleteLogGroup | 需删除底层 Log Stream | retention 策略 |

## Termination of Cloud Resources

### 资源终止最佳实践

```yaml
资源终止清单:
  1. 数据存储服务:
     - [ ] 删除所有 S3 Objects 和 Bucket
     - [ ] 删除所有 EBS Volumes 和 Snapshots
     - [ ] 删除所有 RDS Instances 及自动备份
     - [ ] 删除 DynamoDB Tables
     - [ ] 删除 ElastiCache Clusters
     - [ ] 删除 EFS File Systems

  2. 中间件/消息队列:
     - [ ] 删除所有 Kafka Topics
     - [ ] 删除所有 SQS Queues
     - [ ] 清除其上的消息

  3. 容器/计算:
     - [ ] 保留 ECR 镜像到无敏感数据后删除
     - [ ] 清理容器日志 (CloudWatch/EFS)
     - [ ] 删除 EC2 Instance Store Volumes

  4. Kubernetes (EKS/AKS/GKE):
     - [ ] 删除所有 PV 和 PVC
     - [ ] 清理 ConfigMaps/Secrets
     - [ ] 删除 Persistent Volumes (含 underlying 存储)

  5. 备份/归档:
     - [ ] 删除所有 Snapshot
     - [ ] 删除所有 Backup Vaults
     - [ ] 删除所有 Glacier Archives
```

```terraform
  lifecycle {
    prevent_destroy = false
  }

  bucket = aws_s3_bucket.data.id
  key    = "sensitive.csv"
  # ...
  
  # 先删除对象再删除 bucket
  deletion_marker = true
}
```

## 删除验证

### API 级验证

```bash
# S3 删除验证
aws s3api head-object --bucket my-bucket --key sensitive.csv
# 返回 404 → 对象确认删除

# S3 版本列表检查
aws s3api list-object-versions --bucket my-bucket --prefix sensitive.csv
# 返回空 → 所有版本已删除

# EBS 快照验证
aws ec2 describe-snapshots --owner-ids self --filters "Name=volume-id,Values=vol-xxx"
# 确认快照已被删除

# RDS 删除验证
aws rds describe-db-instances --db-instance-identifier my-db
# 返回 "DBInstanceNotFound" → 确认删除

# ECR 镜像删除验证
aws ecr describe-images --repository-name sensitive-repo
# 确认镜像已被删除
```

## Multi-Tenant Storage Purge 挑战

### 共享存储难点

| 挑战 | 说明 | 风险等级 |
|------|------|----------|
| 存储碎片残留 | 对象删除后物理位置可能残留数据 | 中 |
| 快照依赖 | 共享 Snapshot 中可能包含租户数据 | 高 |
| 加密密钥共享 | 多租户共享加密密钥时的清除问题 | 中 |
| 备份链复杂性 | 增量备份依赖链导致单点删除困难 | 高 |
| 缓存/CDN 残留 | CDN/边缘节点可能缓存数据 | 中 |
| 二次复制 | 跨区域复制副本需单独清除 | 高 |

### 云服务商的责任

| 云服务商 | 存储清理策略 | 认证标准 |
|----------|-------------|----------|
| AWS | FIPS 140-2, 存储清理认证 | SOC 2, NIST SP 800-88 |
| Azure | 自动数据清理, 加密擦除 | SOC 2, ISO 27001 |
| GCP | 数据持久性保障, 删除后数据不可用 | SOC 2, ISO 27001 |

## 共享责任模型

### 删除操作的责任分配

```
SaaS:             服务商负责全部删除操作
PaaS:             用户删除数据 → 服务商清理存储
IaaS:             用户管理存储和计算资源 → 服务商清理物理介质
本地:             用户全权负责

云数据删除责任矩阵:
┌──────────────┬──────────┬─────────────┐
│ 责任领域      │ 用户责任 │ 云商责任    │
├──────────────┼──────────┼─────────────┤
│ 逻辑删除 API  │   执行    │    提供      │
│ 备份管理      │   确认    │    提供      │
│ 访问控制      │   管理    │    提供      │
│ 数据验证      │   执行    │    协助      │
│ 合规合规      │   负责    │    证明      │
│ 存储清理      │    —     │    执行      │
│ 介质销毁      │    —     │    执行      │
└──────────────┴──────────┴─────────────┘
```

### 用户最佳实践

```yaml
云数据销毁检查清单:
  销毁前:
    - [ ] 确认数据已不再需要
    - [ ] 检查合规保留策略
    - [ ] 移除数据访问权限
    - [ ] 通知相关数据共享方
  
  销毁中:
    - [ ] 按正确顺序删除 (依赖关系)
    - [ ] 使用 API 而非控制台（可脚本化）
    - [ ] 记录操作日志 (CloudTrail)
    
  销毁后:
    - [ ] API 验证所有资源已被删除
    - [ ] 检查所有区域和项目
    - [ ] 检查备份和快照
    - [ ] 等待保留期结束后确认
    - [ ] 向云商请求销毁证明 (合规场景)
```
