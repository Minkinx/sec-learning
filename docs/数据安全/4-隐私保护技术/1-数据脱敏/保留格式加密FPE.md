# 保留格式加密 (Format-Preserving Encryption, FPE)

## FPE 定义

**保留格式加密 (Format-Preserving Encryption, FPE)** 是一种加密方式，其输出密文与输入明文具有相同的格式和数据类型。例如，对 16 位信用卡号加密后，输出仍是 16 位数字，且符合 Luhn 校验规则（可选）。这使得 FPE 能够在保持数据格式约束的同时提供加密安全性，非常适合与遗留系统集成。

## 核心算法：FF1 / FF3-1

NIST SP 800-38G 标准定义了两种 FPE 模式：

### FF1
- 基于 **Feistel 网络**结构，使用 AES 作为轮函数。
- 支持任意长度的明文（至少 2 字符）。
- 轮数固定为 10 轮。
- 安全性经过广泛分析。

### FF3-1
- 同样基于 Feistel 网络，但轮数较少（8 轮），性能更高。
- 最小明文长度要求为 2 字符（FF3 原版为 2，FF3-1 修订后提升安全性）。
- 对短字符串的加密效率优于 FF1。
- 近期发现对 2 字符明文的某些攻击，建议最小长度 >= 6。

## 格式保留范围

FPE 支持保留多种数据类型格式：

| 数据类型 | 格式约束 | 示例 |
|----------|----------|------|
| SSN | `XXX-XX-XXXX` | `123-45-6789` → `847-21-3905` |
| 信用卡号 | 16 位数字 + Luhn 校验 | `4111-1111-1111-1111` → `5391-8274-6301-4829` |
| 电话号码 | 10-11 位数字 | `138-0013-8000` → `159-0824-6731` |
| 电子邮件 | `user@domain` | `alice@example.com` → `qrzw@example.com` |
| IP 地址 | 点分十进制 | `10.0.0.1` → `192.168.1.100` |

## 密钥管理

FPE 的密钥管理与传统 AES 加密一致，但需注意：

- **密钥派生** — 使用 KDF 从主密钥派生出每个字段的加密密钥。
- **Tweak 输入** — FF1/FF3-1 支持 tweak 参数，用于为不同上下文提供不同的加密输出（类似 IV）。
- **轮换策略** — 密钥定期轮换，但需确保历史数据可用旧密钥解密。

## FPE vs 格式保留哈希

| 特性 | FPE | 格式保留哈希 |
|------|-----|-------------|
| 可逆性 | 可逆（加密） | 不可逆（单向） |
| 确定性 | 是（相同密钥 + tweak） | 是 |
| 安全性基础 | 加密算法（AES） | 哈希算法 + 盐值 |
| 适用场景 | 需要还原原始数据的场景 | 仅需匹配/关联的场景 |
| 合规要求 | 可能被视为加密（受出口管制） | 不被视为加密 |

**选择建议**：当数据需要被还原（如系统间数据同步）时应选择 FPE；当仅需做数据关联匹配时，格式保留哈希在合规性上更简单。
