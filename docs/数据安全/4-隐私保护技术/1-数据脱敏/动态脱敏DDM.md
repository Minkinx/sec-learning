# 动态脱敏 (Dynamic Data Masking, DDM)

## DDM vs SDM

**动态脱敏 (Dynamic Data Masking, DDM)** 与 SDM 的核心区别在于脱敏时机：

| 维度 | DDM | SDM |
|------|-----|-----|
| 脱敏时机 | 查询时（实时） | 数据导出时（离线） |
| 数据存储 | 保持原始数据不变 | 存储脱敏后的副本 |
| 延迟影响 | 增加查询延迟 | 无运行时开销 |
| 适用场景 | 生产查询、API 响应 | 开发/测试环境 |
| 数据一致性 | 取决于策略配置 | 确定性脱敏 |

## 代理架构

DDM 通常采用 **proxy-based 架构**，在客户端与数据库之间部署脱敏代理：

```
Client → DDM Proxy → Database (原始数据)
              ↓
        脱敏后的响应
```

代理拦截 SQL 查询并对结果集进行实时脱敏处理。架构层次包括：

1. **SQL 解析层** — 解析查询语句，识别敏感列。
2. **策略匹配层** — 根据用户身份、角色、会话上下文匹配合适的脱敏策略。
3. **脱敏执行层** — 应用脱敏算法处理结果集。
4. **结果返回层** — 将脱敏后的数据返回给客户端。

## DB Gateway 部署

DDM 可以作为 **Database Gateway** 部署在数据库前端：

- **透明代理** — 对应用程序完全透明，无需修改应用代码。
- **协议感知** — 支持多种数据库协议（MySQL、PostgreSQL、Oracle、SQL Server）。
- **连接池管理** — 复用后端连接，降低数据库连接压力。

## 策略驱动脱敏

DDM 的脱敏策略可以基于：

- **角色 (Role-based)** — DBA 看到完整数据，普通开发人员看到掩码数据。
- **用户 (User-based)** — 根据用户身份动态调整脱敏级别。
- **会话 (Session-based)** — 根据连接上下文（IP 地址、时间、客户端类型）应用不同策略。

示例策略配置：

```yaml
rules:
  - table: employees
    column: salary
    condition: "role != 'finance_manager'"
    masking: "random_range(30000, 50000)"
```

## 性能考量

DDM 的性能开销主要来自：SQL 解析开销、结果集遍历与转换、策略匹配计算。优化策略包括：使用缓存减少重复解析、结果集分片并行处理、预编译策略规则。

## 典型用例

- **生产查询脱敏** — 运维人员查询生产数据库时自动隐藏 PII。
- **API 响应脱敏** — REST/GraphQL API 返回数据时对敏感字段自动掩码。
- **日志脱敏** — 数据库审计日志中的敏感信息实时过滤。
