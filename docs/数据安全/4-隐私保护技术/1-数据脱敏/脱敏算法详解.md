# 脱敏算法详解

## 算法类型

### 1. 替换 (Substitution)
从预定义的字典或映射表中随机选取替代值替换原始值。使用查找表维护一致性和参照完整性。

- **优势**：数据真实性高，保留统计分布特征。
- **劣势**：需要维护查找表，存储开销大。
- **适用**：姓名、地址、公司名等需要真实感的数据。

### 2. 乱序 (Shuffling)
在列内随机重排数据行，打破行之间的关联关系。

- **优势**：保留原始值的真实性和分布。
- **劣势**：破坏列间关联关系，可能导致业务逻辑错误。
- **适用**：不需要跨列关联的单一属性脱敏。

### 3. 数值变异 (Number Variance)
对数值型数据应用随机偏移量（±百分比或固定值），保持数据有效性。

- **优势**：保留数值范围和统计特征。
- **劣势**：需要确保偏移后的值仍在业务合理范围内。
- **适用**：薪资、年龄、交易金额等数值字段。

### 4. 空值化 (Null-Out)
将敏感字段直接置为 NULL 或空值。

- **优势**：实现简单，隐私保护程度最高。
- **劣势**：数据可用性为零，可能破坏应用逻辑。
- **适用**：非必需敏感字段、合规弃用场景。

### 5. 掩码 (Masking)
保留部分字符可见，其余用 `*` 或 `X` 替代。如 `4111****1111`。

- **优势**：保留部分辨识能力，用户体验好。
- **劣势**：隐私保护程度有限。
- **适用**：显示用数据（UI、报表、日志）。

### 6. 哈希 (Hashing)
使用强哈希算法对数据进行单向转换。

- **优势**：不可逆，适合作为关联标识。
- **劣势**：彩虹表攻击风险，需加盐。
- **适用**：需要确定性关联但不需还原的场景。

## 算法选择标准

| 维度 | 考量因素 |
|------|----------|
| **隐私保护程度** | 是否可逆？重识别风险多高？ |
| **数据可用性** | 脱敏后是否支持业务查询、报表、分析？ |
| **格式保留** | 是否需要保持原始格式？ |
| **一致性** | 相同输入是否产生相同输出？ |
| **性能** | 吞吐量、延迟是否满足 SLA？ |
| **参照完整性** | 跨表关联在脱敏后是否仍然有效？ |

## 可逆性分级

| 级别 | 算法 | 用例 |
|------|------|------|
| 可逆 | FPE、对称加密 | 系统间数据交换 |
| 单向（有密钥可逆） | HMAC | 脱敏后的关联查询 |
| 不可逆 | 加盐哈希、替换 | 开发测试环境 |
| 破坏性 | 空值化、掩码 | 日志脱敏、导出脱敏 |

## 性能基准参考

| 算法 | 相对性能（AES 加密为 1.0） |
|------|------------------------|
| 空值化 | 0.01x |
| 掩码 | 0.05x |
| 替换 | 0.1-0.5x |
| 乱序 | 0.1x |
| 数值变异 | 0.1x |
| 哈希 (SHA-256) | 0.5x |
| FPE (FF1) | 2-5x |

## NIST SP 800-53 参考

NIST SP 800-53 中的 **MP-8 (Media Downgrading)** 和 **SC-13 (Cryptographic Protection)** 控制项直接涉及脱敏要求。建议在实施脱敏方案时参考这些标准框架。
