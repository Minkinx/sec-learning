# 技术对比与选型

## 综合对比

| 维度 | 联邦学习 (FL) | 安全多方计算 (MPC) | 可信执行环境 (TEE) | 同态加密 (HE) |
|------|-------------|-------------------|-------------------|---------------|
| **安全假设** | 信任聚合服务器 + 加密通信 | 不信任任何参与方（特定协议容忍腐败） | 信任 CPU 硬件厂商 | 仅依赖计算复杂性假设 |
| **计算开销** | 低-中（模型训练本身） | 高（密码学操作密集） | 低（硬件加速） | 极高（特别是 FHE） |
| **通信复杂度** | 低（仅传输模型参数） | 高（多次交互 + 大量数据交换） | 低（普通网络通信） | 中-高（密文膨胀） |
| **数据支持类型** | 结构化/非结构化（ML 模型） | 通用（理论上可计算任意函数） | 通用（任意程序） | 受限（支持的操作有限） |
| **部署难度** | 中（需要 ML 基础设施） | 高（协议实现复杂） | 中（依赖特定硬件） | 高（性能调优困难） |
| **抗量子安全性** | 不适用（取决于通信加密） | 取决于密码原语 | N/A（硬件安全） | 否（基于格/LWE 的 HE 可能抗量子） |
| **工业成熟度** | 高（有生产级框架） | 中（金融/医疗试点中） | 高（Intel SGX/AMD SEV） | 低-中（仍以研究为主） |
| **典型延迟** | 分钟-小时 | 秒-分钟 | 毫秒-秒 | 秒-小时 |

## 选型决策树

```
是否存在中央数据聚合的条件？
├── 是 → 数据是否高度敏感需加密计算？
│   ├── 是 → 计算逻辑复杂程度？
│   │   ├── 简单运算 → HE
│   │   └── 复杂逻辑 → TEE（性能优先）或 MPC（无硬件依赖）
│   └── 否 → 传统访问控制即可
└── 否 → 数据是否需要协同建模？
    ├── 是 → ML 模型训练 → FL
    ├── 否 → 多方联合计算（聚合/比较）→ MPC
    └── 否 → 数据共享查询 → 脱敏 + 匿名化
```

## 用例映射表

| 应用场景 | 推荐技术 | 备选技术 | 理由 |
|----------|---------|---------|------|
| 跨机构联合风控建模 | FL | MPC | ML 模型适合 FL 的分治训练 |
| 多家银行隐私集合求交求并 | MPC | HE | MPC 对 PSI 支持成熟 |
| 云上敏感数据计算 | TEE | HE | TEE 性能好、通用性强 |
| 医疗数据统计发布 | DP | HE | DP 提供可量化的隐私保证 |
| 加密数据库查询 | HE/OPPE | TEE | HE 支持服务器端盲查询 |
| 区块链隐私交易 | zk-SNARKs | MPC | 验证效率优先选择 ZKP |

## 组合使用

实践中，高级隐私计算方案通常组合多种技术：

- **FL + TEE** — 模型聚合在 TEE enclave 中进行，防止服务器窥探模型更新。
- **FL + DP** — 在模型更新中添加差分隐私噪声，防止梯度泄漏。
- **MPC + TEE** — 使用 TEE 加速 MPC 协议中的密码学操作。
- **HE + MPC** — 用 HE 处理线性操作、MPC 处理非线性操作。

**选型原则**：优先采用最简单、最成熟的方案；在"不可能三角"（安全、性能、通用性）中找到与业务需求匹配的最佳平衡点；考虑组合使用多种技术。
