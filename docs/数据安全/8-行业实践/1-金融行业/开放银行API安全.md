# 开放银行API安全

> 依据《中国人民银行关于规范金融产品网络营销的意见》（银发[2021]237号）及开放银行技术架构规范。

## API安全架构

开放银行API安全体系分为三层：

- **认证层**：确保调用方的身份可靠性
- **授权层**：控制数据访问范围与操作权限
- **防御层**：保护API免遭攻击和滥用

## 认证机制

| 机制 | 适用场景 | 安全级别 |
|------|---------|---------|
| OAuth 2.0 (Authorization Code) | 第三方应用代表用户访问 | 高 |
| OAuth 2.0 (Client Credentials) | 服务端到服务端通信 | 中 |
| OpenID Connect | 身份认证+用户信息获取 | 高 |
| 双向TLS (mTLS) | 高安全要求的机构间通信 | 极高 |
| 数字签名 (JWS) | 交易类API消息完整性保证 | 高 |

## API网关安全控制

- **流量限制**：按应用/用户/IP的多维度速率限制（QPS/TPS）
- **输入校验**：严格JSON Schema校验，防御注入攻击和参数篡改
- **报文加密**：敏感字段在请求/响应体级别实施字段级加密
- **安全监控**：异常调用模式检测（暴力破解、参数遍历、信用卡检测）
- **数据脱敏**：API响应中敏感字段（身份证、手机号、卡号）自动脱敏

## 数据共享范围控制

| 数据分类 | 开放范围 | 授权要求 |
|---------|---------|---------|
| 账户基本信息 | 账户余额、交易明细 | 用户明确授权+动态口令验证 |
| 信贷信息 | 授信额度、还款计划 | 逐笔授权 |
| 投资理财信息 | 持仓、收益 | 用户授权（可批量） |
| 鉴权信息 | 密码、生物特征 | 禁止开放 |
| 交易执行 | 转账、支付 | 独立授权+交易级签名 |

## 客户授权管理

1. **授权生命周期**：授权有效期、可撤销、自动过期机制
2. **粒度控制**：按数据类别、操作类型、时间范围、金额上限分级授权
3. **知情同意**：授权前明确展示数据范围、使用目的、共享第三方清单
4. **授权记录**：每次授权行为记录审计日志，用户可查询历史授权
5. **动态授权**：高风险交易须在交易时单独授权（如转账金额超过阈值）

## 交易级签名

对于交易类API（转账、支付），需要实施：

- 使用私钥对交易关键要素（金额、收款方、时间戳）进行数字签名
- 接收方验证签名有效后方可执行
- 防重放攻击：请求必须包含nonce和时间戳
- 交易全链路追踪：API调用链生成唯一TraceID，便于事后审计

## 监管合规要求

- API接入须经安全评估（渗透测试+代码审计）
- API日志保留不少于6个月
- 每年至少一次开放平台安全审计
- 第三方应用须通过安全认证
- 异常交易实时监测与熔断机制
