# 互联网平台用户信息保护实践

> 大型互联网平台的用户信息保护须贯彻隐私设计（Privacy by Design）、数据最小化、用户赋权三大核心理念。

## 隐私设计框架

| 原则 | 平台落地实践 |
|------|------------|
| 主动预防而非被动补救 | 产品原型阶段嵌入隐私评审 |
| 默认隐私保护 | 收集开关默认关闭，非必要不收集 |
| 端到端安全 | 数据从采集到销毁全链路加密 |
| 可见性与透明度 | 隐私政策、数据处理清单对用户公开 |
| 用户中心 | 提供数据仪表盘、下载、删除功能 |

## 用户数据资产清单

建立数据资产清单是平台数据保护的基础：

| 数据域 | 数据类型 | 存储位置 | 保留期限 |
|-------|---------|---------|---------|
| 账户数据 | 用户名、手机号、邮箱、密码Hash | 用户中心数据库 | 账户存续期间+180天 |
| 个人资料 | 头像、昵称、性别、生日、地址 | 用户属性服务 | 用户删除后30天 |
| 行为数据 | 浏览记录、点击流、搜索日志 | 数据湖/ClickHouse | 36个月 |
| 交易数据 | 订单、支付记录、收货地址 | 交易数据库 | 交易完成后5年 |
| 关系数据 | 好友列表、关注/粉丝、群组 | 社交图谱服务 | 用户注销后删除 |
| 内容数据 | 发帖、评论、私信、图片/视频 | 对象存储 | 用户删除或平台政策 |

## 访问控制策略

- **ABAC（基于属性的访问控制）**：根据用户属性、资源属性、环境条件动态决策
- **资源级权限**：每类数据资源的CRUD操作独立授权
- **最小特权原则**：内部员工默认无权访问原始用户数据，需逐次申请
- **数据匿名化访问**：数据分析场景只开放匿名化后的聚合数据
- **审计全记录**：每次数据访问记录操作人、数据范围、时间、目的

## 数据最小化API设计

| API设计原则 | 实现方式 |
|------------|---------|
| 按需返回 | 默认返回最小字段集，扩展字段须单独申请 |
| 字段级授权 | 每个API字段对应独立scope |
| 批量限制 | 单次API调用返回记录数上限 |
| 频率限制 | 按应用/用户/IP的速率限制 |
| 无冗余参数 | 移除不必要的筛选、排序参数 |

## 用户隐私功能

- **隐私仪表盘**：集中展示平台收集的所有用户数据分类、用途、共享情况
- **数据导出**：支持用户以通用格式（JSON/CSV）导出自有数据
- **账户注销**：提供在线注销入口，注销后进入30天冷静期，到期永久删除
- **授权管理**：第三方应用授权列表，支持一键撤回
- **隐私偏好设置**：个性化推荐开关、数据共享开关、定向广告退订

## 用户同意记录管理

- 每次用户同意的获取记录（时间、版本、用户ID、同意范围）持久化存储
- 同意记录作为平台的合规证据，保留至用户注销后3年
- 隐私政策更新时，须重新获取用户同意并记录版本号
- 用户撤回同意的，系统应立即停止对应的数据处理行为
