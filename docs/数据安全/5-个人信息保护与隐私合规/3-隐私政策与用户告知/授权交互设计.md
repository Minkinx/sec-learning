# 授权交互设计

## 同意采集UI/UX原则

| 原则 | 描述 | 实现方式 |
|------|------|---------|
| 主动同意 | 用户主动勾选或点击确认，非预先选中 | 使用勾选框（checkbox）而非预选复选框 |
| 颗粒化 | 按处理目的分别征得同意 | 分类展示并支持逐项开关 |
| 同等便利 | 撤回与同意操作同样简便 | 同一入口层级，步骤数相同 |
| 不捆绑 | 不可将同意与使用服务捆绑 | 区分必需授权与可选授权 |
| 明确退出 | 拒绝不应比同意更困难 | 拒绝按钮与同意按钮同等显眼 |

## Cookie横幅设计方案

### 设计对比

| 设计类型 | 模式 | 合规评估 | 用户体验 |
|---------|------|---------|---------|
| 知悉即同意型 | "继续浏览即同意"（横幅告知） | 不合规（GDPR/PIPL均要求主动同意） | 最低干扰 |
| 二元选择型 | "接受全部/拒绝全部" | 合规但不敏捷 | 较高拒绝率 |
| 颗粒化型 | 按目的分类展示、逐项开关 | 最合规 | 最佳平衡 |
| 偏好中心型 | 常驻"Cookie设置"入口 | 推荐方式 | 用户自主控制 |

### 合规Cookies分类

| 类别 | 说明 | 是否需要同意 |
|------|------|-------------|
| 必要Cookies | 网站运行必需的Cookie（会话管理、安全防护） | 否 |
| 功能性Cookies | 记住偏好设置等增强功能 | 是 |
| 分析性Cookies | 统计访问数据、用户行为分析 | 是（除匿名化统计外） |
| 广告性Cookies | 定向广告投放、跨站追踪 | 是 |

## 需避免的暗黑模式

- **隐私僵尸（Privacy Zuckering）** — 诱导用户公开更多信息
- **强制延续（Forced Continuity）** — 关闭隐私选项后服务降级
- **误导对比（Confirmshaming）** — "拒绝个性化推荐将获得更差体验"等措辞
- **隐藏选项（Hidden Opt-out）** — 同意按钮显眼而拒绝入口隐蔽
- **预设框架（Default Framing）** — 默认勾选不利选项
- **反复打扰（Nagging）** — 拒绝后反复弹出同意请求

## APP重新授权流程

### 隐私政策变更触发重新授权

| 变更类型 | 操作要求 |
|---------|---------|
| 新增信息收集类型 | 弹窗展示新增内容 → 获取单独同意 |
| 变更处理目的 | 弹窗解释变更原因 → 获取新目的同意 |
| 新增第三方共享 | 列明接收方信息 → 获取第三方共享同意 |
| 跨境传输新增 | 说明接收国和措施 → 获取跨境传输同意 |

### 交互流程示例

1. APP启动后检测隐私政策版本是否更新
2. 发现更新后弹出"隐私政策更新通知"
3. 展示变更摘要（高亮差异内容）
4. 提供"查看完整政策"和"阅读变更内容"按钮
5. 用户逐项确认新的授权项（或一键同意全部变更）
6. 记录同意日志（时间戳、版本号、用户ID）
