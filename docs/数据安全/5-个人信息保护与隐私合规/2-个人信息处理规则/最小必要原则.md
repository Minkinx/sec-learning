# 最小必要原则

## 法律依据

《个人信息保护法》第六条：收集个人信息应当限于实现处理目的的最小范围，不得过度收集个人信息。

## 实施方法

### 第一步：目的界定

- 明确每类数据处理的具体业务目的
- 目的需具体、清晰，不得泛化描述
- 目的与功能直接关联，区分核心功能与扩展功能

### 第二步：数据识别与必要性评估

| 维度 | 评估要点 |
|------|---------|
| 功能相关性 | 该数据是否为实现该功能所必需 |
| 数量最小化 | 收集的数据量是否超出合理范围 |
| 频率适当性 | 收集频率是否与服务使用场景匹配 |
| 存储期限 | 保存期限是否为实现目的所必需 |

### 第三步：必要性文档记录

- 建立数据收集需求文档模板
- 每项数据收集需记录：业务目的、必要性理由、不收集的影响
- 管理层审核确认机制

### 第四步：定期审查

- 新业务上线前进行必要性预审
- 已上线功能每半年审查一次数据收集范围
- 业务变更时重新评估数据必要性
- 发现超范围收集立即启动清理流程

## 违规案例

| 案例 | 违规情形 | 处罚 |
|------|---------|------|
| APP超范围采集通讯录 | 导航类APP强制读取通讯录权限 | 责令整改+下架 |
| 输入法收集剪贴板内容 | 未明确告知且非输入功能必需 | 约谈+限期整改 |
| 金融APP获取通话记录 | 风控目的超范围收集个人信息 | 行政处罚+通报 |
| 电商APP强制位置权限 | 非配送场景强制获取精确定位 | 工信部通报+下架 |

## 合规清单

- [ ] 建立数据收集必要性评估流程
- [ ] 区分核心功能与扩展功能的数据收集
- [ ] 默认不收集非必要数据（Privacy by Default）
- [ ] 提供非必要功能的单独开关
- [ ] 定期清理已超过存储期限的数据
- [ ] 第三方SDK数据接入同样适用最小必要原则
- [ ] 每次更新前进行增量必要性评估
