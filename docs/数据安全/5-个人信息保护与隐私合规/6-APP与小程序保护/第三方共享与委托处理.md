# 第三方共享与委托处理

## 基本概念区分

| 维度 | 委托处理（Processor） | 第三方共享（Third-party Sharing） | 共同控制（Joint Controller） |
|------|---------------------|-------------------------------|---------------------------|
| 决定权 | 委托方决定目的和方式 | 双方各自独立决定 | 共同决定目的和方式 |
| 法律定性 | PIPL第21条 | PIPL第23条 | PIPL第20条 |
| 告知义务 | 委托方履行告知（说明委托处理的情况） | 双方均有告知义务（告知接收方信息） | 共同安排告知义务 |
| 同意要求 | 委托方获取同意（基于委托处理说明） | 需获单独同意（PIPL第23条） | 需获同意 |
| 法律责任 | 委托方承担主要责任；受托人超出委托范围自行担责 | 各自对违规行为负责 | 连带责任 |
| 典型场景 | 云服务、数据分析服务商、客服外包 | 广告合作方、业务合作方、数据联合营销 | 联合营销合作、合资企业数据项目 |

## PIPL第21条（委托处理）逐条详细解读

**原文**："个人信息处理者委托处理个人信息的，应当与受托人约定委托处理的目的、期限、处理方式、个人信息的种类、保护措施以及双方的权利和义务等。"

### 逐条拆解

| 条款要素 | 合规要求 | 实操落地 |
|---------|---------|---------|
| 委托处理的目的 | 必须具体明确，不能是"改善服务"等模糊表述 | 在DPA中以条款形式明确限定目的，如"仅用于推送消息的通道维护" |
| 处理期限 | 约定具体的起止时间，或终止条件 | DPA中注明生效日期和到期日期，或约定"处理目的达成时自动终止" |
| 处理方式 | 明确数据加工、存储、传输、销毁的具体方式方法 | 约定数据存储于指定服务器、仅通过API接口传输、定期清理机制 |
| 个人信息种类 | 逐项列明委托处理的个人信息字段 | 以数据清单附件形式详细列举，避免兜底条款 |
| 保护措施 | 受托人应采取的技术和组织安全措施 | 明确加密标准（如AES-256）、访问控制（RBAC）、审计日志要求 |

**第二款**："受托人应当按照约定处理个人信息，不得超出约定的处理目的、处理方式等处理个人信息。"

**第三款**："委托合同不生效、无效、被撤销或者终止的，受托人应当将个人信息返还个人信息处理者或者予以删除，不得保留。"

**第四款**："未经个人信息处理者同意，受托人不得转委托他人处理个人信息。"

## PIPL第23条（第三方提供）逐条详细解读

**原文**："个人信息处理者向其他个人信息处理者提供其处理的个人信息的，应当向个人告知接收方的名称或者姓名、联系方式、处理目的、处理方式和个人信息的种类，并取得个人的单独同意。"

### 逐条拆解

| 告知要素 | 要求 | 合规落地方法 |
|---------|------|------------|
| 接收方名称 | 必须提供具体名称，不能写"第三方合作伙伴" | "杭州XX科技有限公司"或"XX集团的关联公司" |
| 联系方式 | 接收方的有效联系方式 | 接收方客服电话、邮箱或数据保护官联系方式 |
| 处理目的 | 接收方使用该数据的具体目的 | "用于向您提供商品配送服务"而非"用于改善用户体验" |
| 处理方式 | 接收方将如何加工、使用该数据 | "通过自动化系统进行地址匹配和路线规划" |
| 个人信息种类 | 明确所共享的数据字段 | "您的收件人姓名、联系电话、收货地址" |

**第二款**："接收方应当在上述处理目的、处理方式和个人信息的种类等范围内处理个人信息。接收方变更原先的处理目的、处理方式的，应当依照本法规定重新取得个人同意。"

## 委托处理 vs 第三方共享 vs 共同控制的判定流程

### 判定决策树

```
是数据处理方自行决定处理目的和方式吗？
├── 是 → 是个人信息处理者（Controller）
│   ├── 将数据提供给另一个独立处理者（对方自行决定用途）
│   │   └── → 第三方共享（PIPL第23条）
│   │       需要：告知接收方信息 + 单独同意
│   │
│   ├── 将数据交给对方处理但自己决定一切
│   │   └── → 委托处理（PIPL第21条）
│   │       需要：DPA + 监督受托人
│   │
│   └── 与另一个主体共同决定处理目的和方式
│       └── → 共同控制（PIPL第20条）
│           需要：合同约定各自权利义务 + 连带责任
│
└── 否 → 是受托人（Processor），受委托方控制
    必须：严格按照委托合同处理数据
    禁止：超出委托范围使用数据
    转委托：需获得委托方事前同意
```

### 典型场景判定示例

| 场景描述 | 判定结果 | 法律依据 |
|---------|---------|---------|
| APP接入第三方支付SDK完成支付 | 委托处理（SDK执行支付指令） | PIPL第21条 |
| APP将用户数据提供给广告平台进行独立分析和投放 | 第三方共享 | PIPL第23条 |
| 两家公司合作打造联合会员体系，共同使用会员数据 | 共同控制 | PIPL第20条 |
| 使用云服务商存储用户数据 | 委托处理 | PIPL第21条 |
| 将用户数据出售给数据中介公司 | 第三方共享（且可能违反PIPL基本原则） | PIPL第23条 + 第6条（最小必要） |
| 向集团内关联公司共享用户数据用于市场营销 | 第三方共享（关联公司是独立处理者） | PIPL第23条 |
| 因合并、分立等原因转移个人信息 | 告知义务（非PIPL第21/23条） | PIPL第22条 |

### 同意记录存储方案

```python
# 同意记录日志存储方案（MQTT + 审计数据库）
import json
import hashlib
from datetime import datetime
from pymongo import MongoClient

class ConsentAuditLogger:
    """第三方共享同意记录审计日志"""
    
    def __init__(self):
        self.client = MongoClient("mongodb://audit-db:27017")
        self.db = self.client["consent_audit"]
        self.collection = self.db["third_party_sharing_consents"]
    
    def log_consent(self, user_id: str, consent_data: dict):
        """
        记录用户对第三方共享的同意
        
        consent_data包含：
        - third_party_name: 第三方名称
        - shared_data_types: 共享的数据类型列表
        - purpose: 共享目的
        - consent_version: 同意版本号
        - granted: True/False
        - consent_timestamp: 同意时间戳
        """
        record = {
            "user_id": user_id,
            "consent_id": self._generate_consent_id(user_id, consent_data),
            "third_party_name": consent_data["third_party_name"],
            "shared_data_types": consent_data["shared_data_types"],
            "purpose": consent_data["purpose"],
            "consent_version": consent_data.get("consent_version", "v1"),
            "granted": consent_data["granted"],
            "consent_timestamp": datetime.utcnow(),
            "ip_address": consent_data.get("ip_address", ""),
            "user_agent": consent_data.get("user_agent", ""),
            "source": consent_data.get("source", "in_app_popup"),
            # 不可篡改哈希链
            "consent_hash": self._compute_hash(consent_data),
            "previous_hash": self._get_last_hash()
        }
        self.collection.insert_one(record)
        
        # MQTT发布审计事件
        self._publish_audit_event("consent.recorded", {
            "consent_id": record["consent_id"],
            "user_id": user_id,
            "timestamp": record["consent_timestamp"].isoformat()
        })
        
        return record["consent_id"]
    
    def check_consent(self, user_id: str, third_party: str) -> bool:
        """检查用户是否已同意某个第三方的数据共享"""
        latest = self.collection.find_one(
            {"user_id": user_id, "third_party_name": third_party},
            sort=[("consent_timestamp", -1)]
        )
        if latest:
            return latest["granted"]
        return False
    
    def revoke_consent(self, user_id: str, third_party: str):
        """记录用户撤回同意"""
        record = {
            "user_id": user_id,
            "third_party_name": third_party,
            "consent_type": "revocation",
            "granted": False,
            "consent_timestamp": datetime.utcnow(),
        }
        self.collection.insert_one(record)
        self._publish_audit_event("consent.revoked", {
            "user_id": user_id,
            "third_party": third_party,
            "timestamp": record["consent_timestamp"].isoformat()
        })
    
    def _generate_consent_id(self, user_id: str, data: dict) -> str:
        raw = f"{user_id}:{data['third_party_name']}:{datetime.utcnow().isoformat()}"
        return hashlib.sha256(raw.encode()).hexdigest()[:16]
    
    def _compute_hash(self, data: dict) -> str:
        raw = json.dumps(data, sort_keys=True)
        return hashlib.sha256(raw.encode()).hexdigest()
    
    def _get_last_hash(self) -> str:
        last = self.collection.find_one(sort=[("_id", -1)])
        return last["consent_hash"] if last else "GENESIS_BLOCK"
    
    def _publish_audit_event(self, topic: str, payload: dict):
        """发布审计事件到MQTT"""
```

## 集团内部数据共享合规要求

| 场景 | 合规要求 | 具体做法 |
|------|---------|---------|
| 同一集团内不同法人实体间共享 | 视为第三方共享，需满足PIPL第23条 | 获取单独同意、告知接收方信息、记录共享活动 |
| 集团集中数据平台（统一存储各子公司数据） | 通常视为共同控制或委托处理 | 签署内部DPA/合作协议，明确各自责任边界 |
| 集团范围内统一用户画像 | 需评估数据处理目的变化，需重新获同意 | 逐项说明共享范围和使用方式 |
| 集团内员工个人信息共享 | 需告知员工并获得单独同意 | 集团内部数据共享政策 + 单独同意书 |

**注意**：关联关系不构成共享的豁免理由。即使在同一集团内，不同法人实体之间共享用户数据依然需要遵守PIPL第23条。PIPL没有"集团内部豁免"的规定。

## 公开披露个人信息的合规要求（PIPL第25条）

| 要求 | 具体内容 | 例外情形 |
|------|---------|---------|
| 原则上禁止公开披露 | 个人信息处理者不得公开披露其处理的个人信息 | 法律另有规定或取得个人单独同意的除外 |
| 取得单独同意 | 必须取得个人对公开披露的专项单独同意 | 不适用 |
| 目的限制 | 公开披露的目的和范围应当明确、合理 | - |
| 风险评估 | 公开披露前应进行个人信息保护影响评估（PIA） | - |
| 记录保存 | 保存公开披露的记录至少3年 | - |

## DPA（数据处理协议）完整条款模板

### 必备条款（中英双语）

**数据处理范围 / Scope of Data Processing**

中文：受托方仅可按照本协议约定的目的和方式处理个人信息。受托方不得将个人信息用于履行本协议之外的任何目的。
English: The Processor shall only process personal data in accordance with the purposes and methods set forth in this Agreement. The Processor shall not use the personal data for any purpose other than the performance of this Agreement.

**保密义务 / Confidentiality Obligations**

中文：受托方应对其知悉的个人信息严格保密，不得向任何第三方泄露。受托方应确保其有权接触个人信息的雇员、分包商均签署同等保密义务的协议。
English: The Processor shall keep strictly confidential all personal data obtained hereunder and shall not disclose it to any third party. The Processor shall ensure that all employees and subcontractors who have access to personal data have signed agreements with equivalent confidentiality obligations.

**安全措施要求 / Security Measures**

中文：受托方应采取不低于以下标准的技术和组织安全措施：（1）传输加密—TLS 1.2及以上；（2）存储加密—AES-256；（3）访问控制—最小权限原则+多因素认证；（4）日志审计—保留至少6个月。
English: The Processor shall implement technical and organizational security measures at least at the following levels: (1) TLS 1.2 or above for transmission; (2) AES-256 for storage; (3) access control based on least privilege principle with MFA; (4) audit logs retained for at least 6 months.

**子处理者管理 / Sub-processor Management**

中文：受托方增加或更换子处理者，应提前30日书面通知委托方，并获得委托方事先书面同意。委托方有权在15日内对子处理者提出异议。
English: The Processor shall notify the Controller in writing at least 30 days in advance of any intended addition or replacement of sub-processors and obtain the Controller's prior written consent. The Controller shall have the right to object to the sub-processor within 15 days.

**数据泄露通报 / Data Breach Notification**

中文：发生个人信息安全事件后，受托方应在24小时内通知委托方，并提供以下信息：（1）事件发生时间和发现时间；（2）受影响的数据类型和数量；（3）已采取或拟采取的应急处置措施；（4）联系人和联系方式。
English: In the event of a personal data breach, the Processor shall notify the Controller within 24 hours, providing: (1) time of occurrence and discovery; (2) types and volume of data affected; (3) emergency response measures taken or planned; (4) contact person and contact information.

**审计权 / Audit Rights**

中文：委托方有权每年一次对受托方进行现场或远程审计，审计内容包括数据处理活动、安全措施有效性、合规管理等。受托方应积极配合并提供必要信息和支持。
English: The Controller shall have the right to conduct on-site or remote audits once per year, covering data processing activities, effectiveness of security measures, and compliance management. The Processor shall cooperate and provide necessary information and support.

**数据删除与归还 / Data Deletion and Return**

中文：本协议终止后30日内，受托方应（1）根据委托方要求归还所有个人信息；（2）彻底删除其系统和介质中保留的所有个人信息，并提供书面确认函。
English: Within 30 days after termination of this Agreement, the Processor shall (1) return all personal data as directed by the Controller; (2) completely delete all personal data from its systems and media, and provide written confirmation thereof.

**跨境传输条款 / Cross-border Data Transfer**

中文：如涉及数据出境，受托方应确保跨境传输符合《个人信息保护法》第38条的规定，包括但不限于通过国家网信部门组织的安全评估、获得个人信息保护认证、或与境外接收方签订标准合同。
English: Where cross-border data transfer is involved, the Processor shall ensure compliance with Article 38 of the PIPL, including but not limited to passing the security assessment organized by the CAC, obtaining personal information protection certification, or entering into standard contractual clauses with the overseas recipient.

## 跨境委托处理的额外要求

| 合规路径 | 适用条件 | 实施要点 |
|---------|---------|---------|
| 数据出境安全评估 | 关键信息基础设施运营者 / 处理100万人以上个人信息 / 累计向境外提供10万人以上个人信息 / 累计向境外提供1万人以上敏感个人信息 | 通过省级网信办向国家网信办申报，评估有效期为2年 |
| 个人信息保护认证（CCRC） | 非上述情形 | 获得CCRC认证（依据GB/T 35273-2020及个人信息保护认证实施规则） |
| 标准合同（SCC） | 非上述情形 | 签署国家网信办发布的《个人信息出境标准合同》，备案至省级网信办（合同生效后10个工作日内） |
| 境外接收方处理限制 | 所有情形 | 境外接收方不得超出合同约定的目的和方式处理，且境内委托方对境外受托人的违规承担连带责任 |

## 委托处理的监督

### 受托人审计清单

| 检查项 | 检查频率 | 审计方法 |
|--------|---------|---------|
| 数据收集范围是否超出合同约定 | 每季度 | 抓包分析 + 日志审查 |
| 数据处理是否在合同约定的目的范围内 | 每半年 | 数据处理活动记录审查 |
| 安全措施是否持续有效 | 每年 | 渗透测试 + 安全扫描报告审查 |
| 子处理者管理是否合规 | 每年 | 核查子处理者清单和DPA |
| 数据删除/归还是否执行 | 合同终止时 | 书面确认函 + 远程验证 |
| 安全事件通报流程是否有效 | 每年 | 模拟演练 + 事件响应报告 |
| 数据跨境传输是否合规 | 每年 | 跨境数据传输安全评估报告审查 |
| 是否配置了独立的日志审计系统 | 首次导入 | 系统架构审查 |

### 定期安全评估内容

| 评估项目 | 具体内容 | 评估标准 |
|---------|---------|---------|
| 组织安全 | DPO任命情况、安全管理制度、员工培训记录 | PIPL第51条 + GB/T 35273-2020 |
| 技术安全 | 加密措施、访问控制、网络安全防护、入侵检测 | 等保2.0三级要求 |
| 数据安全 | 数据分类分级、数据脱敏、备份恢复 | 《数据安全法》第21条 |
| 业务连续性 | 应急预案、容灾备份、业务恢复时间目标 | ISO 22301标准 |
| 合规管理 | 隐私政策一致性、投诉处理机制、跨境传输合规 | PIPL各相关条款 |

### 受托人违规的应急处理

```
发现受托人违规行为
  ↓
(1) 立即要求受托人停止违规行为（口头+书面通知）
  ↓
(2) 启动应急评估：评估数据泄露/滥用的范围和影响
  ↓
(3) 依据PIPL第57条：发生泄露后立即采取补救措施并通知监管（72小时内）+ 通知个人
  ↓
(4) 启动备选受托人切换流程（如有备选）
  ↓
(5) 追究合同违约责任（索赔、解约）
  ↓
(6) 如涉及刑事犯罪，向公安机关报案
  ↓
(7) 经验复盘：更新受托人筛选标准和管理流程
```

## 委托处理终止时的数据处置方案

| 数据处置方式 | 适用场景 | 操作要求 | 验证方法 |
|------------|---------|---------|---------|
| 归还数据 | 委托方有继续处理需求 | 受托人将数据以约定格式（加密传输）归还委托方 | 数据完整性校验+交接确认书 |
| 彻底删除 | 委托方无继续处理需求 | 受托人删除所有系统（生产/备份/日志）中的数据 | 专业工具清除+第三方销毁证明 |
| 匿名化处理 | 数据需用于统计分析、AI训练（需合同明确约定） | 受托人进行匿名化处理，确保无法重新识别个人身份 | 匿名化有效性评估报告 |
| 部分删除+部分保留 | 法律要求保留部分数据（如财务记录） | 明确保留的数据范围和保留期限，其余数据删除 | 数据保留与删除清单 |

## 典型违规案例分析

### 案例1：未经同意向关联公司共享用户数据

**基本情况**：某大型互联网公司（2023年被处罚）将其APP用户的个人数据（设备信息、浏览行为、购物记录）未经用户同意即共享给集团内的金融科技子公司，用于风险控制和信用评估。

**违规认定**：
- 未告知用户数据共享给关联公司
- 未获取共享的单独同意
- 共享目的与原收集目的不一致

**处罚结果**：被责令整改，处数千万元罚款，相关业务暂停3个月。

### 案例2：SDK违规作为"子处理者"共享数据

**基本情况**：某APP嵌入的统计SDK在未经APP开发者和用户同意的情况下，将收集的用户数据共享给广告联盟用于广告投放。

**违规认定**：
- SDK超出委托范围处理数据（违约）
- APP未对SDK的子处理者行为实施有效监督（监管失责）
- 双方均被处罚

**处罚结果**：APP被通报下架，SDK提供商被列入SDK违规名单。

### 案例3：委托处理合同不完整导致责任不清

**基本情况**：某云服务商（受托人）因安全配置不当导致其客户的数据泄露。由于双方签署的合同中未明确安全基线责任划分，导致追责困难。

**教训**：
- DPA中必须明确安全措施的最低标准
- 必须约定安全事件的责任划分
- 审计权条款的缺失导致委托方无法及时发现受托人的配置问题

## 子处理者管理流程详细步骤

```
阶段一：子处理者准入
  ├── Step 1: 受托人识别需要引入子处理者的业务场景
  ├── Step 2: 受托人评估备选子处理者的安全能力
  │     ├── 安全认证（ISO 27701/27001/SOC2）
  │     ├── 数据保护水平评估
  │     └── 历史合规记录审查
  ├── Step 3: 受托人向委托方提交子处理者清单
  │     ├── 子处理者全称和注册地址
  │     ├── 拟提供的服务内容
  │     ├── 拟处理的数据类型
  │     ├── 数据存储位置（国家/地区）
  │     └── 安全能力证明文件
  └── Step 4: 委托方审查并作出决定
        ├── 同意——纳入子处理者名单
        ├── 有条件同意——要求补充安全措施
        └── 不同意——受托人需另行选择

阶段二：子处理者准入后的管理
  ├── Step 5: 签署子处理者DPA（与受托人同等标准）
  ├── Step 6: 配置子处理者访问权限（最小化原则）
  ├── Step 7: 定期监控子处理者的数据处理行为
  └── Step 8: 年度安全审计（可要求子处理者提供SOC2报告）

阶段三：子处理者退出
  ├── Step 9: 合同到期或终止——数据删除/归还确认
  ├── Step 10: 切断所有数据访问通道
  └── Step 11: 获取子处理者的数据销毁证明

管理工具：
  ┌─────────────────────────────────────────────────────┐
  │              子处理者管理台账                         │
  ├─────────┬──────────┬──────────┬──────────┬──────────┤
  │ 子处理者 │ 服务内容 │ 数据范围 │ 存储地点 │ 审计状态 │
  ├─────────┼──────────┼──────────┼──────────┼──────────┤
  │ AWS     │ 云存储   │ 用户文档 │ 新加坡   │ 2024Q1通过 │
  │ SendGrid│ 邮件服务 │ 用户邮箱 │ 美国     │ 2024Q2待审 │
  └─────────┴──────────┴──────────┴──────────┴──────────┘
```
