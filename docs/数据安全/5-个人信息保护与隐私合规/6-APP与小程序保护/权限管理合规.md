# 权限管理合规

## 权限分类体系

### Android权限分类（传统三级体系）

| 权限级别 | 定义 | 示例 | 授权方式 |
|---------|------|------|---------|
| 正常权限（Normal） | 对用户隐私风险较低的权限 | INTERNET、BLUETOOTH、NFC、ACCESS_NETWORK_STATE | 安装时自动授予（Android 6.0以下）或声明即授予（6.0+） |
| 危险权限（Dangerous） | 可能访问用户私密数据的权限 | 位置、联系人、相机、麦克风、短信、存储 | 运行时动态请求，用户可拒绝 |
| 特殊权限（Special） | 系统级敏感权限 | SYSTEM_ALERT_WINDOW、WRITE_SETTINGS、无障碍服务 | 需跳转系统设置手动授权 |

### Android 14（API 34）及以上权限变化

| 变更点 | 具体内容 | 合规影响 |
|--------|---------|---------|
| 照片选择器（Photo Picker）强化 | Android 14起无需READ_MEDIA_IMAGES即可通过系统选择器选取单张/多张照片 | 推荐使用系统的照片选择器替代直接读相册权限 |
| 部分照片访问权限 | Android 14新增只授予"选择的照片"选项（非"全部"或"拒绝"） | APP需适配READ_MEDIA_VISUAL_USER_SELECTED权限 |
| 后台位置权限进一步收紧 | 后台位置请求弹窗增加"不允许"选项且更突出 | 申请后台位置权限的合规门槛显著提高 |
| 通知权限分离 | Android 13开始通知权限需运行时申请，Android 14进一步强化 | APP首次启动即请求通知权限视为违规 |
| 屏幕内容读取限制 | Android 14限制APP读取其他APP的屏幕内容 | 录屏/远程协助类APP需使用MediaProjection API |
| 运行时广播限制 | 应用内广播需使用FLAG_RECEIVER_EXPORTED/NOT_EXPORTED声明 | 静态注册广播接收器且未明确导出标记将被系统拦截 |

```kotlin
// Android 14 照片选择器适配示例
val pickPhotoLauncher = registerForActivityResult(
    ActivityResultContracts.PickVisualMedia()
) { uri: Uri? ->
    uri?.let { /* 处理选中的照片 */ }
}

// 触发选择器（无需READ_MEDIA_IMAGES权限）
pickPhotoLauncher.launch(PickVisualMediaRequest(
    ActivityResultContracts.PickVisualMedia.ImageOnly
))

// 适配"部分照片访问"（Android 14+）
// 1. 声明新权限
// <uses-permission android:name="android.permission.READ_MEDIA_VISUAL_USER_SELECTED" />
// 2. 在权限回调中处理三种情况
override fun onRequestPermissionsResult(requestCode: Int,
    permissions: Array<String>, grantResults: IntArray) {
    when {
        // 用户授权全部照片
        grantResults.contains(PackageManager.PERMISSION_GRANTED) && 
            grantResults.contains(PackageManager.PERMISSION_GRANTED_FOR_ALL_MEDIA) -> {
            // 获得全部访问权限
        }
        // 用户授权部分照片（仅Android 14+）
        grantResults.contains(PackageManager.PERMISSION_GRANTED) -> {
            // 权限受限，使用Photo Picker进一步选择
        }
        // 用户拒绝
        else -> {
            // 降级处理
        }
    }
}
```

### Android 15（API 35）新增权限变化

- **前台服务类型声明**：新增`foregroundServiceType`必须明确定义，后台权限使用更加严格
- **敏感通知检测**：系统可自动检测并拦截包含敏感内容（如验证码）的跨应用通知
- **存储权限进一步淘汰**：推荐使用MediaStore API代替传统文件访问权限
- **屏幕录制权限审查**：持续录制超过60秒需额外提示用户

### iOS权限分类

| 权限类型 | 示例 | 授权方式 |
|---------|------|---------|
| 敏感权限 | 位置、相机、麦克风、相册、联系人、日历 | 首次使用弹窗授权 |
| 基础权限 | 通知、网络（无线局域网）、蓝牙 | 安装后首次启动询问 |
| 推送权限 | 推送通知、本地通知 | 授权后系统级管理 |
| 特殊权限 | 健康数据、HomeKit、运动与健身 | 独立授权弹窗 |

### iOS 17+ 权限变化

| 权限变更 | 具体内容 | 合规要求 |
|---------|---------|---------|
| 敏感内容检测（Communication Safety） | 系统级检测儿童收到的敏感图片 | 若APP涉及通信功能，需适配系统的Communication Safety API |
| 相册权限细化 | iOS 17起提供限制相册访问（Limited Photo Access）更严格 | 需支持用户的"部分照片授权"选择 |
| 自动验证码清理 | iOS 17自动删除短信中的一次性验证码 | 验证码获取功能需适配 |
| 第三方键盘扩展权限收紧 | 需用户主动选择是否允许完全访问 | 键盘类APP需在弹窗中说明"完全访问"的必要性 |
| 日志权限审计 | iOS 17日志系统强化 | APP后台行为更易被审计发现 |

### 鸿蒙OS（HarmonyOS）权限体系

| 权限级别 | 鸿蒙定义 | 对应Android级别 | 授权方式 |
|---------|---------|----------------|---------|
| system_grant | 系统授权，安装时自动授予 | 正常权限 | 安装时自动授予 |
| user_grant | 用户授权，运行时动态请求 | 危险权限 | 弹窗授权 |
| 敏感权限（特殊） | 涉及用户敏感数据的权限 | 特殊权限 | 系统设置手动授权 |

**鸿蒙OS与Android的差异对比**：

| 维度 | Android | HarmonyOS |
|------|---------|-----------|
| 权限声明文件 | AndroidManifest.xml | module.json5 + requestPermissions |
| 权限组定义 | 以permission-group为维度 | 以target为维度，粒度更细 |
| 位置权限 | ACCESS_FINE_LOCATION / ACCESS_COARSE_LOCATION | ohos.permission.LOCATION_IN_BACKGROUND / ohos.permission.APPROXIMATELY_LOCATION |
| 后台位置 | 需单独申请 | 需同时声明LOCATION + LOCATION_IN_BACKGROUND（审核更严） |
| 相机权限 | CAMERA | ohos.permission.CAMERA |
| 麦克风权限 | RECORD_AUDIO | ohos.permission.MICROPHONE |
| 用户授权弹窗UI | 系统弹窗（不同厂商UI不同） | 统一弹窗风格 |
| 权限组变更通知 | 无统一通知机制 | 有统一的权限变更监听回调 |

**鸿蒙OS权限合规要点**：
- 鸿蒙从API 9起要求所有user_grant权限必须在运行时动态申请，不支持预授权
- 鸿蒙取消了Android的"始终允许"和"仅在使用中"的模糊选择，位置权限精确度要求更高
- 鸿蒙4.0+加强了对超范围权限申请的拦截，不合规APP可能被直接拒绝安装
- 鸿蒙应用市场审核对权限的声明与实际API调用的一致性要求高于Google Play

```typescript
// HarmonyOS 运行时权限请求示例
import abilityAccessCtrl from '@ohos.abilityAccessCtrl';
import bundleManager from '@ohos.bundle.bundleManager';
import common from '@ohos.app.ability.common';

async function requestCameraPermission(context: common.UIAbilityContext) {
    const atManager = abilityAccessCtrl.createAtManager();
    // 检查权限是否已授予
    let grantStatus = await atManager.checkAccessToken(
        bundleManager.getApplicationInfoSync(
            bundleManager.getBundleInfoForSelfSync(bundleManager.BundleFlag.GET_BUNDLE_INFO_WITH_APPLICATION)
        .appInfo.accessTokenId,
        "ohos.permission.CAMERA"
    ));
    
    if (grantStatus === abilityAccessCtrl.GrantStatus.PERMISSION_GRANTED) {
        // 权限已授予，可直接使用相机
        return true;
    }
    
    // 请求权限
    try {
        const grantResult = await atManager.requestPermissionsFromUser(
            context, ["ohos.permission.CAMERA"]
        );
        return grantResult.authResults[0] === 0;
    } catch (err) {
        console.error(`权限请求失败: ${JSON.stringify(err)}`);
        return false;
    }
}
```

## 运行时权限请求策略对比

| 策略维度 | Android | iOS | HarmonyOS |
|---------|---------|-----|-----------|
| 预授权说明弹窗 | 常用（先弹自定义说明再弹系统弹窗） | 推荐但非强制 | 必要（系统要求说明目的） |
| 单次授权 | 无（仅"允许/拒绝"） | "允许一次"（iOS 14+） | 支持（选择"仅本次"） |
| 拒绝后再次请求 | 非必需但推荐引导用户去设置 | 系统自动弹"仅连接XXX时允许" | 需自行判断状态后引导 |
| "不再询问"检测 | shouldShowRequestPermissionRationale() | 不直接提供，需状态推断 | 提供grantStatus状态判断 |
| 权限降级处理 | 需开发者自行实现 | 强烈推荐降级处理 | 系统强制要求降级 |
| 多权限一次请求 | 可批量请求（逐条弹窗） | 不支持（逐条请求） | 支持批量请求（合并弹窗） |

## 权限请求最佳实践代码示例

### Android：正确的一次性请求流程

```kotlin
// 1. 先检查权限状态
private fun checkAndRequestLocationPermission() {
    when {
        ContextCompat.checkSelfPermission(
            this, Manifest.permission.ACCESS_FINE_LOCATION
        ) == PackageManager.PERMISSION_GRANTED -> {
            // 已有权限，执行业务逻辑
            startLocationService()
        }
        // 2. 是否需要显示解释说明
        shouldShowRequestPermissionRationale(
            this, Manifest.permission.ACCESS_FINE_LOCATION
        ) -> {
            // 用户之前拒绝过（但没有勾选"不再询问"）
            showRationaleDialog {
                requestLocationPermissionLauncher.launch(
                    Manifest.permission.ACCESS_FINE_LOCATION
                )
            }
        }
        else -> {
            // 3. 首次请求
            requestLocationPermissionLauncher.launch(
                Manifest.permission.ACCESS_FINE_LOCATION
            )
        }
    }
}

// 4. 注册权限结果回调
private val requestLocationPermissionLauncher = 
    registerForActivityResult(
        ActivityResultContracts.RequestPermission()
    ) { isGranted: Boolean ->
        if (isGranted) {
            startLocationService()
        } else {
            // 5. 拒绝后的降级处理
            handlePermissionDenied()
        }
    }

// 6. 降级处理：检查是否为"不再询问"状态
private fun handleLocationPermissionDenied() {
    if (!shouldShowRequestPermissionRationale(
            this, Manifest.permission.ACCESS_FINE_LOCATION
        )) {
        // 用户选择了"不再询问"——必须引导去系统设置
        showPermissionSettingsDialog(
            "位置权限已被拒绝，请前往系统设置开启以使用导航功能"
        )
    } else {
        // 用户只是这次拒绝——下次使用时可以再次申请
        showFeatureNotAvailableMessage("导航功能不可用")
    }
}

// 7. 引导用户前往设置
private fun showPermissionSettingsDialog(message: String) {
    AlertDialog.Builder(this)
        .setTitle("需要权限")
        .setMessage(message)
        .setPositiveButton("去设置") { _, _ ->
            Intent(Settings.ACTION_APPLICATION_DETAILS_SETTINGS).apply {
                data = Uri.fromParts("package", packageName, null)
                startActivity(this)
            }
        }
        .setNegativeButton("取消", null)
        .show()
}
```

### iOS：完整的权限请求流程

```swift
// 1. 位置权限请求封装（含降级处理）
enum LocationPermissionStatus {
    case granted
    case denied
    case restricted
    case notDetermined
}

class LocationPermissionManager: NSObject, CLLocationManagerDelegate {
    private let locationManager = CLLocationManager()
    private var completion: ((LocationPermissionStatus) -> Void)?
    
    func requestWhenInUsePermission(completion: @escaping (LocationPermissionStatus) -> Void) {
        self.completion = completion
        locationManager.delegate = self
        
        let status = locationManager.authorizationStatus
        switch status {
        case .notDetermined:
            // 首次请求——先展示用途说明（在Info.plist中已配置）
            locationManager.requestWhenInUseAuthorization()
        case .denied, .restricted:
            // 2. 已被拒绝——引导前往设置
            completion(.denied)
            showGoToSettingsAlert(reason: "需要位置权限来为您推荐附近服务")
        case .authorizedWhenInUse, .authorizedAlways:
            completion(.granted)
        @unknown default:
            break
        }
    }
    
    // 3. 权限变更回调
    func locationManagerDidChangeAuthorization(_ manager: CLLocationManager) {
        let status = manager.authorizationStatus
        switch status {
        case .authorizedWhenInUse, .authorizedAlways:
            completion?(.granted)
        case .denied:
            completion?(.denied)
        case .restricted:
            completion?(.restricted)
        case .notDetermined:
            break
        @unknown default:
            break
        }
    }
}
```

## 权限弹窗AB测试与授权转化率优化

### AB测试设计要点

| 测试变量 | 对照组 | 实验组A | 实验组B | 合规边界 |
|---------|-------|---------|---------|---------|
| 说明弹窗文案 | "需要位置权限" | "为您推荐附近的餐厅" | "只在使用时获取位置，不跟踪您" | 不得夸大/欺骗 |
| 弹窗图标 | 无图标 | 功能示意图标 | 隐私盾牌图标 | 不得使用误导性图标 |
| 弹窗按钮文案 | "允许/拒绝" | "开启/不了" | "立即开启/稍后再说" | 不得隐藏拒绝选项 |
| 请求时机 | 启动后立即请求 | 使用功能前3秒预告 | 功能场景触发后 | "即用即申请"原则 |
| 预授权说明 | 无 | 文字说明 | 图片+文字说明 | 说明必须真实准确 |

**合规红线**（绝对禁止）：
- 不得使用倒计时弹窗（"5秒后自动同意"）
- 不得将"拒绝"按钮设计为灰色/半透明/难以点击
- 不得在用户拒绝后24小时内再次弹窗（推荐为72小时冷却期）
- 不得在弹窗中直接获取权限（必须先跳转系统弹窗）

### 授权率优化合规方法

1. **价值说明法**：明确解释权限带来的直接用户体验提升（如"开启位置可查看附近3家最低价加油站"）
2. **渐进式请求**：先请求低敏感度权限，建立信任后再请求高敏感度权限
3. **灰度发布**：按版本/地区逐步放开新的权限请求逻辑，监控授权率和投诉率
4. **权限收益可视化**：在说明弹窗中展示开启前后的对比效果

## 位置权限分级合规

| 位置精度 | 对应权限（Android） | 对应权限（iOS） | 合规场景示例 |
|---------|-------------------|----------------|-------------|
| 粗略位置（1km精度） | ACCESS_COARSE_LOCATION | - | 天气APP获取城市级位置 |
| 精确位置（10m精度） | ACCESS_FINE_LOCATION | kCLLocationAccuracyBest | 导航APP、打车APP |
| 后台位置 | ACCESS_BACKGROUND_LOCATION | locationManager.allowsBackgroundLocationUpdates | 运动记录、车辆轨迹 |
| 单次位置 | - | requestLocation（iOS 14+） | 拍照记录地点 |

**合规要点**：
- 非导航类APP不应申请后台位置权限
- 后台位置权限在Android和iOS上均需经过更严格的审核
- Android上后台位置申请弹窗需附带详细的用途说明文本
- iOS上需在Info.plist中提供NSLocationAlwaysAndWhenInUseUsageDescription
- 用户拒绝精确位置后应降级为粗略位置使用，而非反复申请

## Camera + 扫码权限合规

| 场景 | 合规要求 | 具体做法 |
|------|---------|---------|
| 扫码登录 | 仅需相机权限，目的为扫码识别 | 申请CAMERA权限，说明"用于扫描二维码" |
| 人脸识别 | 敏感个人信息，需单独同意 | 单独弹窗说明处理目的、保存期限、是否传输至第三方 |
| 拍照上传头像 | 非必需权限（可用相册选择替代） | 优先使用系统图片选择器，相机权限作为可选项 |
| 实时美颜/滤镜 | 需相机权限且需前置摄像头 | 说明本地处理不上传，获取单独同意 |
| AR/VR应用 | 深度相机+RGB相机 | 逐项说明每种相机数据的用途和处理方式 |

**合规红线**：
- 不得在扫码结果页附带获取用户面部特征
- 人脸数据采集必须获取单独同意，且不能与用户协议捆绑
- 人脸识别需在14岁以下用户中默认关闭（PIPL第31条）
- 相机权限不能用于后台采集环境信息

## ADID / OAID 获取的合规要求

| 设备标识 | 获取方式 | 合规要求 |
|---------|---------|---------|
| Android广告ID（AAID/GAID） | AdvertisingIdClient | 用户可在设置中重置；需声明广告追踪用途；OPT_OUT需要遵守 |
| 开放匿名设备标识（OAID） | MSA SDK | 由移动安全联盟（MSA）管理；2023年起需符合新的API版本要求 |
| 供应商匿名标识（VAID） | 无需权限 | 同一开发者不同APP间不变，厂商不同则不同 |
| 开发者匿名标识（AAID变种） | 需权限声明 | 用于广告归因，需用户同意 |

**合规要点**：
- PIPL明确禁止未经同意的设备标识收集（2023年专项执法重点）
- 获取OAID时必须在隐私政策中明确说明用途
- 不得将OAID/AAID用于用户画像以外的用途（如与个人身份信息关联）
- 用户选择"限制广告追踪"后，APP必须停止使用广告标识符
- Android 14起，如果APP targetSDK 34+且声明了广告服务，Google Play要求必须在隐私政策中说明

## 权限审计——记录和分析APP的权限使用日志

### 权限日志记录架构

```
权限请求入口 → 权限请求日志（时间戳+权限类型+结果）
       ↓
权限使用记录 → 数据访问日志（时间戳+权限+触发功能+频率）
       ↓
审计分析系统 → 合规报告（异常行为告警+趋势分析+合规评分）
```

```kotlin
// Android 权限使用记录实现示例
data class PermissionAccessRecord(
    val permissionName: String,
    val accessTime: Long,
    val callingClass: String,
    val callingMethod: String,
    val foreground: Boolean,
    val granted: Boolean
)

class PermissionAuditor {
    private val accessLogs = mutableListOf<PermissionAccessRecord>()
    
    fun recordPermissionAccess(
        permission: String,
        clazz: String,
        method: String,
        isForeground: Boolean,
        isGranted: Boolean
    ) {
        val record = PermissionAccessRecord(
            permissionName = permission,
            accessTime = System.currentTimeMillis(),
            callingClass = clazz,
            callingMethod = method,
            foreground = isForeground,
            granted = isGranted
        )
        accessLogs.add(record)
        
        // 异常检测：后台使用权限
        if (!isForeground && !permission.isBackgroundAllowed()) {
            Log.w("PermissionAudit", "WARNING: 后台使用权限: $permission")
            reportAnomaly(record)
        }
        
        // 异常检测：频率过高（同一权限1分钟内触发超过10次）
        val recentAccessCount = accessLogs.count {
            it.permissionName == permission &&
            it.accessTime > System.currentTimeMillis() - 60_000
        }
        if (recentAccessCount > 10) {
            Log.w("PermissionAudit", "WARNING: 权限调用频率异常: $permission, $recentAccessCount次/分钟")
            reportAnomaly(record.copy(permissionName = "$permission (频率异常)"))
        }
    }
}
```

### 权限审计检查清单

| 检查项 | 检查方法 | 合规标准 |
|--------|---------|---------|
| 权限调用与实际功能一致性 | 逐项核对权限声明与功能列表 | 每项权限必须有对应的功能场景 |
| 后台权限调用记录 | 分析24小时内后台权限调用日志 | 后台权限调用须有明确的用户触发 |
| 权限调用频率 | 统计每小时权限调用次数 | 不得出现非功能使用状态的持续调用 |
| 权限调用时机 | 检查请求权限时的APP状态 | 禁止在非功能页面的启动阶段请求 |
| 权限返回数据使用 | 追踪权限数据流向 | 权限数据不得超出声明用途使用 |

## 权限撤销机制

| 平台 | 撤销方式 | 用户路径 | APP内建议 |
|------|---------|---------|----------|
| Android | 系统设置中手动关闭 | 设置 → 应用管理 → 权限管理 | 提供"权限管理"入口按钮，跳转至系统设置 |
| iOS | 系统设置中手动关闭 | 设置 → 隐私 → 对应权限 → APP列表 | 使用UIApplication.openSettingsURLString跳转 |
| HarmonyOS | 系统设置或应用权限管理 | 设置 → 应用和服务 → 应用管理 → 权限 | 调用requestPermissionsFromUser重新授权 |
| 应用内 | 内置权限开关 | 我的 → 设置 → 权限管理 | 对非系统级功能（如推送通知）可直接在APP内关 |

**合规要求**（PIPL第15条）：撤回同意应当如同给予同意一样简便。APP内应提供"权限管理"入口，引导用户一键跳转到系统设置或直接在APP内开关权限。撤回同意不影响撤回前基于同意已进行的处理活动。
