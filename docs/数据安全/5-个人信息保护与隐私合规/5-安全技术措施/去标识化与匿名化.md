# 去标识化与匿名化

## 定义区分

| 维度 | 去标识化（De-identification） | 匿名化（Anonymisation） |
|------|---------------------------|------------------------|
| **PIPL定义** | 处理后无法在不借助额外信息的情况下识别特定自然人 | 处理后无法识别特定自然人且不能复原 |
| **GDPR认识** | 处理后的数据仍可能属于个人数据 | Recital 26：匿名化数据不属于个人数据，不受GDPR管辖 |
| **再识别风险** | 存在再识别风险（信息结合） | 理论上不可再识别 |
| **法律豁免** | 仍受个人信息保护法规制 | 一般不受个人信息保护法规制 |
| **技术要求** | 需管理额外信息的安全 | 需证明不可逆性 |

## 技术控制方法

### 常用技术

| 技术 | 原理 | 适用场景 | 再识别风险 |
|------|------|---------|-----------|
| 加盐哈希（Hashing + Salt） | 使用随机盐值对标识符单向哈希 | 用户ID、手机号、邮箱 | 中（盐值泄露后可能遭受彩虹表攻击） |
| 对称加密（Encryption） | 使用密钥加密标识符 | 临时去标识化场景 | 低（密钥未泄露时安全） |
| 令牌化（Tokenization） | 用随机生成的令牌替代原始数据 | 支付数据、敏感ID | 低（需令牌映射表） |
| 泛化（Generalisation） | 降低数据精度（如年龄范围化） | 统计分析、人群画像 | 取决于泛化程度 |
| 抑制（Suppression） | 删除或遮盖部分数据 | 脱敏展示 | 无（但信息丢失） |
| 扰动（Perturbation） | 添加随机噪声 | 统计查询结果保护 | 低 |

### 去标识化技术选择矩阵

| 数据类型 | 推荐技术 | 实施要点 |
|---------|---------|---------|
| 手机号 | 中间四位掩码 + 哈希 | 展示时掩码，去标识化分析用哈希 |
| 身份证号 | 保留前6后4，其余掩码 | 保持部分位可用于应用校验 |
| 邮箱 | 用户名部分哈希 | 域名保留用于统计分析 |
| 详细地址 | 泛化到街道/区级 | 保留到够用的粒度 |
| 设备标识符（IMEI） | AES加密存储 | 应用层加密，密钥专门管理 |
| IP地址 | 泛化到C段 | 常用于分析日志 |

## 再识别风险评估

### 评估维度

| 风险维度 | 评估因素 | 缓解措施 |
|---------|---------|---------|
| 独特性风险 | 数据集中数据主体可区分的程度 | 泛化或聚合（k-匿名、l-多样性） |
| 链接性风险 | 与其他数据源结合后可识别度 | 限制数据集发布范围 |
| 推理风险 | 从非标识属性推断出身份 | 降低数据精度或添加噪声 |
| 数据量风险 | 披露的数据点数量 | 最小化发布的数据字段和记录数 |

### 再识别风险评估工具

- ARX Data Anonymization Tool（开源）
- Amnesia Anonymization Tool（开源）
- k-匿名（k-anonymity）、l-多样性（l-diversity）、t-接近性（t-closeness）
- 差分隐私（Differential Privacy）— 严格的理论保障

## 去标识化数据处理规则

| 处理场景 | 规则要求 |
|---------|---------|
| 内部使用（分析统计） | 实施去标识化措施，控制访问权限 |
| 与关联公司共享 | 签署数据处理协议，限定使用目的 |
| 对外发布（公开统计） | 需满足匿名化标准或差分隐私 |
| 外包加工（数据处理服务） | 签署保密协议，禁止还原 |
| 数据交易 | 需经匿名化处理，禁止交易可识别信息 |

## 合规建议

- 区分去标识化和匿名化，在数据处理策略中明确规定
- 去标识化数据仍履行告知、安全保护等义务
- 定期评估再识别风险并重新调整去标识化方案
- 建立访问额外信息（密钥、映射表）的审批和审计流程
- 匿名化标准建议参考ISO 27559或NIST SP 800-188
