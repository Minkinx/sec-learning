# PIA隐私影响评估

## 法律依据

| 法域 | 条款 | 要求 |
|------|------|------|
| PIPL | 第55-56条 | 特定情形下应当进行个人信息保护影响评估（PIA） |
| GDPR | Art.35 | 可能对自然人的权利和自由产生高风险处理前应进行数据保护影响评估（DPIA） |
| 其他参考 | ISO 27701、ISO 29134 | 提供PIA/DPIA实施框架参考 |

## PIA触发条件

### PIPL规定的情形（第55条）

- 处理敏感个人信息
- 利用个人信息进行自动化决策
- 委托处理、向第三方提供、公开个人信息
- 向境外提供个人信息
- 其他对个人权益有重大影响的处理活动

### GDPR规定的情形（Art.35(3)）

- 对个人进行系统性和全面性的评价（包括画像），且决策产生法律效力
- 大规模处理特殊类别数据或定罪数据
- 对公共场所的大规模系统性监控
- 使用新技术且可能对权利和自由产生高风险

### GDPR黑名单（WP248指引）

- 员工监控系统
- 大规模地理位置数据
- 敏感数据跨境传输
- 数据匹配或合并
- 易受伤害数据主体（如患者、儿童、员工）

## PIA方法论

### 标准评估流程

1. **筛选与识别** — 判断是否存在PIA触发条件
2. **数据流映射** — 识别数据收集、使用、存储、共享、删除全流程
3. **必要性评估** — 评估数据处理对实现目的的必要性
4. **比例性评估** — 评估数据处理对个人权益的影响程度
5. **风险分析** — 识别和评估对个人权利和自由的风险
6. **风险处置** — 制定风险缓解措施
7. **审批与归档** — PIA报告审批通过后归档备查
8. **定期复审** — 业务变更或至少每年复审一次

### 评估维度

| 维度 | 评估要点 |
|------|---------|
| 必要性 | 处理目的合法性、数据最小化程度 |
| 比例性 | 对个人权益的影响程度、是否有替代方案 |
| 风险等级 | 风险可能性 × 影响程度的综合评估 |
| 隐私增强措施 | 已实施的去标识化、加密、访问控制等措施 |
| 数据主体保障 | 告知机制、权利响应、投诉渠道 |
| 第三方依赖 | 委托处理者、共同控制者的隐私保护水平 |

## 文件化要求

### PIA报告核心结构

| 章节 | 内容 |
|------|------|
| 项目概述 | 处理活动描述、业务背景、法律依据 |
| 数据流分析 | 数据类型、来源、存储、共享、保留期限 |
| 合法性评估 | 合法性基础、告知同意机制 |
| 必要性比例性 | 目的实现与数据收集范围的匹配度 |
| 风险评估 | 识别风险 → 评估等级 → 确定处置优先级 |
| 缓解措施 | 已采取/拟采取的技术和管理措施 |
| 剩余风险 | 采取缓解措施后遗余风险的接受决定 |
| 结论和建议 | 是否可推进、需满足的前置条件 |

### 保存要求

- PIA报告需保存至少3年
- 发生安全事件后应重新评估并更新PIA
- 建议使用标准模板以确保一致性

## 利益相关方参与

- 数据保护团队（DPO/隐私团队）
- 业务部门负责人（了解业务目的和需求）
- 法务团队（法律合规审查）
- 技术安全团队（安全措施评估）
- 可能受影响的数据主体代表或其组织
- 外部顾问（复杂场景）
