# PIA隐私影响评估

> 个人信息保护影响评估的触发条件、流程方法与输出模板

## 法律依据与触发条件

### GDPR Art.35
以下处理活动必须开展DPIA（Data Protection Impact Assessment）：
- 基于自动化处理对个人进行系统化评估（画像）
- 大规模处理特殊类别数据（Art.9/Art.10）
- 对公共区域进行系统性监控

### PIPL Art.55-56
以下情形必须开展个人信息保护影响评估：
- 处理敏感个人信息
- 利用个人信息进行自动化决策
- 委托处理个人信息、向第三方提供或公开个人信息
- 向境外提供个人信息
- 其他对个人权益有重大影响的处理活动

## DPIA评估流程

### 阶段一：系统化描述
- 描述个人信息处理活动的性质、范围、语境和目的
- 绘制数据流图（Data Flow Diagram），标注数据来源、存储位置、流转路径、第三方共享
- 识别涉及的个人信息类型、数量、数据主体范围

### 阶段二：必要性和相称性评估
- 处理目的合法性审查（合法性基础是什么？）
- 数据最小化验证（收集的数据是否与目的成比例？）
- 数据保留期限合理性（是否设定了明确的保留和删除策略？）
- 数据主体权利保障机制（访问权、删除权、可携带权等）

### 阶段三：风险识别与评估
- 对数据主体权益的潜在风险（歧视、身份盗窃、名誉损害、经济损失）
- 风险来源分析（技术漏洞、流程缺陷、人员行为）
- 现有控制措施有效性评估
- 残余风险等级判定（高/中/低）

### 阶段四：风险处置措施
- 确定额外控制措施（技术措施 + 组织措施）
- 控制措施有效性验证
- 残余风险是否可接受

## 评估模板核心字段

```
1. 评估基本信息（日期、版本、评估人）
2. 处理活动描述（名称、目的、法律依据）
3. 数据流分析（收集 → 存储 → 使用 → 共享 → 删除）
4. 个人信息类型清单（普通信息/敏感信息）
5. 数据主体范围（客户/员工/第三方）
6. 涉及第三方情况（名称、角色、数据共享范围）
7. 风险分析矩阵（威胁 × 漏洞 × 影响 × 可能性）
8. 现有控制措施（加密/访问控制/匿名化/审计）
9. 残余风险评估（高/中/低）
10. 处置结论（可接受/不可接受-需整改/需咨询监管机构）
```

## 何时咨询监管机构

当DPIA识别出高风险且无法通过现有措施有效降低时，须在开展处理活动前咨询监管机构（GDPR Art.36）。PIPL框架下如果涉及重大数据安全风险，建议主动与网信办等监管机构沟通。监管机构应在收到咨询后规定时限内（GDPR为8周）给出书面意见。
